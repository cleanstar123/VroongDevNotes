## 주요 완료 작업

### 1. [Backend] 잔액 스냅샷 DB 테이블 설계 및 생성

**배경**
- 재무팀 요청: 매일 23:50 거래 동결 → 00:00 전체 잔액 스냅샷 저장 → 00:05 동결 해제
- 라이더(R), 협력사(S), 본사(C) 전체 잔액을 일자별로 보관하기 위한 테이블 신설

**테이블 구조**

```sql
CREATE TABLE vr_balance_snapshot (
  SEQ_NO      BIGINT        NOT NULL AUTO_INCREMENT,
  SNAP_DATE   VARCHAR(8)    NOT NULL,       -- 기준일자 (YYYYMMDD, INS_DATE - 1일)
  ACCT_TARGET VARCHAR(5)    NOT NULL,       -- R=라이더, S=협력사, C=본사
  SUPPLY_ID   VARCHAR(50)   NULL,
  RIDER_ID    VARCHAR(50)   NOT NULL,
  BAL_AMT     DECIMAL(20,2) DEFAULT 0,      -- BAL_AMT + UNPAID_AMT (실효잔액)
  INS_DATE    DATETIME      NULL,
  INS_USER    VARCHAR(50)   NULL,
  PRIMARY KEY (SEQ_NO),
  INDEX idx_snap_date (SNAP_DATE),
  INDEX idx_snap_date_target (SNAP_DATE, ACCT_TARGET)
);
```

**설계 특이사항**
- `BAL_AMT`는 `vr_account_mst.BAL_AMT + UNPAID_AMT`로 계산 (UNPAID_AMT는 0 이하 미수금)
- `SNAP_DATE`는 배치 실행일 기준 -1일 (00:00에 실행되므로 전날 날짜가 기준일)
- Collation 불일치 주의: `vr_balance_snapshot`은 `utf8mb4_0900_ai_ci`, `vr_account_mst`는 `utf8mb4_general_ci` → JOIN 시 `COLLATE utf8mb4_general_ci` 명시 필요

### 2. [Backend] 거래 동결 인터셉터 구현 (MaintenanceModeInterceptor)

**동작 방식**
- `MaintenanceModeService`: `AtomicBoolean`으로 점검 상태 관리 (in-memory)
- `MaintenanceModeInterceptor`: 점검 중일 때 8개 거래 POST 엔드포인트를 503으로 차단
- `WebMvcConfig`: 인터셉터를 `/api/**` 전체 경로에 등록

**차단 대상 엔드포인트**

| 경로 | 기능 |
|------|------|
| `/api/v1/transfer/request` | 본사→협력사 송금 |
| `/api/v1/headWithdrl/request` | 본사 출금 |
| `/api/v1/headWithdrl/deposit` | 본사 입금 |
| `/api/v1/mob/withdrl/request` | 라이더 모바일 출금 |
| `/api/v1/mob/withdrl/deposit` | 라이더 모바일 입금 |
| `/api/v1/mob/withdrl/transfer` | 라이더/협력사 간 송금 |
| `/api/v1/withdrawal-limit/rider/withdrawal` | 라이더 출금한도 출금 |
| `/api/v1/hecto/deposit/process` | 헥토파이낸셜 입금 |

**주의사항**
- `AtomicBoolean`은 서버 메모리에 저장되므로 **API 서버 인스턴스가 1대**인 경우에만 완전 보장
- 다중 인스턴스 환경으로 확장 시 Redis 또는 DB 기반 플래그로 전환 필요

### 3. [Backend] 잔액 스냅샷 API 구현

**엔드포인트** (`/api/v1/balance-snapshot`)

| Method | Path | 설명 |
|--------|------|------|
| POST | `/maintenance/start` | 거래 동결 시작 (배치 호출) |
| POST | `/maintenance/end` | 거래 동결 해제 (배치 호출) |
| GET | `/maintenance/status` | 점검 모드 상태 조회 |
| POST | `/snapshot` | 잔액 스냅샷 실행 (배치 호출) |
| GET | `/list?snapDate=YYYYMMDD` | 날짜별 스냅샷 데이터 조회 (프론트 호출) |

**스냅샷 로직**
1. 라이더별 최신 잔액: `vr_account_mst`에서 `rider_id`별 최대 `seq_no` 레코드 조회 (native query)
2. 협력사별 최신 잔액: `supply_id`별 최대 `seq_no` 레코드 조회 (native query)
3. 본사 잔액: `acct_target='C'` 최대 `seq_no` 레코드 1건
4. 재실행 대비: 동일 `snap_date` 기존 데이터 삭제 후 재저장
