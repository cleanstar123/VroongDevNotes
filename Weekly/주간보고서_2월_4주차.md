### 1. DB 커넥션 풀(HikariCP) 장애 분석 및 해결

**배경**

- 2026-02-20 12:23경 운영 서버에서 대규모 장애 발생
- 배치 작업 실패, API 500 에러, DB 커넥션 타임아웃 다수 발생

**문제**

- MySQL 서버가 유휴 커넥션을 `wait_timeout`에 의해 자동 종료
- HikariCP는 이를 인지하지 못하고 이미 끊어진 커넥션 사용 시도
- `application.yml`에 커넥션 검증 설정 누락

**해결**

```yaml
hikari:
  max-lifetime: 1740000           # 29분 (MySQL wait_timeout보다 짧게)
  connection-test-query: SELECT 1 # 사용 전 커넥션 유효성 검증
  validation-timeout: 5000        # 검증 타임아웃 5초
```

### 2. 프로젝트 폴더 구조 대규모 리팩토링 (Clean Architecture)

**배경**

- 기존 프로젝트 폴더 구조가 혼재되어 파일 목적 파악이 어려움
- 크롤링 관련 코드와 웹/앱 백엔드 코드가 섞여 있음

**해결**

- **domain 계층**: `crawling/baemin/`, `batch/`, `portal/`, `mobile/` 분리
- **infrastructure 계층**: `crawling/`, `payment/hecto/`, `crypto/` 분리
- **app 계층**: `controller/`, `request/`, `response/` 하위 도메인별 분리

```
api/src/main/java/com/vroong/api/
├── domain/
│   ├── crawling/baemin/  # 배민 크롤링 (79개+ 파일)
│   ├── batch/            # 배치 처리 (10개 파일)
│   ├── portal/           # 포털 비즈니스 (90개+ 파일)
│   └── mobile/           # 모바일 앱 비즈니스
├── infrastructure/
│   ├── crawling/         # 크롤링 인프라 (14개 파일)
│   └── payment/hecto/    # Hecto 결제 (3개 파일)
└── app/
    ├── controller/       # 도메인별 API
    ├── request/          # 도메인별 요청 DTO
    └── response/         # 도메인별 응답 DTO
```

### 3. relay-server 세션 관리 개선

**문제 1: 배치 실행 오류 (400 Bad Request)**

- 서버 재배포 후 Redis `active_sessions` Set이 비어있어 세션을 찾지 못함
- keep-alive가 세션 생성보다 먼저 실행되어 유효한 세션도 삭제

**해결**: Redis에 유효한 세션 데이터가 있으면 유지하도록 이중 확인 로직 추가

**문제 2: 세션 검증 성능**

- `getActiveSession()` 호출마다 배민 API 검증 → 동시 요청 시 충돌

**해결**: 메모리 상태만 확인, 실제 검증은 60초 간격 Health Check로 분리

**문제 3: 서버 재시작 시 세션 불일치**

- Redis 세션만 삭제하고 배민 측 로그아웃은 하지 않아 새 로그인 실패

**해결**: 서버 시작 시 배민 로그아웃 API 호출 후 Redis 세션 삭제

### 4. 라이더 상세 정보 개별 트랜잭션 저장

**배경**

- 라이더 상세 정보 배치 작업 중 일부 403 에러 발생 시 전체 롤백
- "저장 완료" 로그 출력 후에도 DB에 데이터 없음

**문제**

- `@Transactional` 메서드 전체가 하나의 트랜잭션
- 중간에 에러 발생 시 이미 저장한 데이터도 모두 롤백

**해결**

- `RiderDetailInfoSaveService` 신규 생성
- `@Transactional(propagation = Propagation.REQUIRES_NEW)` 적용
- 각 라이더별 try-catch로 실패해도 계속 진행
- 성공/실패 건수 로깅 추가

### 6. 배달비 배치 스케줄 최적화

**배경**

- 배민 측에서 API 호출 횟수 과다 컴플레인

**문제**

- 30분 간격 × 59개 협력사 × 48회/일 = 약 2,832회/일 호출
- 배치 스케줄이 "테스트용"으로 설정된 상태로 운영 중

**해결**

- 배치 스케줄: 30분 간격 → 매일 오전 10시, 오후 1시 (1일 2회)
- 호출 횟수: 2,832회/일 → 118회/일 (96% 감소)
- 협력사 배달비 다운로드 버튼 클릭 로직 비활성화 (배민 서버 부하 방지)

### 7. 어드민 계정 로그인 시 잔액 조회 N+1 문제 해결

**현상**

- 어드민 계정 로그인 시 `vr_account_mst` 테이블 조회 쿼리가 라이더 수만큼 반복 실행되는 N+1 문제 발견
- 로그 분석 결과, 라이더 목록 조회 시 각 라이더마다 개별적으로 잔액을 조회하는 패턴 확인

**원인**

- 라이더 목록 렌더링 시 각 항목마다 별도의 잔액 조회 쿼리 발생
- 어드민(`prm_code = 'A'`) 계정에서도 불필요하게 잔액 컴포넌트가 렌더링

**해결**

프론트엔드 (앱)

- `DeliveryPage.tsx`에서 `prmCode='A'`인 경우 `BalanceDisplay` 컴포넌트 렌더링 제외
- 어드민 계정에서 불필요한 잔액 조회 쿼리 완전 제거

백엔드 (포털 API)

- `VrAccountMstQueryDslRepository`에 배치 조회 메서드 추가 (`findLatestBalAmtByRiderIds`)
- `RiderListController`에서 개별 조회 대신 배치 조회 방식으로 전환
- N+1 쿼리를 단일 IN 쿼리로 최적화

## 미해결 이슈

### 자동 재인증 SMS 웹훅 간헐적 미수신

**현상**

- 배포 후 2FA 인증이 간헐적으로 실패
- Macrodroid가 SMS 웹훅 전송 후 200 응답을 받았으나 서버에서 미수신

**원인 추정**

- DNS 캐싱: Macrodroid가 이전 인스턴스 IP를 캐시
- 로드밸런서 라우팅 지연: 새 인스턴스로 트래픽 전환에 시간 소요

**검토 중인 해결 방안**

1. Macrodroid 웹훅 URL에 IP 직접 사용
2. 배포 후 재인증 지연 (1-2분 대기)
3. 재인증 실패 시 재시도
