# vr_rider_info_his 계약 상태 변동 이력 저장 방식 변경

## 1. 변경 개요

`vr_rider_info_his` 테이블에 라이더 이력을 저장하는 조건을 변경하였습니다.

| 구분 | 변경 전 | 변경 후 |
|------|---------|---------|
| 저장 조건 | 계약 상태가 **계약종료**일 때만 저장 | 계약 상태가 **변동될 때마다** 저장 |

---

## 2. 변경 이유

### 2.1 라이더 주정산 일차감 계산 로직과의 연관성

라이더 주정산서에서 **일차감(공제)** 을 계산할 때, `vr_rider_info_his` 테이블의 계약 상태 변동 이력을 참조합니다.

일차감 계산 시 정산 기간 내 **유효 계약 일수**를 산정하는데, 이 과정에서 다음 두 가지 상태 전환을 감지해야 합니다.

```
계약 중 → 계약종료   : 종료일 이전까지만 일차감 적용
계약종료 → 계약 중   : 재계약 이후 다시 일차감 적용 재개
```

### 2.2 기존 방식의 문제점

기존에는 **계약종료 상태일 때만** `vr_rider_info_his`에 이력을 남겼기 때문에, 아래와 같은 시나리오에서 문제가 발생했습니다.

```
[문제 시나리오]

1. 라이더가 계약종료 상태가 됨
   → vr_rider_info_his에 이력 저장 ✅

2. 라이더가 다시 계약 중 상태로 변경됨 (재계약)
   → vr_rider_info_his에 이력 저장 안 됨 ❌

3. 주정산 일차감 계산 시
   → 재계약 사실을 감지하지 못함
   → 일차감이 있어도 적용되지 않음 ❌
```

즉, **계약종료 → 계약 중** 전환이 이력에 남지 않아 재계약 이후 정산 기간에 일차감이 정상적으로 적용되지 않는 버그가 존재했습니다.

---

## 3. 변경 후 동작 방식

계약 상태가 변동될 때마다 `vr_rider_info_his`에 이력을 저장합니다.

```
[변경 후 시나리오]

1. 라이더가 계약종료 상태가 됨
   → vr_rider_info_his에 이력 저장 ✅

2. 라이더가 다시 계약 중 상태로 변경됨 (재계약)
   → vr_rider_info_his에 이력 저장 ✅

3. 주정산 일차감 계산 시
   → 재계약 사실을 정상 감지
   → 재계약일 이후 일차감 정상 적용 ✅
```

---

## 4. 일차감 계산에서의 이력 활용 흐름

```
┌──────────────────────────────────────────────────────────────┐
│              주정산 일차감 계산 흐름                           │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│  1. vr_rider_info_his 조회                                   │
│     - 정산 기간 내 해당 라이더의 계약 상태 변동 이력 조회       │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│  2. 계약 상태 분석                                            │
│     - 계약종료 이력 있음 → 종료일 이전까지만 일차감 적용        │
│     - 재계약(계약종료→계약 중) 이력 있음 → 재계약일부터 재적용  │
│     - 이력 없음 → 정산 기간 전체 일수 적용                    │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│  3. 유효 계약 일수 계산                                       │
│     - 계약 중인 기간의 일수만 합산                             │
│     - WORK_DAY: 유효 기간 내 실제 배달 완료 이력 기준 근무일수  │
│       (재계약 케이스: 종료 전 구간 + 재계약 후 구간 각각 조회)  │
│     - DAILY: 유효 기간 전체 일수 기준                         │
│     - 계약종료 당일은 유효 일수에서 제외                       │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│  4. 일차감 금액 산출                                          │
│     - 일차감 단가 × 유효 계약 일수                            │
└──────────────────────────────────────────────────────────────┘
```

## 5. 관련 테이블

### 5.1 vr_rider_info_his (라이더 정보 이력)

계약 상태 변동 시 스냅샷 형태로 저장되는 이력 테이블입니다.

| 주요 컬럼 | 설명 |
|----------|------|
| SEQ_NO | 시퀀스 (PK) |
| RIDER_ID | 라이더 ID |
| CONT_STATUS | 계약 상태 (UNDER_CONTRACT / TERMINATED_CONTRACT 등) |
| CONT_ED_DATE | 계약 종료일시 |
| INS_DATE | 이력 저장일시 (상태 변경 시점 기준) |

### 5.2 연관 테이블

| 테이블 | 설명 |
|--------|------|
| `vr_rider_info` | 라이더 현재 정보 |
| `rider_acct_weekly` | 라이더 주정산 내역 (일차감 컬럼: `DEDUCT_AMT`) |

---

## 6. 영향 범위

| 기능 | 영향 |
|------|------|
| 라이더 주정산 일차감 계산 | 재계약 라이더의 일차감이 정상 적용됨 |
| 라이더 계약 상태 변경 처리 | 모든 상태 변동 시 이력 저장 |
| 기존 계약종료 이력 | 기존 데이터 그대로 유지, 추가 저장만 확대 |

---

## 7. 주의사항

- 이력 저장 빈도가 증가하므로 `vr_rider_info_his` 테이블의 데이터 증가량을 모니터링해야 합니다.
- 동일 상태로의 중복 변경(예: 계약 중 → 계약 중)은 이력 저장 대상에서 제외하는 것을 권장합니다.
- 기존에 재계약 이력이 누락된 라이더의 경우, 과거 정산 데이터는 소급 적용되지 않습니다.
