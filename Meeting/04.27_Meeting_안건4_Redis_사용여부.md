## [안건 4] Redis 사용 여부 검토

---

### 1. 현재 Redis 사용 현황

#### 1-1. 연결 구조

```
Java API 서버 (Spring Boot)
  └── RedisConfig.java
      ├── LettuceConnectionFactory (host/port)
      ├── RedisTemplate<String, String>      ← 세션 저장
      └── ReactiveRedisTemplate<String, Object>

Node.js Relay 서버
  └── server.js
      └── redis v4.6.10 (AWS_REDIS_URL)
```

두 서버가 동일한 Redis 인스턴스를 공유하여 세션 상태를 동기화합니다.

---

#### 1-2. 용도별 상세

| 용도           | 구현체                                  | 키 패턴                            | TTL  | 비고                                |
| ------------ | ------------------------------------ | ------------------------------- | ---- | --------------------------------- |
| 배민 로그인 세션 저장 | `RedisBaeminSessionRepository.java`  | `vroong:baemin:session:*`       | 24시간 | **핵심**                            |
| 배민 세션 상태 추적  | 동일                                   | `vroong:baemin:status:*`        | 24시간 | PENDING / AUTHENTICATED / EXPIRED |
| 활성 세션 목록     | 동일                                   | `vroong:baemin:active_sessions` | 24시간 | Set 자료구조                          |
| 파트너 포털 세션    | `RedisPartnerSessionRepository.java` | 유사 구조                           | 24시간 |                                   |
| 센터 목록 캐싱     | `relay-server/server.js`             | (인메모리)                          | 6시간  | Redis 아님 — 로컬 캐시                  |

---

#### 1-3. 세션 저장에 Redis가 필요한 이유

```
배민 로그인 흐름:

Java API 서버                relay-server (Puppeteer)
     │                              │
     │── 로그인 요청 ──────────────>│
     │                    Chrome 자동 로그인
     │<── sessionId 반환 ───────────│
     │
     │── Redis에 sessionId 저장 ──> Redis
     │
     │ (다음 배치 실행 시)
     │── Redis에서 sessionId 조회 → relay-server로 API 호출
```

- relay-server는 **Node.js 프로세스**이고 API 서버는 **Java 프로세스**
- 두 프로세스 간 로그인 세션 공유를 위해 외부 저장소(Redis)가 필요
- Redis 없이는 배치 실행 시마다 재로그인 필요 → 크롤링 지연, 봇 탐지 위험 증가

### 2. Redis 제거 시 대안 검토

| 대안                     | 구현 방법               | 문제점                                                                                                                                              |
| ---------------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **DB(MySQL) 저장**       | `vr_session` 테이블 생성 | RDS I/O 증가, 세션 조회 빈도가 높아 부하 발생 가능                                                                                                                |
| **relay-server 내 메모리** | `server.js` 로컬 Map  | API 서버와 상태 공유 불가, relay-server 재시작 시 세션 소실 (relay-server 재시작 시 현재도 세션 소실이라 구현 가능 여부 확인 필요) (그 전에는 서버 재시작 시에 로그인 상태 유지 BrightData 방식이라 redis로 구현) |
| **API 서버 내 메모리**       | Spring 캐시           | relay-server와 프로세스 분리 → 공유 불가                                                                                                                    |
| **파일 기반**              | JSON 파일 저장          | ECS Fargate 환경에서 영구 스토리지 없음                                                                                                                      |

**결론**: 현재 아키텍처(Java API + Node.js relay-server의 2-프로세스 구조)에서 Redis는 필수 의존성입니다. 단순 제거는 어렵습니다.

---

### 3. 현재 Redis 운영 상태 점검

#### 확인 필요 사항

| 항목   | 현재 상태                                  | 권장                           |
| ---- | -------------------------------------- | ---------------------------- |
| 인프라  | AWS ElastiCache인지 직접 EC2 Redis인지 확인 필요 | ElastiCache 권장 (자동 failover) |
| 인증   | 비밀번호 없음 (VPC 격리 의존)                    | ElastiCache AUTH 토큰 설정 권장    |
| 고가용성 | Single node인지 Multi-AZ인지 확인 필요         | Redis 다운 시 배치 전체 중단          |
| 모니터링 | CloudWatch 메트릭 연결 여부 확인 필요             | 메모리 사용률, 연결 수 알림 설정          |

---

### 4. Redis 의존성을 줄이는 방향 (장기)

Redis를 완전 제거하기보다 의존 범위를 줄이는 것이 현실적입니다.

| 현재                  | 변경 방향                                     |
| ------------------- | ----------------------------------------- |
| 세션 전체 객체를 Redis에 저장 | sessionId만 Redis에 저장, 세션 상세는 DB           |
| TTL 기반 세션 만료        | DB 컬럼(`expired_at`)으로 관리 가능               |
| 두 서버 간 상태 동기화       | relay-server에 REST API 추가 → API 서버가 직접 조회 |

이 방향으로 리팩토링하면 Redis 장애 시에도 DB 기반으로 동작 가능해져 Redis가 선택적 의존성이 됩니다.

---

### 5. 결론 및 권장 사항

**단기 (현 상태 유지)**: 제거보다 안정화 우선

1. Redis 인프라 확인: AWS ElastiCache인지, Single node인지 확인
2. AUTH 토큰 설정: 비밀번호 없는 Redis 접근 제한 (B-05 보안 이슈와 연계)
3. CloudWatch 알림: Redis 다운 시 배치 실패 즉각 감지

**장기 (아키텍처 개선)**: relay-server와 Java API 서버 간 통신 구조를 직접 REST 호출로 개선하면 Redis 의존성 축소 가능.
