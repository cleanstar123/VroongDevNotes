#### 2.3 문제 원인 3가지

**① `xhr.js` — Unhandled Promise Rejection**

`triggerAndWaitForXhr` 함수에서 `triggerFn()` 실패 시 `xhrPromise`가 고아 Promise가 되어, 30초 후 reject 시 전역으로 Unhandled Promise Rejection이 전파됨.

```javascript
// 문제 코드 (before)
async function triggerAndWaitForXhr(page, triggerFn, xhrPattern, timeout = 30000) {
    const xhrPromise = waitForXhrResponse(page, xhrPattern, timeout);
    await triggerFn();          // ← 여기서 throw 시
    const result = await xhrPromise; // xhrPromise 고아 상태로 방치
    return result;
}
```

**② `BrowserManager` — Execution context 파괴 후 자동 복구 없음**

`enqueueRequest` 내에서 `Execution context was destroyed` 오류를 감지하고 로그만 남길 뿐, 브라우저 페이지를 재시작하는 로직이 없었음.

**③ Keep-Alive — 브라우저 파괴 상태를 "살아있음"으로 오탐**

Navigation timeout 발생 시 "일시적 오류"로 판단하고 세션을 Redis에 계속 유효 상태로 저장.  
결과적으로 브라우저 컨텍스트가 파괴된 상태에서도 백엔드는 릴레이가 정상이라 판단하고 계속 요청을 전송함.  
→ 장애 감지가 불가능하고 알림도 발송되지 않음.

### 3. 해결 방안

#### 3.1 `relay-server/services/browser/xhr.js`

`triggerAndWaitForXhr`에 try-catch 추가 → `triggerFn()` 실패 시 xhrPromise 고아 방지

```javascript
// 수정 후 (after)
async function triggerAndWaitForXhr(page, triggerFn, xhrPattern, timeout = 30000) {
    const xhrPromise = waitForXhrResponse(page, xhrPattern, timeout);

    try {
        await triggerFn();
    } catch (triggerError) {
        xhrPromise.catch(() => {}); // 고아 Promise rejection 억제
        throw triggerError;
    }

    const result = await xhrPromise;
    return result;
}
```

> `navigateAndWaitForXhr`는 이미 동일한 패턴(`xhrPromise.catch(() => {})`)으로 보호되어 있었음.  
> `triggerAndWaitForXhr`만 누락되어 있었던 것.

#### 3.2 `relay-server/services/browser/core.js`

**`restartSessionPage()` 함수 신규 추가**:  
Incognito 컨텍스트(쿠키)는 유지한 채 Puppeteer 페이지만 새로 생성

```javascript
async function restartSessionPage(sessionId) {
    const session = sessionPages.get(sessionId);
    if (!session) return false;

    // 기존 페이지 닫기 (에러 무시)
    if (session.page && !session.page.isClosed()) {
        await session.page.close().catch(() => {});
    }

    // 같은 컨텍스트에서 새 페이지 생성 (쿠키 자동 공유)
    const newPage = await session.context.newPage();
    await setupAntiDetection(newPage);
    session.page = newPage;

    return true;
}
```

**`enqueueRequest()` 수정**:  
`Execution context was destroyed` 감지 시 `restartSessionPage()` 자동 호출

```javascript
} catch (error) {
    const isContextDestroyed = error.message.includes('Execution context was destroyed');
    if (isContextDestroyed) {
        await restartSessionPage(sessionId); // 자동 페이지 재시작
    }
    throw error;
}
```

#### 3.3 `relay-server/server.js` — Keep-Alive 연속 타임아웃 감지

세션별 연속 타임아웃 카운터 추가 → **3회 연속 시 이메일 알림 발송**

```javascript
// 전역 카운터 (세션별)
const sessionNavTimeoutCount = new Map();
const NAV_TIMEOUT_ALERT_THRESHOLD = 3;

// keepSessionAlive 내 타임아웃 처리 시
if (isTemporaryError) {
    const newCount = (sessionNavTimeoutCount.get(session.sessionId) || 0) + 1;
    sessionNavTimeoutCount.set(session.sessionId, newCount);

    if (newCount >= NAV_TIMEOUT_ALERT_THRESHOLD) {
        sessionNavTimeoutCount.set(session.sessionId, 0); // 리셋
        mailService.sendCrawlingFailureAlert(session.sessionId, error.message, newCount);
    }
}

// Keep-Alive 성공 시 카운터 리셋
if (result && result.data) {
    sessionNavTimeoutCount.set(session.sessionId, 0);
    ...
}
```

#### 3.4 `relay-server/services/mail-service.js` — 크롤링 장애 알림 추가

기존 세션 만료 알림과 동일한 이메일 방식으로 `CRAWLING_FAILURE` 타입 신규 추가

```javascript
const sendCrawlingFailureAlert = (sessionId, errorMessage, count) => {
    const detail = `연속 ${count}회 Navigation/XHR 타임아웃 발생\n마지막 오류: ${errorMessage}`;
    return sendErrorAlert('CRAWLING_FAILURE', null, detail, { sessionId });
};
```

**발송 조건**: Navigation timeout 또는 XHR timeout 연속 3회 (약 3분 이상 지속 시)  
**발송 방식**: 기존 SESSION_EXPIRED 알림과 동일한 SMTP 이메일  
**중복 방지**: 기존 `ALERT_COOLDOWN` 메커니즘 그대로 적용 (10분 쿨다운)
