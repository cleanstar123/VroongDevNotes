## 1. RDS 처리량 크레딧 소진으로 인한 DB 응답 지연

> [!info] 배경
> 
> - `vraccountdb` RDS 인스턴스는 **범용 SSD (gp2)** 스토리지 타입을 사용 중
> - gp2는 버스트 크레딧 방식으로 처리량(Throughput)을 관리하며, 크레딧이 소진되면 베이스라인 수준으로 처리량이 급격히 제한됨
> - Quartz 배치 작업이 자정(00:00), 새벽(03:00), 업무 시간(09:00~) 등 주기적으로 실행되며 대량의 Write 작업을 발생시키는 환경

> [!warning] 문제
> 
> - **발생 시각:** 2026-06-24 13:05 ~ 13:20 (약 15분간)
> - **증상:** DBeaver에서 DB 접속 시 무한 로딩, API 응답 극도로 지연
> - **CloudWatch 지표 이상:**
>     - `EBSByteBalance%` : 13:05에 **0%** 도달 (처리량 크레딧 완전 소진)
>     - `DiskQueueDepth` : **0.949** (I/O 대기 발생)
>     - `FreeableMemory` : **93~94MB** (낮은 수준)
>     - CPU, DB Connections : 정상 범위
> - **PROCESSLIST 확인 결과:** 실행 중인 무거운 쿼리 없음 → 쿼리 자체 문제 아님
> - **WriteThroughput 그래프 분석:**
>     - 00:00에 9.527MB/s 최고 스파이크 (자정 배치)
>     - 03:00에 7.54MB/s 스파이크 (새벽 배치)
>     - 09:00~10:00에 3~4MB/s 반복 스파이크 (업무 시간 배치)
>     - 자정부터 누적된 Write 처리량이 크레딧을 서서히 소진시킴
> - **근본 원인:** 특정 이벤트 단일 원인이 아닌, **Quartz 배치 Write 작업의 누적 소진**으로 13:05에 크레딧 완전 바닥

> [!success] 해결
> 
> - 13:20경 부하 감소로 EBSByteBalance% 크레딧이 자동 충전되어 자연 회복 (99%로 급등)
> - DB 접속 및 API 응답 정상화 확인
> - **근본 해결은 미완료 → gp3 변경 필요 (TODO 참고)**

> [!tip] 배운 점
> 
> - **gp2 스토리지의 구조적 취약점:** 버스트 크레딧 소진 시 처리량이 베이스라인으로 제한되어 쿼리가 없어도 DB 전체가 느려짐. CPU·커넥션 수 정상이어도 장애 발생 가능
> - **장애 진단 우선순위:** CPU/Connection 정상이면 EBSByteBalance%, DiskQueueDepth를 먼저 확인
> - **CloudWatch 지표 해석:**
>     - `EBSByteBalance% = 0` → 처리량 크레딧 소진, I/O 처리량 제한 상태
>     - `DiskQueueDepth > 0` → I/O 대기 발생 중, 스토리지 병목 신호
>     - `SHOW PROCESSLIST` 에서 무거운 쿼리 없을 때 스토리지 지표를 의심할 것
> - **gp2 → gp3 전환 필요성:** gp3는 크레딧 개념 없이 128MB/s 처리량을 고정 제공하며 비용도 약 20% 저렴. 배치 작업이 많은 환경에서 gp2는 부적합

## 2. Quartz 배치 스케줄 분석 — WriteThroughput 스파이크 원인 특정

> [!info] 배경
> 
> - `QRTZ_TRIGGERS`, `QRTZ_CRON_TRIGGERS` 테이블 조회를 통해 등록된 전체 배치 스케줄 확인
> - 자정(00:00) 및 03:00 실행 작업을 필터링하여 WriteThroughput 스파이크 원인 특정

> [!warning] 문제 — 주요 원인 배치 2개 특정

> **00:00 — `balanceSnapshotJob`** `0 0 0 * * ?`
> 
> - 매일 자정 정각 실행 (batchGroup)
> - 전체 라이더 잔액 스냅샷을 찍는 작업으로 추정 (`vr_account_mst` 전체 읽기 → 스냅샷 테이블 INSERT)
> - WriteThroughput 그래프 **9.527MB/s 최고 스파이크** 주범
> 
> **03:00 — `tableBackupJob`** `0 0 3 * * ?`
> 
> - 매일 03:00 정각 실행 (backupGroup)
> - 테이블 백업 작업 (`INSERT INTO ... SELECT` 방식으로 추정)
> - WriteThroughput 그래프 **7.54MB/s 두 번째 스파이크** 주범
> 
> **자정~새벽 동시 실행 배치 (크레딧 누적 소진 가중)**
> 
> |시간|Job|Cron|
> |---|---|---|
> |00:05|`maintenanceEndJob`|`0 5 0 * * ?`|
> |00:05|`batchWithdrawalJob`|`0 5 0 * * ?`|
> |00:10~매10분|`riderDeliveryHistoriesNowJob`|`0 0/10 0-2 * * ?`|
> |00:10~매10분|`riderDeliveryHistoriesNowIfJob`|`0 0/10 0-2 * * ?`|
> |00:10|`riderInfoJob`|`0 10 0-3,7-23 * * ?`|
> |03:00~매10분|`riderDeliveryHistoriesNowJob`|`0 0,10,20,30 3 * * ?`|
> |03:00~매10분|`riderDeliveryHistoriesNowIfJob`|`0 0,10,20,30 3 * * ?`|

> [!success] 원인 타임라인 재구성
> 
> ```
> 23:50  maintenanceStartJob 실행
> 00:00  balanceSnapshotJob (9.5MB/s) ← 최대 Write 스파이크
> 00:05  batchWithdrawalJob
> 00:10~ riderDeliveryHistoriesNowJob (10분마다, 03:30까지 지속)
> 03:00  tableBackupJob (7.5MB/s) ← 두 번째 Write 스파이크
>        ↓
> 새벽 내내 크레딧 누적 소진
>        ↓
> 업무 시간에도 크레딧 회복 불가
>        ↓
> 13:05  EBSByteBalance% = 0% 도달 → DB 응답 지연 장애 발생
> ```

> [!tip] 배운 점
> 
> - **배치 집중 시간대 파악의 중요성:** `QRTZ_CRON_TRIGGERS` 조회만으로 Write 부하 집중 구간을 사전에 파악 가능
> - **단일 배치가 아닌 누적 효과:** `balanceSnapshotJob` 하나가 아니라 자정~새벽 배치들이 겹쳐서 실행되며 크레딧을 지속 소진하는 구조
> - **gp2 환경에서 대량 Write 배치는 구조적 위험:** 배치 스케줄 분산이나 쿼리 최적화로 완화 가능하나, gp3 전환이 근본적인 해결책
