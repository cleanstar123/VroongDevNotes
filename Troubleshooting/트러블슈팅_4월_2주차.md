## [이슈 1] ECS 태스크 재시작 - HikariCP MySQL 커넥션 타임아웃

---

### 1. 문제 상황

**발생 일자**: 2026-04-10

**발생 시간**: 14:10 ~ 14:17 (UTC+9)

ECS 서비스(`vroong-api-service`)에서 태스크가 unhealthy 판정을 받아 강제 교체됨.

```
14:10 - Amazon ECS replaced 2 tasks due to an unhealthy status.
14:13 - task 0c9d91d2e8f84176b3c92619aa574152 port 8080 is unhealthy
        due to (reason Request timed out)
14:17 - Amazon ECS replaced 1 tasks due to an unhealthy status.
```

---

### 2. 원인 분석

#### 2.1 CloudWatch 로그에서 확인된 핵심 에러

```
14:17:54 WARN  HikariPool-1 - Failed to validate connection
                (No operations allowed after connection closed.)
                Possibly consider using a shorter maxLifetime value.

14:17:57 ERROR AsyncRequestTimeoutException
                /api/v1/rider/work-status?fromDate=20260403&toDate=20260410

14:18:53 ERROR Communications link failure
                The last packet successfully received from the server
                was 656,508 milliseconds ago.  ← 약 10.9분
```

#### 2.2 발생 메커니즘

```
1. DB 커넥션이 오랫동안 idle 상태 유지
2. MySQL이 서버 측에서 idle 커넥션 강제 종료 (wait_timeout ≈ 600초, 10분)
3. HikariCP는 이를 즉시 감지하지 못하고 끊긴 커넥션을 pool에 보유
4. Actuator 헬스체크(/actuator/health)가 DB 상태 확인 시 끊긴 커넥션 사용 → 타임아웃
5. ECS → "Request timed out" → unhealthy 판정 → 태스크 강제 교체
```

#### 2.3 근본 원인: HikariCP 설정값 문제

| 설정 | 기존값 | 문제 |
|---|---|---|
| `max-lifetime` | 1,800,000ms (30분) | MySQL wait_timeout(10분)보다 훨씬 길어 끊긴 커넥션 장기 보유 |
| `idle-timeout` | 600,000ms (10분) | MySQL wait_timeout과 동일 → 경쟁 조건 발생 |
| `keepalive-time` | 미설정 | idle 커넥션에 주기적 ping 없어 MySQL이 커넥션 끊음 |

---

### 3. 해결 방안

#### 3.1 HikariCP 설정 수정

`application.yml`, `application-local.yml`, `application-dev.yml` 모두 수정

```yaml
datasource:
  hikari:
    idle-timeout: 540000      # 9분 (max-lifetime보다 짧아야 함)
    max-lifetime: 580000      # 9분 40초 (MySQL wait_timeout ~10분보다 짧게)
    keepalive-time: 300000    # 5분마다 keepalive 쿼리 실행 (핵심 수정)
    connection-test-query: SELECT 1
```

#### 3.2 변경 요약

| 항목 | 기존 | 변경 | 이유 |
|---|---|---|---|
| `max-lifetime` | 1,800,000 (30분) | 580,000 (9분 40초) | MySQL 커넥션 종료 전에 Hikari가 먼저 교체 |
| `idle-timeout` | 600,000 (10분) | 540,000 (9분) | max-lifetime보다 짧아야 함 |
| `keepalive-time` | 없음 | 300,000 (5분) | 5분마다 SELECT 1 실행해 MySQL idle 판단 방지 |

---

### 4. 추가 확인 사항

- MySQL `wait_timeout` 확인 쿼리:
  ```sql
  SHOW VARIABLES LIKE 'wait_timeout';
  ```
  → 결과값보다 `max-lifetime`을 최소 30초 이상 짧게 설정

- `keepalive-time`은 반드시 `max-lifetime`보다 짧아야 함

---

### 5. 재발 방지

- [ ] application.yml, application-local.yml, application-dev.yml HikariCP 설정 수정 후 배포
- [ ] CloudWatch 알람 설정: HikariPool WARN 로그 발생 시 알림

## [이슈 2] 라이더 일차감 협력사 주정산 공제금액 미반영

---

### 1. 문제 상황

**발생 일자**: 2026-04-10 확인 (실제 원인 발생: 2026-02-03 추정)

협력사 주정산 Excel 업로드 시 라이더(kimgujin12)의 일차감(DAILY 타입 공제)이 협력사 공제금액(DEDUCT_VAL)에 반영되지 않음.

- `vr_partner_acct_weekly` 조회 결과: `DEDUCT_VAL = 0`
- 해당 라이더에게는 활성 상태의 DAILY 일차감이 존재함

---

### 2. 원인 분석

#### 2.1 공제금액 계산 흐름

```
PartnerSettlementUseCase.calculateDeductionOptimized
→ calculateRiderEffectiveDays(kimgujin12)
→ contractPeriodInfoMap 조회 (vr_rider_info_his 기반)
→ 최신 이력이 TERMINATED_CONTRACT (2026-01-21)
→ terminatedDate(01-21) < acctStDate(04-01) → 유효일수 = 0
→ DEDUCT_VAL = 0
```

#### 2.2 vr_rider_info_his 상태

| INS_DATE | CONT_STATUS |
|---|---|
| 2026-01-21 18:49:05 | UNDER_CONTRACT (변경 전) |
| 2026-01-21 18:50:05 | TERMINATED_CONTRACT (변경 후) |
| **재계약 이력 없음** | ← 문제 |

`vr_rider_info`에는 CONT_STATUS = UNDER_CONTRACT로 되어 있지만, `vr_rider_info_his`에는 재계약 이력이 존재하지 않아 시스템이 해당 라이더를 "정산 기간 내 계약 없는 라이더"로 판단.

#### 2.3 재계약 이력이 누락된 이유

**`handleUnderContractStatus` 트랜잭션 구조 문제:**

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void handleUnderContractStatus(...) {
    saveRiderInfoHis(beforeHis);        // 1. 이력 저장
    saveRiderInfoHis(afterHis);         // 2. 이력 저장
    clearMemberWithdrawalFields(...);   // 3. 탈퇴 필드 초기화
    assignVirtualAccountToRider(...);   // 4. 가상계좌 할당 ← 예외 발생 시
}                                       //    1~3도 함께 롤백
```

4번에서 예외 발생 시 `REQUIRES_NEW` 트랜잭션 전체 롤백 → 이력(1,2)도 사라짐.

`handleContStatusChange`의 catch에서 예외를 로그만 남기고 계속 진행.

이후 배치 후반부의 **가상계좌 자동 할당 루프**가 실행:

```java
// 가상계좌 미할당 UNDER_CONTRACT 라이더 자동 할당 (이력 저장 없음)
vrRiderInfoQueryDslRepository.assignVirtualAccountToRider(suppId, riderId, now, "BATCH");
```

→ 가상계좌만 할당됨 (UPD_DATE = 2026-02-03 18:50:05, UPD_USER = BATCH), 이력은 복구 안 됨.

**증거**: `vr_virtual_account_list` 조회 결과
- `INS_DATE = NULL` (원래부터 있던 미할당 계좌)
- `UPD_DATE = 2026-02-03 18:50:05`, `UPD_USER = BATCH` (자동 할당 루프가 UPDATE)

---

### 3. 해결 방안

#### 3.1 즉시 수습 (데이터 수동 보정)

`vr_rider_info_his`에 누락된 재계약 이력 수동 INSERT 필요.

#### 3.2 코드 수정 (재발 방지)

`handleUnderContractStatus`에서 이력 저장과 가상계좌 할당을 별도 트랜잭션으로 분리.

```
현재: [이력저장 + 가상계좌할당] → 하나의 REQUIRES_NEW 트랜잭션
수정: [이력저장] REQUIRES_NEW 트랜잭션
     [가상계좌할당] 별도 REQUIRES_NEW 트랜잭션
```

→ 가상계좌 할당 실패 시에도 이력은 보존됨.
