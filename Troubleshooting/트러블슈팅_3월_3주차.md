# 트러블슈팅 3월 3주차 - 협력사 수수료 계산 로직 수정

## 1. 문제 상황

**발생 일자**: 2026-03-16

정산 기간 중 수수료가 변경된 경우, 전체 배달 건수에 최신 수수료만 적용되어 잘못된 수수료가 계산됨.

### 1.1 구체적 사례

- **라이더**: rmsdlf1
- **정산 기간**: 3/4 ~ 3/10
- **수수료 변경**: 3/8까지 건당 20원 → 3/9부터 건당 120원
- **총 배달 건수**: 130건

**잘못된 계산 (변경 전)**:
```
130건 × 120원 = 15,600원
```

**올바른 계산 (변경 후)**:
```
(3/4~3/8 건수 × 20원) + (3/9~3/10 건수 × 120원)
```

### 1.2 실제 데이터 (vr_delivery_list)

| DELIVERY_DATE | FINISH_CNT |
|---------------|------------|
| 20260304 | 18 |
| 20260305 | 16 |
| 20260306 | 9 |
| 20260307 | 30 |
| 20260308 | 23 |
| 20260309 | 18 |
| 20260310 | 16 |

### 1.3 올바른 계산 결과

- 3/4~3/8: (18+16+9+30+23) = **96건** × 20원 = **1,920원**
- 3/9~3/10: (18+16) = **34건** × 120원 = **4,080원**
- **총합: 6,000원**

## 2. 원인 분석

### 2.1 기존 로직 문제점

`RiderSettlementUseCase.calculatePartnerFeeOptimized` 메서드에서 정산 기간 내에 유효한 수수료 1개만 선택하여 전체 배달 건수에 적용.

```java
// 기존 로직 - 문제점
private BigDecimal calculatePartnerFeeOptimized(...) {
    // 우선순위: R(라이더 특정) > RA(협력사 전체)
    FeeMst applicableFee = findApplicablePartnerFee(supplierFees, riderId, acctStDateStr, acctEdDateStr);

    if (applicableFee == null) return BigDecimal.ZERO;
    return calculateFeeAmount(applicableFee, deliveryCnt, deliveryVal);  // ← 전체 건수에 하나의 수수료만 적용
}
```

### 2.2 문제 발생 조건

- 정산 기간 내에 수수료 변경이 있는 경우 (START_DAY/END_DAY가 정산 기간 중간에 존재)
- `findApplicablePartnerFee`가 최신 수수료만 반환하여 이전 기간의 수수료가 무시됨

## 3. 해결 방안

### 3.1 수정 전략

1. **날짜별 배달 건수 조회**: `vr_delivery_list` 테이블에서 날짜별 FINISH_CNT 조회
2. **다중 수수료 필터링**: 정산 기간 내 적용 가능한 모든 수수료 목록 조회
3. **날짜별 수수료 적용**: 각 날짜에 유효한 수수료를 찾아 해당 날짜의 배달 건수에 적용
4. **합산**: 날짜별 계산된 수수료를 모두 합산

## 4. 수정 내용

### 4.1 VrDeliveryListQueryDslRepository.java - 신규 메서드 추가

**파일 경로**: `api/src/main/java/com/vroong/api/infrastructure/persistence/repository/vroong/VrDeliveryListQueryDslRepository.java`

```java
/**
 * 특정 라이더의 날짜별 배달 건수를 조회합니다.
 *
 * <p>수수료 계산 시 날짜별로 해당 시점의 수수료를 적용하기 위해 사용합니다.
 *
 * @param supplyId 협력사 ID
 * @param riderId 라이더 ID
 * @param fromDate 시작일 (yyyyMMdd)
 * @param toDate 종료일 (yyyyMMdd)
 * @return Map<날짜(yyyyMMdd), 배달건수>
 */
public Map<String, BigDecimal> findDailyFinishCntByRider(String supplyId, String riderId,
        String fromDate, String toDate) {

    List<Tuple> results = queryFactory
            .select(deliveryList.deliveryDate, deliveryList.finishCnt)
            .from(deliveryList)
            .where(deliveryList.supplyId.eq(supplyId)
                    .and(deliveryList.riderId.eq(riderId))
                    .and(deliveryList.deliveryDate.between(fromDate, toDate))
                    .and(deliveryList.finishCnt.gt(BigDecimal.ZERO)))
            .fetch();

    Map<String, BigDecimal> resultMap = new HashMap<>();
    for (Tuple tuple : results) {
        String date = tuple.get(deliveryList.deliveryDate);
        BigDecimal cnt = tuple.get(deliveryList.finishCnt);
        resultMap.put(date, cnt != null ? cnt : BigDecimal.ZERO);
    }
    return resultMap;
}
```

### 4.2 RiderSettlementUseCase.java - calculatePartnerFeeOptimized 메서드 수정

**파일 경로**: `api/src/main/java/com/vroong/api/domain/portal/settlement/usecase/RiderSettlementUseCase.java`

```java
/**
 * 협력사운영수수료 계산 (최적화 버전).
 *
 * <p>정산 기간 내 수수료 변경이 있는 경우, 날짜별로 해당 시점의 수수료를 적용합니다.
 * 예: 3/4~3/8 건당 20원, 3/9~ 건당 120원인 경우
 * - 3/4~3/8 배달 건수 × 20원 + 3/9~ 배달 건수 × 120원
 */
private BigDecimal calculatePartnerFeeOptimized(String supplyId, String riderId,
        BigDecimal deliveryCnt, BigDecimal deliveryVal, LocalDate acctStDate,
        LocalDate acctEdDate, List<FeeMst> supplierFees) {

    String acctStDateStr = acctStDate.format(DATE_FORMATTER);
    String acctEdDateStr = acctEdDate.format(DATE_FORMATTER);

    // 정산 기간 내에 적용 가능한 수수료들 필터링 (R 또는 RA 타입)
    List<FeeMst> applicableFees = filterApplicablePartnerFees(supplierFees, riderId, acctStDateStr, acctEdDateStr);

    if (applicableFees.isEmpty()) {
        return BigDecimal.ZERO;
    }

    // 수수료가 1개만 있으면 기존 로직 사용 (성능 최적화)
    if (applicableFees.size() == 1) {
        return calculateFeeAmount(applicableFees.get(0), deliveryCnt, deliveryVal);
    }

    // 정산 기간 내 수수료 변경이 있는 경우: 날짜별 배달 건수로 분할 계산
    Map<String, BigDecimal> dailyFinishCnt =
            vrDeliveryListQueryDslRepository.findDailyFinishCntByRider(supplyId, riderId, acctStDateStr, acctEdDateStr);

    if (dailyFinishCnt.isEmpty()) {
        // 날짜별 데이터가 없으면 기존 로직 사용
        FeeMst latestFee = findApplicablePartnerFee(supplierFees, riderId, acctStDateStr, acctEdDateStr);
        return latestFee != null ? calculateFeeAmount(latestFee, deliveryCnt, deliveryVal) : BigDecimal.ZERO;
    }

    // 날짜별로 해당 시점의 수수료 적용하여 합산
    BigDecimal totalFee = BigDecimal.ZERO;
    for (Map.Entry<String, BigDecimal> entry : dailyFinishCnt.entrySet()) {
        String dateStr = entry.getKey();
        BigDecimal dailyCnt = entry.getValue();

        // 해당 날짜에 유효한 수수료 찾기
        FeeMst feeForDate = findApplicablePartnerFeeForDate(applicableFees, riderId, dateStr);
        if (feeForDate != null) {
            BigDecimal dailyFee = calculateFeeAmount(feeForDate, dailyCnt, BigDecimal.ZERO);
            totalFee = totalFee.add(dailyFee);
        }
    }

    return totalFee;
}
```

### 4.3 신규 헬퍼 메서드

```java
/**
 * 정산 기간 내 적용 가능한 협력사 수수료 목록 필터링.
 * R 타입(특정 라이더)이 있으면 R만, 없으면 RA 타입 반환.
 */
private List<FeeMst> filterApplicablePartnerFees(
        List<FeeMst> fees, String riderId, String acctStDateStr, String acctEdDateStr) {
    // 1. R 타입 (특정 라이더) 필터링
    List<FeeMst> rFees = fees.stream()
            .filter(fee -> "R".equals(fee.getFeeTarget())
                && riderId.equals(fee.getRiderId())
                && isDateInRange(fee, acctStDateStr, acctEdDateStr))
            .collect(Collectors.toList());

    if (!rFees.isEmpty()) {
        return rFees;
    }

    // 2. RA 타입 (협력사 전체) 필터링
    return fees.stream()
            .filter(fee -> "RA".equals(fee.getFeeTarget())
                && isDateInRange(fee, acctStDateStr, acctEdDateStr))
            .collect(Collectors.toList());
}

/**
 * 특정 날짜에 유효한 협력사 수수료 찾기.
 * START_DAY <= 날짜 <= END_DAY인 수수료 반환.
 */
private FeeMst findApplicablePartnerFeeForDate(List<FeeMst> fees, String riderId, String dateStr) {
    LocalDate targetDate = LocalDate.parse(dateStr, DATE_FORMATTER);

    for (FeeMst fee : fees) {
        LocalDate startDay = fee.getStartDay();
        LocalDate endDay = fee.getEndDay();

        // 시작일/종료일이 null이면 무기한으로 처리
        boolean afterStart = (startDay == null) || !targetDate.isBefore(startDay);
        boolean beforeEnd = (endDay == null) || !targetDate.isAfter(endDay);

        if (afterStart && beforeEnd) {
            return fee;
        }
    }
    return null;
}
```

## 10. 추가 수정 사항 (2026-03-17)

### 10.1 수수료 변경 시 END_DAY 관리 방식 변경

**기존 로직 문제:**
- 수수료 변경 시 기존 수수료를 `USE_YN='N'`으로 비활성화
- 새 수수료는 수요일 기준으로 `START_DAY` 설정 (목~화 변경 시 다음 주 수요일)
- 기존 수수료 비활성화와 새 수수료 시작일 사이에 공백 발생

**예시 (3/11 화요일 수수료 변경):**
```
기존: 120원, USE_YN='N' (변경 시점에 비활성화)
신규: 100원, START_DAY=3/12(수), END_DAY=9999-12-31

→ 3/11 당일 정산 시 적용할 수수료 없음!
```

**수정된 로직:**
- `USE_YN='N'` 대신 `END_DAY`를 새 수수료 `START_DAY - 1`로 업데이트
- `USE_YN`은 'Y' 유지 (과거 정산 조회 가능)

**수정 후:**
```
기존: 120원, END_DAY=3/11, USE_YN='Y'
신규: 100원, START_DAY=3/12, END_DAY=9999-12-31, USE_YN='Y'

→ 3/11 정산: 120원 적용 (START_DAY <= 3/11 <= END_DAY)
→ 3/12~ 정산: 100원 적용
```

### 10.4 기존 데이터 마이그레이션

기존 `USE_YN='N'`, `END_DAY='9999-12-31'` 데이터를 새 로직에 맞게 수정

**마이그레이션 쿼리:**
```sql
UPDATE vr_fee_mst f
INNER JOIN (
    SELECT
        SEQ_NO,
        DATE_SUB(LEAD(START_DAY) OVER (PARTITION BY SUPP_ID, RIDER_ID ORDER BY START_DAY, SEQ_NO), INTERVAL 1 DAY) AS NEW_END_DAY,
        START_DAY
    FROM vr_fee_mst
    WHERE FEE_TYPE = 'S' AND FEE_TARGET = 'R'
) next_fee ON f.SEQ_NO = next_fee.SEQ_NO
SET f.END_DAY = next_fee.NEW_END_DAY,
    f.UPD_DATE = NOW(),
    f.UPD_USER = 'MIGRATION'
WHERE f.FEE_TYPE = 'S'
  AND f.FEE_TARGET = 'R'
  AND f.END_DAY = '9999-12-31'
  AND next_fee.NEW_END_DAY IS NOT NULL
  AND next_fee.NEW_END_DAY >= f.START_DAY;
```

**결과:** 약 130건 업데이트 완료
