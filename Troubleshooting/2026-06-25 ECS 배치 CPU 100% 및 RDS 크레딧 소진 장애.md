## 1. ECS 배치 CPU 100% 스파이크 및 RDS T-class CPU 크레딧 소진

> [!info] 배경
>
> - `vroong-3pl-platform-batch-prd` ECS 태스크 1개가 `vraccountdb` RDS (db.t4g.micro)와 동일한 인스턴스를 사용
> - 개발 배치 (`vroong-3pl-platform-batch-dev`)도 같은 RDS의 `vraccountdev` DB를 사용 중 — 실질적으로 개발/운영이 동일 RDS 공유
> - RDS db.t4g.micro: max_connections ≈ 85, T-class CPU 버스트 크레딧 방식

> [!warning] 문제
>
> - **발생 시각:** 2026-06-25 11:35 ~ 11:50 KST (약 15분)
> - **증상:** RDS CPU 100%, CPU 크레딧 잔액 급감, API 응답 지연
> - **CloudWatch 지표:**
>     - `CPUUtilization`: 11:35~11:50 구간 100% 도달
>     - `CPUCreditBalance`: 해당 구간에서 크레딧 급격 소진
>     - `DatabaseConnections`: 이전보다 높은 수치 (커넥션 풀 과다 프로비저닝 영향)
> - **11:59 로그에 `balanceSnapshotTrigger` 등록 로그 확인** → Job 실행이 아닌 ECS 태스크 교체 후 Spring Boot 재기동으로 인한 Quartz 트리거 재등록 (`autoScheduleAllJobs` 빈 기동 시 "Trigger 등록 완료" 로그 출력)
> - **근본 원인 3가지:**
>     1. 개발 배치 서비스가 같은 RDS를 바라보며 동시 실행 (Quartz Job 이중 실행)
>     2. HikariCP `maximum-pool-size: 200` 과다 설정 → API 3태스크 × 200 = 600 커넥션 시도 (DB 한계 85개 초과)
>     3. `balanceSnapshotJob` — JPA `saveAll()`이 `GenerationType.IDENTITY`로 인해 N개 개별 INSERT 발생 → 대량 Write 부하

> [!success] 해결
>
> **즉시 조치**
> - ECS `vroong-3pl-platform-batch-dev` 원하는 태스크 수 → **0** 설정 (개발 배치 중단)
> - 개발 배치 QRTZ 테이블 데이터 삭제 불필요 (태스크 중단으로 충분)
>
> **코드 수정 (vroong-backend, dev 브랜치)**
> - `application.yml`, `application-dev.yml`, `application-test.yml` 커넥션 풀 수정:
>     - `maximum-pool-size: 200` → `${DATABASE_MAX_POOL_SIZE:15}`
>     - `minimum-idle: 10` → `${DATABASE_MIN_IDLE:3}`
> - `application-dev.yml` / `application-test.yml` SQL 로깅 제거:
>     - `com.vroong: DEBUG` → `INFO`
>     - `org.hibernate.type.descriptor.sql.BasicBinder: TRACE` → `WARN` 제거 (운영 로그 폭증 방지)
> - `BalanceSnapshotUseCase.java` — `vrBalanceSnapshotRepository.saveAll()` → `JdbcTemplate.batchUpdate()` (500건 단위 배치 INSERT)

> [!tip] 배운 점
>
> - **T-class RDS 주의:** db.t4g.micro/small은 CPU 버스트 크레딧 방식. 단기 스파이크에는 견디지만 크레딧 소진 시 CPU 제한 → 배치 부하가 집중되는 환경에서 구조적 위험
> - **Quartz 트리거 등록 로그 ≠ Job 실행:** Spring Boot 기동 시 `autoScheduleAllJobs`가 DB에 없는 트리거를 등록하면서 "Trigger 등록 완료" 로그 출력. 장애 원인으로 오인하지 말 것
> - **`GenerationType.IDENTITY`와 JPA 배치 INSERT:** `saveAll()`은 IDENTITY 전략에서 배치 최적화 불가 → N개 개별 INSERT 발생. 대량 INSERT는 `JdbcTemplate.batchUpdate()` 사용
> - **개발/운영 RDS 분리 필요성:** 같은 RDS를 공유하면 개발 배치가 운영 부하에 직접 영향

## 3. BalanceSnapshotUseCase — JPA saveAll → JdbcTemplate.batchUpdate

> [!info] 배경
>
> - `balanceSnapshotJob`은 매일 자정 전체 라이더/협력사/본사 잔액 스냅샷을 `vr_balance_snapshot` 테이블에 INSERT
> - 기존: `vrBalanceSnapshotRepository.saveAll(snapshots)` 사용

> [!warning] 문제
>
> - JPA `@GeneratedValue(strategy = GenerationType.IDENTITY)` 사용 시 `saveAll()`이 배치 INSERT 불가
> - 스냅샷 대상이 N명이면 N번의 개별 `INSERT` 실행 → 대량 Write 부하

> [!success] 해결
>
> `JdbcTemplate.batchUpdate()`로 교체, 500건 단위 배치 INSERT:
>
> ```java
> private static final String BATCH_INSERT_SQL =
>     "INSERT INTO vr_balance_snapshot "
>         + "(SNAP_DATE, ACCT_TARGET, SUPPLY_ID, RIDER_ID, BAL_AMT, INS_DATE, INS_USER) "
>         + "VALUES (?, ?, ?, ?, ?, ?, ?)";
>
> jdbcTemplate.batchUpdate(
>     BATCH_INSERT_SQL, snapshots, 500,
>     (ps, entity) -> {
>         ps.setString(1, entity.getSnapDate());
>         ps.setString(2, entity.getAcctTarget());
>         ps.setObject(3, entity.getSupplyId());
>         ps.setObject(4, entity.getRiderId());
>         ps.setBigDecimal(5, entity.getBalAmt());
>         ps.setTimestamp(6, Timestamp.valueOf(entity.getInsDate()));
>         ps.setString(7, entity.getInsUser());
>     });
> ```
>
> N번 → ceil(N/500)번 으로 DB 왕복 대폭 감소

## 4. Quartz 트리거 — 11:10 충돌 방지 및 hectoDeposit misfire 수정

> [!info] 배경
>
> - `vrRiderAcctDailyIfJob`: 매일 11:10 정각 실행 (고정, 변경 불가)
> - `riderDeliveryHistoriesNowIfJob` / `riderDeliveryHistoriesNowJob`: `riderDeliveryHistoriesNowIfTrigger06to11` (`0 0/10 6-11 * * ?`) — 10분 단위로 11:00, **11:10**, 11:20, 11:30 ... 실행
> - 11:10에 두 Job이 동시 실행되어 RDS CPU 경합 발생
> - `hectoDepositProcessTrigger`: 1분마다 실행, misfire 지시 없음 → ECS 태스크 재기동 시 밀린 실행이 한꺼번에 몰려 실행될 수 있음

> [!warning] 문제
>
> - `06to11` 트리거가 11시간대 전체를 커버하며 11:10 슬롯 포함
> - `hectoDepositProcessTrigger` misfire 누적 위험 (재기동 후 수십 개 대기 실행)

> [!success] 해결
>
> **파일:** `MigrationQuartzConfig.java`, `DeliveryQuartzConfig.java` (prod 브랜치)
>
> `06to11` (`0 0/10 6-11 * * ?`) 을 두 트리거로 분리:
>
> | 트리거명 | Cron | 설명 |
> |---|---|---|
> | `*Trigger06to10` | `0 0/10 6-10 * * ?` | 06:00~10:50 10분 간격 |
> | `*Trigger11` | `0 0,15,20,30,40,50 11 * * ?` | 11:00~11:50 (11:10 제외) |
>
> `hectoDepositProcessTrigger`에 misfire 지시 추가:
>
> ```java
> .withSchedule(CronScheduleBuilder.cronSchedule("0 * * * * ?")
>     .withMisfireHandlingInstructionDoNothing())
> ```
>
> **배포 후 운영 DB에서 반드시 실행 필요:**
>
> ```sql
> DELETE FROM QRTZ_CRON_TRIGGERS
> WHERE SCHED_NAME = 'schedulerFactoryBean'
>   AND TRIGGER_NAME IN (
>     'riderDeliveryHistoriesNowIfTrigger06to11',
>     'riderDeliveryHistoriesNowTrigger06to11'
>   );
>
> DELETE FROM QRTZ_TRIGGERS
> WHERE SCHED_NAME = 'schedulerFactoryBean'
>   AND TRIGGER_NAME IN (
>     'riderDeliveryHistoriesNowIfTrigger06to11',
>     'riderDeliveryHistoriesNowTrigger06to11'
>   );
> ```
>
> 배포 후 `autoScheduleAllJobs`가 `06to10`, `11` 트리거를 신규 생성하며, 위 SQL로 기존 `06to11`을 제거해야 충돌 완전 해소

> [!tip] 배운 점
>
> - **Quartz JDBC JobStore에서 트리거 이름 변경 = 신규 생성 + 수동 삭제:** 코드에서 트리거 빈을 삭제해도 DB의 이전 트리거는 그대로 남아 계속 실행됨. 반드시 SQL DELETE 병행
> - **`autoScheduleAllJobs`의 동작:** `checkExists()` 후 없으면 생성, 있으면 reschedule. 제거는 하지 않음
> - **misfire 지시 없는 고빈도 트리거:** 재기동 후 밀린 misfire가 한꺼번에 실행될 수 있음. 1분 단위 이하 트리거는 `withMisfireHandlingInstructionDoNothing()` 권장
