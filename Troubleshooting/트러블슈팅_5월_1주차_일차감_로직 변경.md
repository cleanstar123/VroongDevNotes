## 배경 / 문제 정의

기존 주정산 로직에서 `totAcctVal`(실지급 배달료)이 **음수**가 되는 케이스가 존재했다.

```
[기존 공식]
totAcctVal = finalDeliveryVal - preAcctVal(선정산액) - totalDeductAmt(일차감)
```

배달 수익이 선정산액 + 일차감보다 적을 때 실지급이 음수로 계산되어 Vroong 예치금이 의도치 않게 손실되는 문제가 발생.  
또한 선정산 라이더의 일차감이 이론값으로 계산되어 실제 적용값과 불일치하는 경우가 있었음.

## v1 — 차감 유보(Defer) 로직 도입

**커밋**: `2cc47cf0` | **날짜**: 2026-05-07 08:32

### 핵심 변경: 음수 방지를 위한 3-way 분기

| 조건 | actualDeductAmt | deferredDeductAmt |
|---|---|---|
| `baseVal >= totalDeductAmt` | `totalDeductAmt` | `0` |
| `0 < baseVal < totalDeductAmt` | `baseVal` | `totalDeductAmt - baseVal` |
| `baseVal <= 0` | `0` | `totalDeductAmt` |

```java
// [변경 전]
BigDecimal totAcctVal = calc.finalDeliveryVal.subtract(preAcctVal).subtract(totalDeductAmt);

// [변경 후]
BigDecimal baseVal = calc.finalDeliveryVal.subtract(preAcctVal);
// ... 3-way 분기로 actualDeductAmt / deferredDeductAmt 계산 ...
BigDecimal totAcctVal = baseVal.subtract(actualDeductAmt); // 항상 >= 0
```

### 미수금(UNPAID) 이월 회수 로직

- `createAccountEntityWithType` 메서드에 `prevUnpaidAmt` 파라미터 추가
- 이전 정산 누적 미수금(`vr_account_mst.UNPAID_AMT`)을 조회하여 현재 정산액에서 회수
- **누적 UNPAID_AMT 계산**: `이전미수금 + 회수액 - 신규유보액` (양수이면 0으로 클리핑)

```java
// 이월 미수금 회수
BigDecimal unpaidAbs = prevUnpaidAmt.negate().max(ZERO);
BigDecimal recoveryAmt = baseAcctVal.min(unpaidAbs);
BigDecimal tranAmt = baseAcctVal.subtract(recoveryAmt);
BigDecimal newBalance = previousBalance.add(tranAmt); // ← v2에서 버그 수정됨
```

> ⚠️ v1의 `newBalance` 계산에는 버그 있음 (recoveryAmt 미포함) → v2에서 수정

## v2 — 잔액 기반 차감 Full-or-Nothing + 잔액 계산 버그 수정

**커밋**: `0d276a17` | **날짜**: 2026-05-07 14:42

### BaeminToVroongIfUseCase — 잔액 기반 차감 방식 변경

```java
// [변경 전] 가능한 만큼 차감 (partial)
BigDecimal balanceDeductAmt = todayDeferred.min(availableBal);

// [변경 후] Full-or-Nothing (잔액이 전액 커버할 때만 차감)
BigDecimal balanceDeductAmt =
    availableBal.compareTo(todayDeferred) >= 0 ? todayDeferred : BigDecimal.ZERO;
```

- 이유: 부분 차감 시 매일 조금씩 차감되어 라이더 체감 혼란 및 계산 복잡도 증가

### TOT_ACCT_VAL 처리 변경

```java
// [변경 전] 잔액 차감액을 음수로 반영 → 라이더에게 혼란
acctDaily.setTotAcctVal(currentTotAcctVal.subtract(balanceDeductAmt));

// [변경 후] TOT_ACCT_VAL 변경 안 함 (라이더 배달수익 그대로 표시)
// → RECOVER_AMT에 합산하여 별도 집계
acctDaily.setRecoverAmt(prevRecoverAmt.add(balanceDeductAmt));
```

### mergeAccountMst 호출 수정

```java
// [변경 전] balanceDeductAmt를 0으로 전달 (회계 불일치)
vrAccountMstQueryDslRepository.mergeAccountMst(accountMst, recoveryAmt, BigDecimal.ZERO);

// [변경 후] 실제 잔액 차감액 전달
vrAccountMstQueryDslRepository.mergeAccountMst(accountMst, recoveryAmt, balanceDeductAmt);
```

### RiderSettlementUseCase — 잔액 계산 버그 수정 (v1 버그)

```java
// [변경 전] recoveryAmt 미포함 → 잔액이 과소 계산됨

// [변경 후] 선정산 mergeAccountMst와 동일 공식
// newBalance = prev + tranAmt + recoveryAmt = prev + baseAcctVal
```

## v3 — 선정산 라이더 일차감 실적값 전환 + vr_deduct_daily 소급 수정

**커밋**: `5f526d85` | **날짜**: 2026-05-08 11:18

### RiderSettlementUseCase — 선정산 라이더 일차감 계산 방식 전환

기존에는 선정산/주정산 라이더 구분 없이 `vr_deduct_mst` 규칙 기반 이론값으로 일차감을 계산했음.  
선정산 라이더는 이미 선정산 과정에서 실제 차감이 이루어졌으므로 이론값 재계산 시 중복 차감 발생 가능.

```java
// [변경 전] 모든 라이더 이론값 계산
BigDecimal totalDeductAmt = calculateDeductAmountOptimized(...);

// [변경 후] 선정산 라이더: vr_deduct_daily 실적값 사용
//           주정산 라이더: 기존 이론값 계산 유지
if (batchData.preAcctValMap.containsKey(mapKey)) {
    // 선정산 라이더 → 실제 차감 합계 사용
    totalDeductAmt = batchData.deductDailyAmtMap.getOrDefault(mapKey, BigDecimal.ZERO);
} else {
    // 주정산 라이더 → 이론값 계산
    totalDeductAmt = calculateDeductAmountOptimized(...);
}
```

- `deductDailyAmtMap`: `sumDeductAmtByPeriodBatch`로 일괄 조회 (기간 내 `vr_deduct_daily.DEDUCT_AMT` 합산)
- 미수금 당일 회수분도 자동 포함됨

### BaeminToVroongIfUseCase — vr_deduct_daily 소급 수정

미수금 회수가 발생할 때 원래 미수금 발생일의 `vr_deduct_daily` 레코드를 소급 수정.  
목적: 주정산에서 `SUM(DEDUCT_AMT)`로 집계할 때 회수분이 누락되지 않도록 정합성 보장.

```java
// 미수금 회수 후 원래 발생일 레코드 소급 수정
vrDeductDailyQueryDslRepository.updateDeductDailyForRecovery(
    acctDaily.getSupplyId(), acctDaily.getRiderId(), recoveryAmt);
```

- 가장 최근 `UNPAID_AMT > 0` 레코드부터 순서대로 분배 (`tranDay DESC, seqNo DESC`)
- 해당 레코드의 `DEDUCT_AMT += covered`, `UNPAID_AMT -= covered`

### 주정산 vr_deduct_daily INSERT 제거

v1에서 주정산 완료 후 `vr_deduct_daily`에 INSERT하던 블록 삭제.  
선정산 시점에 이미 기록되므로 중복 삽입 불필요.

### calculateEffectiveDays 버그 수정

```java
// [변경 전] contractInfo.effectiveDays 그대로 반환 (정산 기간 클리핑 미적용)
return contractInfo.getEffectiveDays();

// [변경 후] acctStDate/acctEdDate ∩ 계약 유효 기간 교집합으로 계산
LocalDate clippedStart = max(acctStDate, contractInfo.getEffectiveStartDate());
LocalDate clippedEnd   = min(acctEdDate,  contractInfo.getEffectiveEndDate());
if (clippedStart.isAfter(clippedEnd)) return 0;
return ChronoUnit.DAYS.between(clippedStart, clippedEnd) + 1;
```

| `updateDeductDailyForRecovery` | 미수금 회수 시 원래 발생일 레코드 소급 수정 |

## 전체 변경 흐름 요약

```
[기존]
totAcctVal = finalDelivery - preAcct - totalDeduct  ← 음수 가능
선정산 일차감 = 이론값(vr_deduct_mst 규칙)
미수금 이월 없음

    ↓ v1

차감 유보(defer): 정산 가능 금액 내에서만 차감, 나머지 deferredDeductAmt로 이월
이월 미수금 회수: 다음 정산 시 이전 unpaidAmt 자동 회수
vr_deduct_daily 기록: 주정산 일차감 내역 INSERT

    ↓ v2

잔액 기반 차감 Full-or-Nothing: 잔액 >= 유보액 전액이어야만 차감
잔액 계산 버그 수정: newBalance에 recoveryAmt 포함
TOT_ACCT_VAL 불변, RECOVER_AMT로 회수액 집계

    ↓ v3

선정산 라이더 일차감 = 실적값(vr_deduct_daily.DEDUCT_AMT)으로 전환 (이론값 폐기)
미수금 회수 시 원래 발생일 vr_deduct_daily 소급 수정 → 주정산 집계 정합성 확보
calculateEffectiveDays 버그 수정 (정산 기간 클리핑 미적용 오류)
주정산 vr_deduct_daily INSERT 블록 제거 (중복)
```
