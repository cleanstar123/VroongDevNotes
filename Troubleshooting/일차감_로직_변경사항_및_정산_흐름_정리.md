## 1. 변경 전 vs 변경 후 요약

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 차감 알고리즘 | 총합 단순 cap (금액 내 부분 차감 허용) | **Largest Fits First** (규칙별 전부-or-전무) |
| `vr_deduct_daily` 역할 | 이론적 차감액 기록 (선정산 라이더 제외) | **실제 적용액 + 미납금** 기록 (선정산 포함) |
| `vr_deduct_daily.UNPAID_AMT` | 존재하지 않음 | **신규 컬럼** 추가 |
| 선정산 라이더의 차감 기록 | `vr_deduct_daily`에 없음 | 선정산 처리 시 규칙별 삽입 |
| 주정산 라이더의 차감 기록 | 없음 | 주정산 처리 시 `vr_deduct_daily`에 집계 1건 삽입 |
| 협력사 주정산 `DEDUCT_VAL` | `vr_deduct_mst` 기반 이론값 계산 | `vr_deduct_daily.DEDUCT_AMT` 실적용액 합산 |
| 미수금(`UNPAID_AMT`) 추적 | `vr_account_mst`에만 저장, 활용 안 함 | `vr_deduct_daily`에도 기록, 다음 정산 시 자동 회수 |

## 2. 핵심 알고리즘: Largest Fits First

### 개념

모든 일차감 규칙을 **금액 내림차순**으로 정렬한 뒤, 정산 가능 금액 범위 내에서 **각 규칙을 전부 적용하거나 전부 보류**한다.  
부분 차감(partial deduction)은 없다. 각 규칙은 100% 적용 또는 0% 적용만 가능하다.

### 알고리즘 흐름

```
baseAcctVal = 배달료 - 수수료 - 보험료 - 원천징수  (차감 이전 정산 가능 금액)
remaining   = max(baseAcctVal, 0)

모든 차감 규칙을 금액 내림차순 정렬

for each 차감규칙 in 정렬된_규칙목록:
    if remaining >= 차감규칙.금액:
        → 적용 (DEDUCT_AMT = 차감금액, UNPAID_AMT = 0)
        remaining -= 차감규칙.금액
    else:
        → 보류 (DEDUCT_AMT = 0, UNPAID_AMT = 차감금액)

totAcctVal = baseAcctVal - sum(DEDUCT_AMT) >= 0
```

### 예시

**상황**: baseAcctVal = 15,000원, 일차감 규칙 3개

| 규칙 | 이론 금액 | 적용 결과 |
|---|---|---|
| 일차감A | 25,000원 | 보류 (remaining=15,000 < 25,000) |
| 일차감B | 12,000원 | **적용** (remaining=15,000 ≥ 12,000) → remaining=3,000 |
| 일차감C | 10,000원 | 보류 (remaining=3,000 < 10,000) |

- `DEDUCT_AMT` 합계 = **12,000원** (실제 적용)
- `UNPAID_AMT` 합계 = **35,000원** (25,000 + 10,000, 미납 이월)
- `totAcctVal` = 15,000 - 12,000 = **3,000원** (항상 ≥ 0)

## 3. 테이블 구조 변경

### `vr_deduct_daily` 변경

```sql
-- 기존 구조
SEQ_NO | DEDUCT_SEQ_NO | TRAN_DAY | SUPP_ID | RIDER_ID | DEDUCT_NAME | DEDUCT_AMT

-- 변경 후 (UNPAID_AMT 컬럼 추가)
SEQ_NO | DEDUCT_SEQ_NO | TRAN_DAY | SUPP_ID | RIDER_ID | DEDUCT_NAME | DEDUCT_AMT | UNPAID_AMT
```

| 컬럼 | 변경 전 의미 | 변경 후 의미 |
|---|---|---|
| `DEDUCT_AMT` | 이론적 차감액 (항상 = 규칙 금액) | **실제 적용된 차감액** (0 또는 규칙 금액) |
| `UNPAID_AMT` | 없음 | **미납금** (보류된 차감액, 다음 정산으로 이월) |

## 4. DB 스키마 변경 쿼리

> DBeaver에서 직접 실행하세요.

```sql
-- vr_deduct_daily 테이블에 미납금 컬럼 추가
ALTER TABLE vr_deduct_daily
    ADD COLUMN UNPAID_AMT DECIMAL(17, 0) NOT NULL DEFAULT 0 COMMENT '미납금(미수 차감액)' AFTER DEDUCT_AMT;
```

> **주의**: 기존 데이터(변경 전 이론값이 들어있는 행)는 `UNPAID_AMT = 0`으로 초기화됩니다.  
> 기존 데이터의 `DEDUCT_AMT`는 이론값이므로, 변경 후 새로 생성되는 행과 혼재합니다.  
> 필요하다면 배치를 한 번 재실행하거나, 기존 데이터를 삭제 후 재생성하는 것을 권장합니다.

## 5. 선정산(RP) 흐름 — 변경 후

### 처리 위치

`BaeminToVroongIfUseCase.java` > `calculateAcctDaily()` + 저장 루프

### 흐름도

```
[매일 선정산 배치]
  1. 라이더별 배달료, 배달건수 집계
  2. 수수료 계산 (본사/협력사)
  3. 보험료, 원천징수 계산
  4. baseAcctVal 계산 = 배달료 - 수수료 - 보험료 - 원천징수
  5. 정액 공제 목록 + 정률 공제 목록 조회 (vr_deduct_mst)
  6. 🆕 Largest Fits First 적용
       → DeductApplicationResult 목록 생성 (규칙별 actual/deferred)
  7. vr_rider_acct_daily 저장 (DEDUCT_VAL = sum(actual))
  8. 🆕 vr_deduct_daily 기록 (규칙별 1행씩, 멱등성 보장)
       DELETE WHERE suppId + riderId + tranDay
       INSERT per rule (DEDUCT_AMT, UNPAID_AMT)
  9. prevUnpaidAmt 조회 (이전 미납금)
  10. 회수 계산: recoveryAmt = min(totAcctVal, abs(prevUnpaidAmt))
  11. totAcctVal -= recoveryAmt (회수분 차감)
  12. newUnpaidAmt = prevUnpaidAmt + recoveryAmt - todayDeferred
  13. vr_account_mst (ACCT_TYPE=RP) 저장 (UNPAID_AMT = newUnpaidAmt)
```

### 구체적 예시

**전제**: 라이더A, baseAcctVal = 9,000원, 이전 미납금(UNPAID_AMT) = 0

| 규칙 | 금액 |
|---|---|
| 일차감1 (장비비) | 25,000원 |
| 일차감2 (교육비) | 10,000원 |
| 일차감3 (유니폼) | 7,500원 |

**처리 결과** (내림차순 정렬: 25,000 → 10,000 → 7,500):
- 25,000 > 9,000 → 보류 (UNPAID_AMT=25,000)
- 10,000 > 9,000 → 보류 (UNPAID_AMT=10,000)
- 7,500 ≤ 9,000 → **적용** (DEDUCT_AMT=7,500, remaining=1,500)

**vr_deduct_daily 삽입**:
```
TRAN_DAY=20260504, RIDER_ID=A, 일차감1: DEDUCT_AMT=0,     UNPAID_AMT=25000
TRAN_DAY=20260504, RIDER_ID=A, 일차감2: DEDUCT_AMT=0,     UNPAID_AMT=10000
TRAN_DAY=20260504, RIDER_ID=A, 일차감3: DEDUCT_AMT=7500,  UNPAID_AMT=0
```

**vr_rider_acct_daily**:
- `DEDUCT_VAL` = 7,500원
- `TOT_ACCT_VAL` = 9,000 - 7,500 = **1,500원**

**vr_account_mst (RP)**:
- `TRAN_AMT` = 1,500원
- `UNPAID_AMT` = 0 + 0 - (25,000 + 10,000) = **-35,000원** (미납금 누적)

---

**다음날** (baseAcctVal = 50,000원, 동일 규칙):

처리 결과:
- 25,000 ≤ 50,000 → **적용** (remaining=25,000)
- 10,000 ≤ 25,000 → **적용** (remaining=15,000)
- 7,500 ≤ 15,000 → **적용** (remaining=7,500)

모두 적용. todayDeferred = 0

**회수 처리**: prevUnpaidAmt = -35,000
- unpaidAbs = 35,000
- totAcctVal 계산 전 = 50,000 - 42,500 = 7,500원
- recoveryAmt = min(7,500, 35,000) = **7,500원** 회수
- 실 지급 totAcctVal = 7,500 - 7,500 = **0원**

**vr_account_mst (RP)**:
- `TRAN_AMT` = 0원
- `UNPAID_AMT` = -35,000 + 7,500 - 0 = **-27,500원** (잔여 미납금)

## 6. 주정산(RW) 흐름 — 변경 후

### 처리 위치

`RiderSettlementUseCase.java` > `calculateRiderSettlement()` + 저장 루프

### 흐름도

```
[주 1회 엑셀 업로드 기반]
  1. 엑셀에서 라이더별 주간 배달료, 선정산 금액 파싱
  2. 주간 일차감 총액 계산 (calculateDeductAmountOptimized)
  3. 수수료/보험료/세금 계산
  4. baseVal = finalDeliveryVal - preAcctVal
  5. 기존 cap 로직으로 actualDeductAmt / deferredDeductAmt 결정
     (주정산은 아직 per-rule 알고리즘 미적용, 총합 cap 방식 유지)
  6. totAcctVal = baseVal - actualDeductAmt (항상 >= 0)
  7. 🆕 vr_deduct_daily 기록 (기간 집계 1건, tranDay = 정산종료일)
       DELETE WHERE suppId + riderId + tranDay
       INSERT (DEDUCT_SEQ_NO=null, DEDUCT_NAME='주정산 일차감',
               DEDUCT_AMT=actualDeductAmt, UNPAID_AMT=deferredDeductAmt)
  8. prevUnpaidAmt 조회 및 UNPAID_AMT 갱신
  9. vr_account_mst (ACCT_TYPE=RW) 저장
```

### 주정산 예시

**라이더B**: 주간 배달료 60,000원, preAcctVal 35,000원, 주간 일차감 합계 40,000원

- `baseVal` = 60,000 - 35,000 = 25,000원
- 40,000 > 25,000 → 부분 차감
  - `actualDeductAmt` = 25,000원 (가능한 최대)
  - `deferredDeductAmt` = 15,000원 (미납 이월)
- `totAcctVal` = 25,000 - 25,000 = **0원** (음수 방지)

**vr_deduct_daily 삽입** (주정산 기간 종료일 tranDay):
```
TRAN_DAY=20260504, RIDER_ID=B, DEDUCT_NAME='주정산 일차감',
DEDUCT_AMT=25000, UNPAID_AMT=15000
```

**vr_account_mst (RW)**:
- `TRAN_AMT` = 0원
- `UNPAID_AMT` = 이전미납 + 0(회수없음) - 15,000 = 갱신

> **참고**: 주정산은 현재 단순 총합 cap 방식입니다. 향후 선정산과 동일하게 per-rule Largest Fits First로 전환 가능합니다.

## 7. 협력사 주정산(SW) 흐름 — 변경 후

### 처리 위치

`PartnerSettlementUseCase.java` > `sumActualDeductFromDeductDailyByPeriod()`

### 변경 내용

```java
// 변경 전: vr_deduct_mst 기반 이론값 계산
deductVal = calculateDeductionOptimized(supplyId, ...);

// 변경 후: vr_deduct_daily 실적용액 합산
deductVal = vrRiderAcctDailyQueryDslRepository
    .sumActualDeductFromDeductDailyByPeriod(
        supplyId, acctStDate, acctEdDate, registeredRiderIds);
```

### 조회 쿼리 (의미)

```sql
SELECT SUM(DEDUCT_AMT)
FROM vr_deduct_daily
WHERE SUPP_ID = :suppId
  AND TRAN_DAY BETWEEN :acctStDate AND :acctEdDate
  AND RIDER_ID IS NOT NULL
  AND RIDER_ID IN (:registeredRiderIds)
```

### 협력사 정산 예시

**정산 기간**: 2026-04-28 ~ 2026-05-04  
**협력사 소속 라이더 3명**

| 구분 | 라이더A | 라이더B | 라이더C |
|---|---|---|---|
| 이론 일차감 합계 | 42,500원 | 40,000원 | 30,000원 |
| 실제 적용액 (DEDUCT_AMT 합계) | 7,500원 | 25,000원 | 30,000원 |
| 미납금 (UNPAID_AMT) | 35,000원 | 15,000원 | 0원 |

**협력사 `DEDUCT_VAL`**:
- 변경 전 (이론값): 42,500 + 40,000 + 30,000 = **112,500원** (과대계상)
- 변경 후 (실적용): 7,500 + 25,000 + 30,000 = **62,500원** (정확한 실적)

**효과**: 라이더가 실제로 차감받은 금액만 협력사 정산에 반영 → Vroong 예치금 손실 차단

## 8. 비선정산(주정산 전용) 라이더 — `migrateRiderFeeDeductDaily`

### 현재 동작

- 비선정산 라이더의 `vr_deduct_daily` 기록은 `migrateRiderFeeDeductDaily` 배치에서 처리
- **이 배치는 이론적 금액을 기록하며, `UNPAID_AMT = 0`으로 삽입**
- 실제 차감 cap은 주정산 처리 시점(`RiderSettlementUseCase`)에서 이루어지며, 그 결과가 `vr_deduct_daily`에 덮어쓰기됨

### 주간 덮어쓰기 흐름

```
migrateRiderFeeDeductDaily (일별 배치)
  → vr_deduct_daily에 이론값 삽입 (DEDUCT_AMT=이론값, UNPAID_AMT=0)

주정산 업로드 (주 1회)
  → deleteDeductDailyByRiderAndDate(suppId, riderId, tranDay=정산종료일)
  → insertDeductDaily(실적용액, 미납금) 1건 삽입
  
협력사 주정산
  → SUM(DEDUCT_AMT) from vr_deduct_daily 기간 조회
    (선정산 라이더: 일별 실적용액 합계 / 주정산 라이더: 정산종료일 1건)
```

## 11. 주의사항

### 기존 `vr_deduct_daily` 데이터 처리

- 컬럼 추가 후 기존 행은 `UNPAID_AMT = 0`이 됨
- 기존 데이터는 이론값(변경 전 로직) 기반이므로 협력사 정산 조회 시 오염 가능성 있음
- **권장**: 컬럼 추가 후 배치를 날짜 지정으로 재실행하거나 기존 데이터 수동 삭제

### 재실행 멱등성

- 선정산 저장 루프: `deleteDeductDailyByRiderAndDate` 후 삽입 → 재실행 안전
- 주정산 저장 루프: `deleteDeductDailyByRiderAndDate` 후 삽입 → 재실행 안전
- `migrateRiderFeeDeductDaily`: `deleteDeductDailyByPreSettlement` (기존) 후 삽입 → 재실행 안전

### `vr_deduct_daily`의 `tranDay` 기준

| 정산 유형 | `TRAN_DAY` 기준 |
|---|---|
| 선정산 라이더 | 배달 발생 날짜 (매일) |
| 주정산 라이더 (migrateRiderFeeDeductDaily) | 배달 발생 날짜 (매일, 이론값) |
| 주정산 라이더 (RiderSettlementUseCase) | 정산 종료일 (`acctEdDate`) |

협력사 주정산 조회 시 `TRAN_DAY BETWEEN acctStDate AND acctEdDate`로 모두 포함됩니다.
