## 1. 라이더 잔액 50,000원 누락 - Race Condition 문제

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-03-13 14:10 ~ 14:11 |
| **영향 범위** | 라이더 잔액 정산 |
| **심각도** | 🔴 Critical |
| **관련 시스템** | vroong-api, VR_ACCOUNT_MST 테이블 |

### 증상

- 라이더 잔액이 예상보다 50,000원 적게 기록됨
- SEQ 65827 (TI - 입금)과 SEQ 66237 (RP - 선정산) 레코드에서 BAL_AMT 불일치
- 두 트랜잭션이 거의 동시에 발생하여 잔액 계산 오류

### 문제 데이터

| SEQ_NO | ACCT_TYPE | ACCT_AMT | BAL_AMT     | INS_DATE            |
| ------ | --------- | -------- | ----------- | ------------------- |
| 65827  | TI (입금)   | 50,000   | 50,320      | 2026-03-13 14:10:36 |
| 66237  | RP (선정산)  | 99,912   | **100,232** | 2026-03-13 14:11:04 |

**예상 BAL_AMT**: 50,320 + 99,912 = **150,232**
**실제 BAL_AMT**: **100,232** (50,000원 누락)

### 장애 타임라인

| 시간       | 이벤트          | 상세                                          |
| -------- | ------------ | ------------------------------------------- |
| 14:10:36 | TI 레코드 생성 시작 | HectoDepositUseCase에서 입금 처리 시작              |
| 14:10:36 | TI 트랜잭션 A    | 잔액 조회 (320원) → 계산 (50,320) → 저장             |
| 14:11:04 | RP 레코드 생성 시작 | BaeminToVroongIfUseCase에서 선정산 처리 시작         |
| 14:11:04 | RP 트랜잭션 B    | 잔액 조회 (**320원 - 락 없음**) → 계산 (100,232) → 저장 |
| 14:11:04 | 문제 발생        | TI 커밋 전에 RP가 stale 데이터 읽음                   |

### 문제 코드 분석

#### 1. 잔액 조회 메서드 - 락 없음 (문제의 핵심)

**파일**: `VrAccountMstQueryDslRepository.java:33-47`

```java
/**
 * 특정 라이더의 최신 잔액 조회 (락 없음 - Race Condition 발생 원인)
 */
public BigDecimal findLatestBalAmtBySupplyIdAndRiderId(String supplyId, String riderId) {
  return queryFactory
      .select(accountMst.balAmt)
      .from(accountMst)
      .where(
          accountMst.riderId.eq(riderId)  // ⚠️ supplyId가 사용되지 않음!
              .and(accountMst.acctType.notIn("RD", "SD")))
      .orderBy(accountMst.seqNo.desc())
      .limit(1)
      .fetchOne();
}
```

**문제점**:
1. **Pessimistic Lock 미적용**: 동시 트랜잭션이 같은 데이터를 읽을 수 있음
2. **supplyId 파라미터 미사용**: 파라미터로 받지만 쿼리에서 사용하지 않음

#### 2. 락이 있는 메서드 (해결책)

**파일**: `VrAccountMstQueryDslRepository.java:50-66`

```java
/**
 * 특정 라이더의 최신 잔액 조회 (락 적용 - Race Condition 방지)
 */
public BigDecimal findLatestBalAmtBySupplyIdAndRiderIdWithLock(String supplyId, String riderId) {
  return queryFactory
      .select(accountMst.balAmt)
      .from(accountMst)
      .where(
          accountMst.riderId.eq(riderId)
              .and(accountMst.acctType.notIn("RD", "SD")))
      .orderBy(accountMst.seqNo.desc())
      .limit(1)
      .setLockMode(jakarta.persistence.LockModeType.PESSIMISTIC_WRITE)  // ✅ 락 적용
      .fetchOne();
}
```

### 검증 방법

1. **단위 테스트**: 동시 트랜잭션 시뮬레이션 테스트 작성
2. **통합 테스트**: 실제 환경에서 동시 입금/정산 테스트
3. **모니터링**: 잔액 불일치 감지 쿼리 추가

```sql
-- 잔액 불일치 감지 쿼리
SELECT
    a.SEQ_NO,
    a.RIDER_ID,
    a.ACCT_TYPE,
    a.ACCT_AMT,
    a.BAL_AMT,
    a.INS_DATE,
    LAG(a.BAL_AMT) OVER (PARTITION BY a.RIDER_ID ORDER BY a.SEQ_NO) as PREV_BAL_AMT,
    a.BAL_AMT - (LAG(a.BAL_AMT) OVER (PARTITION BY a.RIDER_ID ORDER BY a.SEQ_NO) + a.ACCT_AMT) as DIFF
FROM VR_ACCOUNT_MST a
WHERE a.ACCT_TYPE NOT IN ('RD', 'SD')
HAVING DIFF != 0;
```

## 📝 교훈 및 개선 사항

### 근본 원인

- **동시성 제어 미흡**: 잔액 조회 시 Pessimistic Lock이 적용되지 않아 Race Condition 발생
- **코드 중복**: 락이 있는 메서드와 없는 메서드가 별도로 존재하여 혼란 유발

### 주요 교훈

1. **금융 데이터는 반드시 락 사용**
   - 잔액, 거래 내역 등 금융 관련 데이터는 Pessimistic Lock 필수
   - 동시성 문제는 재현이 어려워 사전 예방이 중요

2. **메서드 네이밍 컨벤션**
   - 락이 있는 메서드와 없는 메서드 구분 명확히
   - 또는 기본 메서드에 락을 적용하고 특수 케이스만 락 없이 사용

3. **파라미터 검증**
   - 사용하지 않는 파라미터 제거 또는 사용하도록 수정
   - `supplyId` 파라미터가 쿼리에서 사용되지 않는 문제 발견

## 📋 참고: JPA Pessimistic Lock 타입

| LockModeType | 설명 | 사용 케이스 |
|--------------|------|------------|
| PESSIMISTIC_READ | 공유 락 (읽기 허용, 쓰기 차단) | 읽기 전용 트랜잭션 |
| PESSIMISTIC_WRITE | 배타 락 (읽기/쓰기 모두 차단) | 잔액 변경 등 쓰기 작업 |
| PESSIMISTIC_FORCE_INCREMENT | 배타 락 + 버전 증가 | 버전 관리 필요 시 |

### QueryDSL에서 Pessimistic Lock 적용

```java
// QueryDSL에서 락 적용 방법
queryFactory
    .select(accountMst.balAmt)
    .from(accountMst)
    .where(...)
    .setLockMode(LockModeType.PESSIMISTIC_WRITE)  // 락 적용
    .fetchOne();
```

### JPA Repository에서 Pessimistic Lock 적용

```java
// @Lock 어노테이션 사용
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a.balAmt FROM VrAccountMst a WHERE a.riderId = :riderId ORDER BY a.seqNo DESC LIMIT 1")
BigDecimal findLatestBalAmtWithLock(@Param("riderId") String riderId);
```
