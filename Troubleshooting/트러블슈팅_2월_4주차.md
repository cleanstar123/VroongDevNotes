### 최종 결론

#### 확인된 사실

| 항목 | 결과 | 의미 |
|------|------|------|
| CloudTrail API 호출 | 없음 | 수동 재시작 아님 |
| CloudWatch 지표 | 일직선 | 애플리케이션 부하 문제 아님 |
| Health Dashboard | RDS 알림 없음 | 예정된 유지보수 아님 |
| RDS 이벤트 | "Recovery started" | AWS 자동 복구 실행됨 |
| 애플리케이션 로그 | `Connect timed out` | MySQL 프로세스가 없거나 응답 불가 |

#### 원인 판단

**AWS 인프라 레벨에서 자동 복구(Recovery)가 실행됨**

- `Connect timed out` = TCP 연결 자체가 안 됨 = MySQL 프로세스가 없거나 네트워크 불가
- RDS 이벤트에 "Recovery started" 기록 = AWS가 복구 프로세스 실행
- 사용자가 트리거한 작업 없음 (CloudTrail 기록 없음)

#### 복구가 필요했던 가능한 원인

| 가능성 | 설명 | 확인 방법 |
|--------|------|-----------|
| **하드웨어 장애** | 호스트 서버 문제로 자동 이전 | AWS Support 문의 |
| **OS 패치/업데이트** | AWS 내부 패치 적용 | AWS Support 문의 |
| **메모리 부족 (OOM)** | 1GB RAM으로 MySQL 8.0 운영 중 OOM 발생 가능 | CloudWatch 60초 간격으로 순간 스파이크 놓칠 수 있음 |

**정확한 원인 확인**: AWS Support 유료 플랜 필요 (Technical Support)

## 📝 교훈 및 개선 사항

### 공통 교훈

1. **RDS 인스턴스 사양 검토**
   - db.t4g.micro (RAM 1GB)는 MySQL 8.0 프로덕션 환경에 부족
   - 메모리 부족으로 인한 불안정 가능성 있음

2. **단일 AZ의 위험성**
   - 다중 AZ 미사용 시 장애 발생 시 자동 failover 불가
   - 복구까지 다운타임 발생 (이번 케이스: 약 35분)

3. **AWS 인프라 장애 대응**
   - AWS 내부 문제로 인한 장애는 사전 예측 불가
   - 다중 AZ, 읽기 전용 복제본 등으로 가용성 확보 필요

### 후속 조치

- [ ] RDS 인스턴스 사양 업그레이드 검토 (db.t4g.small 이상)
- [ ] 다중 AZ 활성화 검토
- [ ] CloudWatch 알람 설정 (CPU, Memory, Connections)
- [ ] RDS 이벤트 SNS 알림 설정
- [ ] HikariCP 설정 최적화 검토
- [ ] (선택) AWS Support 케이스 생성하여 정확한 원인 확인
