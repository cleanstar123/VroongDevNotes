#### 신규 메서드: findLatestBalanceByTargetIdWithLock

```java
/**
 * 수신자의 최신 잔액 조회 (RR/RD/SR/SD 타입 제외) - 비관적 락 적용
 * ACCT_DATE DESC, SEQ_NO DESC로 정렬하여 가장 최근 정산일의 최신 레코드를 가져옴
 * Race Condition 방지를 위해 PESSIMISTIC_WRITE 락 적용
 *
 * @param targetId 수신자 ID
 * @param isSupplier 수신자가 협력사인지 여부
 * @return 최신 잔액 (없으면 0)
 */
public BigDecimal findLatestBalanceByTargetIdWithLock(String targetId, boolean isSupplier) {
  BigDecimal balAmt;

  if (isSupplier) {
    balAmt =
        queryFactory
            .select(vam.balAmt)
            .from(vam)
            .where(
                vam.supplyId.eq(targetId)
                    .and(vam.acctTarget.eq("S"))
                    .and(vam.acctType.notIn("RD", "SD")))
            .orderBy(vam.seqNo.desc())
            .limit(1)
            .setLockMode(LockModeType.PESSIMISTIC_WRITE)
            .fetchOne();
  } else {
    balAmt =
        queryFactory
            .select(vam.balAmt)
            .from(vam)
            .where(
                vam.riderId.eq(targetId)
                    .and(vam.acctTarget.eq("R"))
                    .and(vam.acctType.notIn("RD", "SD")))
            .orderBy(vam.seqNo.desc())
            .limit(1)
            .setLockMode(LockModeType.PESSIMISTIC_WRITE)
            .fetchOne();
  }

  return balAmt != null ? balAmt : BigDecimal.ZERO;
}
```

## 5. 핵심 변경 사항

### Pessimistic Lock 적용 방법

```java
// QueryDSL에서 비관적 락 적용
queryFactory
    .select(accountMst.balAmt)
    .from(accountMst)
    .where(...)
    .orderBy(accountMst.seqNo.desc())
    .limit(1)
    .setLockMode(LockModeType.PESSIMISTIC_WRITE)  // 핵심: 락 적용
    .fetchOne();
```

### 적용 효과

1. **동시 트랜잭션 직렬화**: 같은 레코드에 대한 동시 접근 시 순차 처리
2. **Stale 데이터 방지**: 항상 최신 커밋된 데이터 기반으로 잔액 계산
3. **데이터 정합성 보장**: 입금/출금/정산 간 Race Condition 해결

### PESSIMISTIC_WRITE 락 동작 범위

**중요: 락은 "특정 행(row)"에만 걸립니다. 메서드나 테이블 전체가 아닙니다.**

```
┌─────────────────────────────────────────────────────────────┐
│                    VR_ACCOUNT_MST 테이블                      │
├─────────────────────────────────────────────────────────────┤
│ SEQ_NO │ SUPPLY_ID │ RIDER_ID │ BAL_AMT │ ...               │
├────────┼───────────┼──────────┼─────────┼───────────────────┤
│   1    │  SUPP001  │  R001    │ 50000   │  ← 트랜잭션 A 락   │
│   2    │  SUPP001  │  R002    │ 30000   │  ← 자유롭게 접근   │
│   3    │  SUPP002  │  R003    │ 80000   │  ← 자유롭게 접근   │
└────────┴───────────┴──────────┴─────────┴───────────────────┘
```

#### 시나리오별 동작

| 상황 | 결과 |
|------|------|
| 정산 중 **같은 라이더**가 출금 시도 | 대기 후 순차 처리 (의도한 동작) |
| 정산 중 **다른 라이더**가 출금 시도 | 병렬 처리 (영향 없음) |
| 라이더 A 입금 중 라이더 B 출금 | 병렬 처리 (영향 없음) |

#### 타임라인 예시

```
시간 →
────────────────────────────────────────────────────────

트랜잭션 A (라이더 R001 주정산)
  │── 락 획득 (R001 행) ──│── 잔액 계산 ──│── 커밋 ──│
                                                    ↓ 락 해제

트랜잭션 B (라이더 R001 출금) - 대기 후 실행
                              │── 대기 ──│── 락 획득 ──│── 처리 ──│

트랜잭션 C (라이더 R002 출금) - 즉시 실행 (다른 행이므로)
  │── 락 획득 (R002 행) ──│── 처리 ──│── 커밋 ──│
```

#### 핵심 정리

- **같은 라이더/협력사**에 대한 동시 작업 → 순차 처리 (Race Condition 방지)
- **다른 라이더/협력사** 작업 → 병렬 처리 (성능 영향 없음)
- 락은 트랜잭션 커밋 시 즉시 해제됨
- "정산 중 모든 사용자 기능 중단"은 발생하지 않음

#### TC-003: TI + RP 동시 실행 테스트 (핵심 테스트)

| 항목 | 내용 |
|------|------|
| **테스트 ID** | TC-003 |
| **테스트명** | 입금(TI)과 선정산(RP) 동시 실행 시 잔액 정합성 테스트 |
| **목적** | 동시에 입금과 선정산이 발생해도 잔액이 정확하게 계산되는지 확인 |
| **사전 조건** | 라이더 초기 잔액 = 1,000원 |

**테스트 시나리오**:
```
초기 잔액: 1,000원
TI 입금: +50,000원
RP 선정산: +100,000원
예상 최종 잔액: 1,000 + 50,000 + 100,000 = 151,000원
```

**테스트 절차**:
1. 라이더 초기 잔액 1,000원 설정
2. 2개의 스레드 생성:
   - Thread A: TI 입금 처리 (+50,000원)
   - Thread B: RP 선정산 처리 (+100,000원)
3. 동시에 실행 (CountDownLatch 사용)
4. 두 스레드 완료 후 최종 잔액 확인

**예상 결과**: 최종 잔액 = 151,000원

```java
@Test
void testConcurrentTIAndRP_BalanceConsistency() throws InterruptedException {
    // Given
    String supplyId = "TEST_SUPP_001";
    String riderId = "TEST_RIDER_001";
    BigDecimal initialBalance = new BigDecimal("1000");
    BigDecimal tiAmount = new BigDecimal("50000");
    BigDecimal rpAmount = new BigDecimal("100000");

    createTestAccountRecord(supplyId, riderId, initialBalance);

    CountDownLatch startLatch = new CountDownLatch(1);
    CountDownLatch endLatch = new CountDownLatch(2);

    // When
    ExecutorService executor = Executors.newFixedThreadPool(2);

    // Thread A: TI 입금
    executor.submit(() -> {
        try {
            startLatch.await();
            hectoDepositUseCase.processDeposit(supplyId, riderId, tiAmount);
        } finally {
            endLatch.countDown();
        }
    });

    // Thread B: RP 선정산
    executor.submit(() -> {
        try {
            startLatch.await();
            baeminToVroongIfUseCase.processRiderSettlement(supplyId, riderId, rpAmount);
        } finally {
            endLatch.countDown();
        }
    });

    startLatch.countDown(); // 동시 시작
    endLatch.await(10, TimeUnit.SECONDS);

    // Then
    BigDecimal finalBalance =
        vrAccountMstQueryDslRepository.findLatestBalAmtBySupplyIdAndRiderIdWithLock(supplyId, riderId);

    BigDecimal expectedBalance = initialBalance.add(tiAmount).add(rpAmount); // 151,000
    assertThat(finalBalance).isEqualTo(expectedBalance);
}
```

### 6.5 데이터 검증 쿼리

#### 잔액 불일치 감지 쿼리

테스트 후 잔액 정합성을 검증하는 SQL:

```sql
-- 라이더별 잔액 불일치 검출
SELECT
    a.SEQ_NO,
    a.SUPPLY_ID,
    a.RIDER_ID,
    a.ACCT_TYPE,
    a.TRAN_AMT,
    a.BAL_AMT as CURRENT_BAL,
    a.INS_DATE,
    LAG(a.BAL_AMT) OVER (
        PARTITION BY a.SUPPLY_ID, a.RIDER_ID
        ORDER BY a.SEQ_NO
    ) as PREV_BAL,
    a.BAL_AMT - COALESCE(
        LAG(a.BAL_AMT) OVER (
            PARTITION BY a.SUPPLY_ID, a.RIDER_ID
            ORDER BY a.SEQ_NO
        ), 0
    ) - a.TRAN_AMT as DIFF
FROM VR_ACCOUNT_MST a
WHERE a.ACCT_TYPE NOT IN ('RD', 'SD')
  AND a.RIDER_ID = 'TEST_RIDER_001'
ORDER BY a.SEQ_NO;

-- DIFF가 0이 아니면 잔액 불일치 발생
```
