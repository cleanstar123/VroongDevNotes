## 근본 원인

### Spring AOP Self-Invocation 문제

Spring의 `@Transactional`은 **외부에서 Spring 프록시(CGLIB)를 통해 호출될 때만** 동작한다.
같은 클래스 안에서 `this.메서드()` 로 직접 호출하면 프록시를 우회하므로 트랜잭션이 생성되지 않는다.

### 문제가 된 호출 체인

```
Controller
  └─ baeminToVroongIfUseCase.migrateTable()       ← CGLIB 프록시 통해 호출
       └─ @Transactional 없음 → 트랜잭션 미생성

  migrateTable() [L.96]
    └─ this.processTableMigration()               ← 직접 호출(this.) → 프록시 우회

  processTableMigration() [L.165]
    └─ this.migrateVrRiderAcctDaily()             ← 직접 호출(this.) → 프록시 우회
                                                    @Transactional(REQUIRES_NEW) 완전 무시됨

  migrateVrRiderAcctDaily()
    └─ deleteDeductDailyByRiderAndDate()          ← DELETE 실행
         → TransactionRequiredException !!
```

### 핵심 포인트

- `migrateTable()`: `@Transactional` **없음** → 프록시가 인터셉트해도 트랜잭션 미생성
- `processTableMigration()`: `@Transactional` **없음** + `this.` 직접 호출
- `migrateVrRiderAcctDaily()`: `@Transactional(REQUIRES_NEW)` **있으나** `this.`로 호출되어 완전히 무시됨
- DELETE/UPDATE 는 JPA 트랜잭션 필수 → `TransactionRequiredException` 발생

### `isolation = READ_COMMITTED`는 원인이 아님

커밋 `b56792bd`에서 `isolation = Isolation.READ_COMMITTED`를 추가했지만, 이 애너테이션은 메서드가 프록시를 통해 호출될 때만 효과가 있다. `this.` 직접 호출 체인에서는 런타임 영향 없음.

**실제 문제의 시작**: 커밋 `50b5a875` ("prod에 일차감 변경 로직 반영 v1")에서 `deleteDeductDailyByRiderAndDate`, `insertDeductDaily`, `mergeRiderAcctDaily` 등 첫 쓰기 작업이 `migrateVrRiderAcctDaily`에 추가되었으나, 트랜잭션 없이 DML이 실행되는 구조였음.

### 수정 원리

`applicationContext.getBean(BaeminToVroongIfUseCase.class)` 는 Spring이 관리하는 CGLIB 프록시 인스턴스를 반환한다. 이 프록시를 통해 메서드를 호출하면 `@Transactional(REQUIRES_NEW)` 이 정상적으로 적용되어 새로운 독립 트랜잭션이 생성된다.

```
processTableMigration()
  └─ applicationContext.getBean() → Spring 프록시를 통해 호출
       └─ @Transactional(REQUIRES_NEW) 정상 적용 → 새 트랜잭션 생성
            └─ deleteDeductDailyByRiderAndDate() → ✅ 트랜잭션 있음
```

## 재발 방지

같은 클래스 내부에서 `@Transactional`이 붙은 메서드를 호출할 때는 반드시 `applicationContext.getBean()`을 통해 프록시를 거쳐야 한다. `this.메서드()` 로 직접 호출하면 `@Transactional`은 아무 효과가 없다.
