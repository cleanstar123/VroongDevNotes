# 트러블슈팅 - 선정산 배치 실패 (2026-05-20)

## 무슨 일이 있었나요?

2026-05-20 오전 11시 10분, 선정산 배치가 실패하여 라이더들이 당일 선정산을 받지 못했습니다.

에러 메시지는 이것이었습니다:
```
TransactionRequiredException: Executing an update/delete query
```

직역하면: **"UPDATE/DELETE 쿼리를 실행하려면 트랜잭션이 필요한데, 트랜잭션이 없습니다"**

---

## 트랜잭션이 뭔가요? (이해를 위한 배경 지식)

데이터베이스에 데이터를 **저장, 수정, 삭제**할 때는 반드시 **트랜잭션(Transaction)** 이 열려 있어야 합니다.

- **조회(SELECT)** 는 트랜잭션 없어도 됩니다.
- **저장/수정/삭제(INSERT/UPDATE/DELETE)** 는 트랜잭션이 반드시 있어야 합니다.

Spring에서는 메서드에 `@Transactional` 어노테이션을 붙이면 그 메서드가 실행될 때 자동으로 트랜잭션을 열어줍니다.

---

## `@Transactional`의 함정 - 같은 클래스 내부 호출

Spring의 `@Transactional`은 **한 가지 중요한 제약**이 있습니다.

> **같은 클래스 안에서 `this.메서드()` 로 직접 부르면 `@Transactional`이 작동하지 않습니다.**

왜 그럴까요?

Spring은 `@Transactional`을 처리하기 위해 **프록시(대리인)** 라는 중간 객체를 만듭니다.
외부에서 메서드를 호출할 때는 이 프록시가 중간에 끼어들어 트랜잭션을 열어줍니다.

그런데 같은 클래스 안에서 `this.메서드()` 로 직접 부르면 **프록시를 거치지 않고 직접 호출**됩니다.
프록시가 끼어들지 못하니 트랜잭션도 열리지 않습니다.

```
외부 → [프록시] → 메서드()    ← 트랜잭션 열림 ✅
외부 → [프록시] → A메서드() → this.B메서드()   ← 트랜잭션 안 열림 ❌
```

---

## 실제로 어디서 문제가 생겼나요?

`BaeminToVroongIfUseCase.java` 파일 안에서 메서드들이 이렇게 연결되어 있었습니다:

```
[컨트롤러]
  └─▶ migrateTable()          ← @Transactional 없음. 트랜잭션 안 열림.
        └─▶ processTableMigration()    ← this. 직접 호출. 트랜잭션 안 열림.
              └─▶ migrateVrRiderAcctDaily()   ← this. 직접 호출.
                                                @Transactional 있지만 무시됨!
                    └─▶ deleteDeductDailyByRiderAndDate()
                              ← DELETE 실행 시도
                              ← 트랜잭션 없음 → 💥 에러 발생
```

`migrateVrRiderAcctDaily()` 에는 `@Transactional(REQUIRES_NEW)` 이 붙어 있었지만,
`processTableMigration()` 이 **`this.`로 직접 호출**했기 때문에 프록시를 거치지 않아 완전히 무시된 것입니다.

결과적으로 트랜잭션이 전혀 없는 상태에서 DELETE를 실행하려다 에러가 난 것입니다.

---

## `isolation = READ_COMMITTED` 가 원인인가요?

아닙니다. 이 어노테이션 변경은 **원인이 아닙니다.**

`isolation = READ_COMMITTED` 는 메서드가 프록시를 통해 호출될 때만 효과가 있습니다.
`this.` 직접 호출 체인에서는 애초에 `@Transactional` 자체가 무시되므로, `isolation` 파라미터 추가 여부와 상관없이 동작이 달라지지 않습니다.

**실제 문제는** `deleteDeductDailyByRiderAndDate()` (DELETE 쿼리)가 처음으로 이 코드에 추가된 시점부터 잠재되어 있었습니다. 트랜잭션 없이 DELETE를 실행하는 코드가 배포된 것이 근본 원인입니다.

---

## 어떻게 고쳤나요?

`processTableMigration()` 에서 메서드를 직접 호출하는 대신,
**Spring이 관리하는 프록시 객체**를 통해 호출하도록 변경했습니다.

`applicationContext.getBean(BaeminToVroongIfUseCase.class)` 를 사용하면
Spring이 만들어둔 프록시 객체를 가져올 수 있고,
이 객체를 통해 메서드를 호출하면 `@Transactional` 이 정상 작동합니다.

### 수정 전 (문제 코드)

```java
case "partner_details_if":
    return migratePartnerDetailsIf(command, startTime);          // this. 직접 호출 → @Transactional 무시

case "partner_delivery_fees_if":
    return migratePartnerDeliveryFeesIf(command, startTime);     // this. 직접 호출 → @Transactional 무시

case "rider_info_if":
    return migrateRiderInfoIf(command, startTime);               // this. 직접 호출 → @Transactional 무시

case "vr_rider_acct_daily":
    return migrateVrRiderAcctDaily(command, startTime);          // this. 직접 호출 → @Transactional 무시

case "rider_acct_fee_deduct_daily":
    return migrateRiderFeeDeductDaily(command, startTime);       // this. 직접 호출 → @Transactional 무시

case "supp_acct_fee_deduct_daily":
    return migrateSuppFeeDeductDaily(command, startTime);        // this. 직접 호출 → @Transactional 무시
```

### 수정 후 (정상 코드)

```java
case "partner_details_if":
    return applicationContext.getBean(BaeminToVroongIfUseCase.class)
        .migratePartnerDetailsIf(command, startTime);            // 프록시 통해 호출 → @Transactional 정상 작동 ✅

case "partner_delivery_fees_if":
    return applicationContext.getBean(BaeminToVroongIfUseCase.class)
        .migratePartnerDeliveryFeesIf(command, startTime);       // 프록시 통해 호출 → @Transactional 정상 작동 ✅

case "rider_info_if":
    return applicationContext.getBean(BaeminToVroongIfUseCase.class)
        .migrateRiderInfoIf(command, startTime);                 // 프록시 통해 호출 → @Transactional 정상 작동 ✅

case "vr_rider_acct_daily":
    return applicationContext.getBean(BaeminToVroongIfUseCase.class)
        .migrateVrRiderAcctDaily(command, startTime);            // 프록시 통해 호출 → @Transactional 정상 작동 ✅

case "rider_acct_fee_deduct_daily":
    return applicationContext.getBean(BaeminToVroongIfUseCase.class)
        .migrateRiderFeeDeductDaily(command, startTime);         // 프록시 통해 호출 → @Transactional 정상 작동 ✅

case "supp_acct_fee_deduct_daily":
    return applicationContext.getBean(BaeminToVroongIfUseCase.class)
        .migrateSuppFeeDeductDaily(command, startTime);          // 프록시 통해 호출 → @Transactional 정상 작동 ✅
```

> **참고:** `rider_delivery_histories_if` 와 `rider_delivery_histories_now_if` 는 메서드 자체에 `@Transactional` 이 없고, 내부에서 이미 `getBean()` 패턴을 사용하고 있어서 수정하지 않았습니다.

---

## 수정 후 흐름 (정상)

```
[컨트롤러]
  └─▶ migrateTable()
        └─▶ processTableMigration()
              └─▶ applicationContext.getBean().migrateVrRiderAcctDaily()
                        ↑ 프록시를 통해 호출
                        → @Transactional(REQUIRES_NEW) 정상 작동
                        → 새 트랜잭션 열림
                              └─▶ deleteDeductDailyByRiderAndDate()
                                        ← 트랜잭션 있음 → ✅ 정상 실행
```

---

## 재발 방지 - 기억해야 할 것

같은 클래스 안에서 `@Transactional` 이 붙은 메서드를 호출할 때는,
**반드시 `applicationContext.getBean()`을 통해 프록시로 호출해야 합니다.**

| 호출 방식 | @Transactional 작동 여부 |
|-----------|--------------------------|
| `this.메서드()` | ❌ 작동 안 함 |
| `applicationContext.getBean(클래스.class).메서드()` | ✅ 정상 작동 |
| 외부 클래스에서 주입받아 호출 | ✅ 정상 작동 |
