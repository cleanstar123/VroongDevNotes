## 1. DB 커넥션 풀(HikariCP) 장애

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-02-20 12:23 ~ 12:32 |
| **영향 범위** | 전체 API 서버, 배치 작업 |
| **심각도** | 🔴 Critical |

### 증상

- 배치 작업 실패
- API 500 에러 다수 발생
- DB 커넥션 타임아웃

### 에러 로그

```
2026-02-20 12:23:08.821 WARN HikariPool-1 - Failed to validate connection
com.mysql.cj.jdbc.ConnectionImpl@152036fe (No operations allowed after connection closed.)
Possibly consider using a shorter maxLifetime value.

2026-02-20 12:23:17 HikariPool - Connection is not available, request timed out after 30000ms

2026-02-20 12:23:52 Communications link failure during rollback()
```

### 원인 분석

```
┌─────────────────────────────────────────────────────────────┐
│  API 서버 (HikariCP)              MySQL 서버                │
│                                                             │
│  커넥션 풀에 10개 연결 유지        wait_timeout 설정         │
│       📞──────────────────📞                                │
│                                                             │
│  ⏰ 유휴 시간 경과...                                        │
│                                                             │
│  MySQL: "timeout! 커넥션 끊음"                               │
│       📞─ ─ ─ ✂️ ─ ─ ─ ─📞  ← 서버가 몰래 끊어버림          │
│                                                             │
│  HikariCP: "커넥션 있으니까 바로 쓰자!"                      │
│           → 📞 들었는데 신호 없음 → 💥 장애                  │
└─────────────────────────────────────────────────────────────┘
```

**근본 원인**: `application.yml`에 커넥션 검증 설정 누락

| 문제 | 설명 |
|------|------|
| `max-lifetime` 미설정 | HikariCP 기본값 30분 사용, MySQL wait_timeout보다 길 수 있음 |
| `connection-test-query` 미설정 | 커넥션 사용 전 유효성 검증 안함 |

### 해결 방법

**application.yml 수정**

```yaml
# 기존 설정 (문제)
hikari:
  maximum-pool-size: 200
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000

# 수정된 설정 (해결)
hikari:
  maximum-pool-size: 200
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000
  max-lifetime: 1740000           # 29분 (MySQL wait_timeout보다 짧게)
  connection-test-query: SELECT 1 # 사용 전 커넥션 유효성 검증
  validation-timeout: 5000        # 검증 타임아웃 5초
```

### 장애 타임라인

| 시간 | 상황 |
|------|------|
| 12:20 이전 | 트래픽 적음, DB 커넥션 유휴 상태 |
| (일정 시간 경과) | MySQL이 유휴 커넥션 자동 종료 (wait_timeout) |
| 12:23:08 | 요청 시 커넥션 사용 시도 → 이미 끊어진 상태 감지 |
| 12:23:17 | 전체 풀 커넥션 invalid → 30초 타임아웃 발생 |
| 12:23~12:32 | 연쇄 장애 (API 실패, 배치 실패) |

### 예방 조치

| 설정                      | 설명        | 권장값                     |
| ----------------------- | --------- | ----------------------- |
| `max-lifetime`          | 커넥션 최대 수명 | MySQL wait_timeout - 1분 |
| `connection-test-query` | 커넥션 검증 쿼리 | `SELECT 1`              |
| `validation-timeout`    | 검증 타임아웃   | 5000ms                  |

### 수정 파일

```
api/src/main/resources/application.yml
```

## 2. 배민 클릭 횟수 과다 컴플레인

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-02-20 |
| **영향 범위** | relay-server, 배민 연동 |
| **심각도** | 🟡 Warning |

### 증상

- 배민 측에서 **버튼 클릭 횟수가 비정상적으로 많다**고 컴플레인
- 계정 제재 또는 API 차단 가능성 우려

### 원인 분석

```
┌─────────────────────────────────────────────────────────────┐
│  relay-server                     배민 어드민               │
│                                                             │
│  /relay/fast/partner/delivery-fee                           │
│       ↓                                                     │
│  Puppeteer (브라우저 자동화)                                 │
│       ↓                                                     │
│  실제 버튼 클릭 시뮬레이션 → 🖱️ 클릭! 🖱️ 클릭! 🖱️ 클릭!    │
│       ↓                                                     │
│  협력사별 반복 호출 (수십~수백 회)                           │
│       ↓                                                     │
│  배민: "클릭이 너무 많아요!" 😡                              │
└─────────────────────────────────────────────────────────────┘
```

**근본 원인**:
- `/relay/fast/partner/delivery-fee` API가 Puppeteer를 통해 실제 버튼 클릭 시뮬레이션
- 협력사별로 배달비 엑셀 다운로드를 위해 반복 호출
- 자동화된 클릭이 배민 서버에 과도한 부하

### 해결 방법

**relay-server/server.js 수정**

```javascript
// 기존 코드 주석 처리
// app.get('/relay/fast/partner/delivery-fee', async (req, res) => {
//     ...실제 버튼 클릭 로직...
// });

// 비활성화 stub으로 대체
app.get('/relay/fast/partner/delivery-fee', async (req, res) => {
    console.log('[Fast 배달비] 임시 비활성화됨 - 배민 요청으로 인한 조치');
    res.status(503).json({
        success: false,
        error: '배민 요청으로 인해 임시 비활성화되었습니다'
    });
});
```

### 예방 조치

| 권장 사항 | 설명 |
|-----------|------|
| 호출 빈도 제한 | Rate limiting 적용 (예: 1분당 10회) |
| 공식 API 사용 | 버튼 클릭 시뮬레이션보다 공식 API 우선 |
| 이용약관 확인 | 자동화 전 해당 서비스의 제한사항 확인 |
| 배민 협의 | 대량 데이터 필요 시 공식 채널 통해 협의 |

### 수정 파일

```
relay-server/server.js
```
