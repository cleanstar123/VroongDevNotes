## 1. DB 커넥션 풀(HikariCP) 장애

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-02-20 12:23 ~ 12:32 |
| **영향 범위** | 전체 API 서버, 배치 작업 |
| **심각도** | 🔴 Critical |

### 증상

- 배치 작업 실패

- DB 커넥션 타임아웃

```

│  API 서버 (HikariCP)              MySQL 서버                │
│                                                             │
```

**근본 원인**: `application.yml`에 커넥션 검증 설정 누락

| 문제 | 설명 |
|------|------|
| `max-lifetime` 미설정 | HikariCP 기본값 30분 사용, MySQL wait_timeout보다 길 수 있음 |
| `connection-test-query` 미설정 | 커넥션 사용 전 유효성 검증 안함 |

### 해결 방법

**application.yml 수정**

```yaml
# 기존 설정 (문제)
hikari:
  maximum-pool-size: 200
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000

# 수정된 설정 (해결)
hikari:
  maximum-pool-size: 200
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000
  max-lifetime: 1740000           # 29분 (MySQL wait_timeout보다 짧게)
  connection-test-query: SELECT 1 # 사용 전 커넥션 유효성 검증
  validation-timeout: 5000        # 검증 타임아웃 5초
```

### 장애 타임라인

| 시간 | 상황 |
|------|------|
| 12:20 이전 | 트래픽 적음, DB 커넥션 유휴 상태 |
| (일정 시간 경과) | MySQL이 유휴 커넥션 자동 종료 (wait_timeout) |
| 12:23:08 | 요청 시 커넥션 사용 시도 → 이미 끊어진 상태 감지 |
| 12:23:17 | 전체 풀 커넥션 invalid → 30초 타임아웃 발생 |

### 예방 조치

| 설정                      | 설명        | 권장값                     |
| ----------------------- | --------- | ----------------------- |
| `max-lifetime`          | 커넥션 최대 수명 | MySQL wait_timeout - 1분 |

## 2. 배민 클릭 횟수 과다 컴플레인

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-02-20 |
| **영향 범위** | relay-server, 배민 연동 |
| **심각도** | 🟡 Warning |

### 증상

- 배민 측에서 **버튼 클릭 횟수가 비정상적으로 많다**고 컴플레인
- 계정 제재 또는 API 차단 가능성 우려

### relay-server 아키텍처 (배경 지식)

relay-server는 배민 어드민 사이트에서 데이터를 수집하기 위해 두 가지 모드를 지원합니다.

#### 운영 모드 (Fast Mode - 직접 API 호출)

```
┌─────────────────────────────────────────────────────────────┐
│  VPay 서버                relay-server         배민 서버    │
│                                                             │
│  데이터 요청 ──────→  /relay/fast/*                         │
│                           ↓                                 │
│                      Axios HTTP 요청 ──────→ 배민 내부 API  │
│                           ↓                                 │
│                      JSON 데이터 ←────────── 응답           │
└─────────────────────────────────────────────────────────────┘
```

- 배민 내부 API를 **직접 HTTP 요청**으로 호출
- 페이지 네비게이션 없음, 속도 빠름 (Puppeteer 대비 10-20배)
- 버튼 클릭 없음

#### 개발/테스트 모드 (Puppeteer Mode - 브라우저 자동화)

```
┌─────────────────────────────────────────────────────────────┐
│  VPay 서버                relay-server         배민 서버    │
│                                                             │
│  데이터 요청 ──────→  /relay/*                              │
│                           ↓                                 │
│                      Puppeteer (Chromium)                   │
│                           ↓                                 │
│                      🖱️ 로그인 → 페이지 이동 → 버튼 클릭    │
│                           ↓                                 │
│                      실제 사용자처럼 행동 ────→ 배민 사이트  │
└─────────────────────────────────────────────────────────────┘
```

- 실제 브라우저(Chromium)를 띄워서 **사람처럼 행동**
- 로그인 → 페이지 이동 → 버튼 클릭 → 데이터 추출
- "사람이 사이트를 탐색 중"이라는 시그널을 배민 서버에 전달

#### Puppeteer 모드를 개발한 이유

| 문제 | 설명 |
|------|------|
| AWS IP 차단 | AWS 서버에서 직접 API 호출 시 **상업적 크롤링으로 판단**되어 세션 차단 |
| Cloudflare 방어 | 배민 사이트는 Cloudflare를 사용, 봇 트래픽 감지 시 차단 |

**우회 전략**:
1. relay-server를 **가정용 IP** (로컬 PC)에서 실행
2. **Puppeteer 브라우저 자동화**로 실제 사용자 행동 시뮬레이션
3. AWS 서버 → relay-server → 배민 순으로 요청 중계

#### 현재 운영 구조 (하이브리드)

| 용도 | 모드 | 설명 |
|------|------|------|
| 로그인/세션 유지 | Puppeteer | 실제 브라우저로 로그인, keep-alive |
| 데이터 조회 | Fast Mode | 직접 API 호출 (속도 최적화) |

### 원인 분석

```
┌─────────────────────────────────────────────────────────────┐
│  relay-server                     배민 어드민               │
│                                                             │
│  /relay/fast/partner/delivery-fee                           │
│       ↓                                                     │
│  직접 API 호출 (또는 Puppeteer 버튼 클릭)                    │
│       ↓                                                     │
│  협력사별 반복 호출: 59개 × 48회/일 = 2,832회/일            │
│       ↓                                                     │
│  배민 모니터링 시스템 감지                                   │
│       ↓                                                     │
│  배민: "자동화 접근이 너무 많아요!" 😡                       │
└─────────────────────────────────────────────────────────────┘
```

**근본 원인**:
- `/relay/fast/partner/delivery-fee` API가 협력사별로 배달비 엑셀 다운로드를 반복 호출
- **59개 협력사 × 30분 간격 배치(48회/일) = 하루 약 2,832회 호출**
- 자동화된 대량 접근이 배민 모니터링에 감지됨

### 해결 방법

**relay-server/server.js 수정**

```javascript
// 기존 코드 주석 처리
// app.get('/relay/fast/partner/delivery-fee', async (req, res) => {
//     ...실제 버튼 클릭 로직...
// });

// 비활성화 stub으로 대체
app.get('/relay/fast/partner/delivery-fee', async (req, res) => {
    console.log('[Fast 배달비] 임시 비활성화됨 - 배민 요청으로 인한 조치');
    res.status(503).json({
        success: false,
        error: '배민 요청으로 인해 임시 비활성화되었습니다'
    });
});
```

### Fast Mode(직접 API)로 변경하면 해결될까?

**결론: 아니오. 직접 API 호출 방식으로 변경해도 컴플레인 해결이 어렵습니다.**

| 관점 | 설명 |
|------|------|
| **호출 횟수 동일** | Fast Mode든 Puppeteer든 호출 횟수(2,832회/일)는 동일 |
| **더 쉬운 탐지** | 직접 API 호출은 패턴이 일정해 오히려 봇 탐지가 더 쉬움 |
| **속도 증가** | Fast Mode가 더 빠르므로 단시간 내 더 많은 요청 발생 가능 |
| **이용약관 위반** | 방식과 관계없이 자동화된 대량 데이터 수집 자체가 문제 |

```
┌─────────────────────────────────────────────────────────────┐
│  배민 모니터링 시스템이 감지하는 것                          │
│                                                             │
│  ❌ 버튼 클릭 vs API 호출 (구분하지 않음)                    │
│  ✅ 호출 빈도 (분당/시간당/일당 요청 수)                     │
│  ✅ 호출 패턴 (동일 시간대 반복, 기계적 간격)                │
│  ✅ 접근 계정 (하나의 계정으로 전체 협력사 접근)             │
│  ✅ 데이터 양 (전체 협력사 배달비 일괄 다운로드)             │
└─────────────────────────────────────────────────────────────┘
```

### 근본적 해결 방안

| 방안 | 설명 | 실현 가능성 |
|------|------|-------------|
| **배민 공식 API 협의** | 배민과 협의하여 공식 데이터 연동 API 사용 | ⭐⭐⭐ 권장 |
| **호출 빈도 대폭 감소** | 30분 → 6시간 또는 1일 1회로 변경 | ⭐⭐ 가능 |
| **필요 협력사만 조회** | 59개 전체가 아닌 변경된 협력사만 조회 | ⭐⭐ 가능 |
| **수동 업로드 전환** | 자동화 대신 담당자가 직접 다운로드 후 업로드 | ⭐ 비효율적 |

### 예방 조치

| 권장 사항 | 설명 |
|-----------|------|
| 호출 빈도 제한 | Rate limiting 적용 (예: 1분당 10회) |
| 공식 API 사용 | 버튼 클릭 시뮬레이션보다 공식 API 우선 |
| 이용약관 확인 | 자동화 전 해당 서비스의 제한사항 확인 |
| 배민 협의 | 대량 데이터 필요 시 공식 채널 통해 협의 |
