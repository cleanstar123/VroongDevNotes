## 1. DB 커넥션 풀(HikariCP) 장애

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-02-20 12:23 ~ 12:32 |

**근본 원인**: `application.yml`에 커넥션 검증 설정 누락

| 문제 | 설명 |
|------|------|
| `max-lifetime` 미설정 | HikariCP 기본값 30분 사용, MySQL wait_timeout보다 길 수 있음 |
| `connection-test-query` 미설정 | 커넥션 사용 전 유효성 검증 안함 |

### 해결 방법

**application.yml 수정**

```yaml
# 기존 설정 (문제)
hikari:
  maximum-pool-size: 200
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000

# 수정된 설정 (해결)
hikari:
  maximum-pool-size: 200
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000
  max-lifetime: 1740000           # 29분 (MySQL wait_timeout보다 짧게)
  connection-test-query: SELECT 1 # 사용 전 커넥션 유효성 검증
  validation-timeout: 5000        # 검증 타임아웃 5초
```

### 예방 조치

| 설정                      | 설명        | 권장값                     |

## 2. 배민 클릭 횟수 과다 컴플레인

### 개요

| 항목 | 내용 |
|------|------|
| **발생 시각** | 2026-02-20 |
| **영향 범위** | relay-server, 배민 연동 |
| **심각도** | 🟡 Warning |

### 증상

- 배민 측에서 **버튼 클릭 횟수가 비정상적으로 많다**고 컴플레인
- 계정 제재 또는 API 차단 가능성 우려

### relay-server 아키텍처 (배경 지식)

relay-server는 배민 어드민 사이트에서 데이터를 수집하기 위해 **운영/개발 환경별로 다른 방식**을 사용합니다.

#### 운영 환경 (prod 브랜치) - Axios + BrightData Proxy

```
┌─────────────────────────────────────────────────────────────┐
│  운영 환경 (prod)                                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [AWS 서버] ──요청──▶ [relay-server] ──▶ [BrightData Proxy] │
│                           │                    │            │
│                      Axios HTTP 요청      가정용 IP 우회    │
│                           │                    │            │
│                           └──────────────▶ [배민 API]       │
│                                                             │
│  ❌ Puppeteer 없음                                          │
│  ❌ 브라우저 없음                                            │
│  ❌ 페이지 네비게이션 없음                                    │
│  ❌ 버튼 클릭 없음                                           │
│  ✅ 순수 HTTP API 호출 (Axios)                               │
│  ✅ BrightData Residential Proxy (가정용 IP 우회)            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**baemin-service.js (prod)**:
- **Puppeteer 없이** 순수 **Axios**로 배민 API 직접 호출
- **BrightData Residential Proxy** 사용 (가정용 IP로 Cloudflare 우회)
- 로그인: RSA 암호화 → Axios POST 요청
- 데이터 조회: Axios로 배민 내부 API 직접 호출

#### 개발 환경 (dev 브랜치) - Puppeteer + Fast Mode

```
┌─────────────────────────────────────────────────────────────┐
│  개발 환경 (dev)                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [Puppeteer 브라우저] ← 항상 실행 필요                       │
│       │                                                     │
│       ├─ 로그인/2FA ──────→ 실제 페이지 이동 + 버튼 클릭     │
│       │                                                     │
│       └─ Fast API ───────→ page.evaluate() + fetch()        │
│           (/relay/fast/*)      (페이지 이동 없이 직접 호출)  │
│                                                             │
│  ✅ Puppeteer 브라우저 (Chromium)                            │
│  ✅ 로그인 시 페이지 자동화                                   │
│  ✅ Fast Mode: fetch()로 직접 API 호출 (속도 최적화)         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**services/baemin/fast-api.js (dev)**:
- Puppeteer 브라우저 컨텍스트 사용 (쿠키/세션)
- `page.evaluate()` 내에서 `fetch()` 실행
- 페이지 이동 없이 직접 API 호출

#### 운영 vs 개발 비교

| 구분 | prod (운영) | dev (개발) |
|------|------------|-----------|
| **API 호출 방식** | Axios + BrightData Proxy | Puppeteer + fetch() |
| **브라우저** | ❌ 없음 | ✅ Chromium |
| **Proxy** | ✅ BrightData (가정용 IP) | Optional |
| **페이지 네비게이션** | ❌ 없음 | 로그인 시에만 |
| **Fast Mode 엔드포인트** | ❌ 없음 | ✅ `/relay/fast/*` |
| **로그인 방식** | RSA 암호화 → Axios POST | Puppeteer 페이지 자동화 |

#### Puppeteer 모드를 개발한 이유

| 문제 | 설명 |
|------|------|
| AWS IP 차단 | AWS 서버에서 직접 API 호출 시 **상업적 크롤링으로 판단**되어 세션 차단 |
| Cloudflare 방어 | 배민 사이트는 Cloudflare를 사용, 봇 트래픽 감지 시 차단 |

**우회 전략 (운영)**:
1. **BrightData Residential Proxy** 사용 (가정용 IP로 요청)
2. AWS 서버 → relay-server → BrightData Proxy → 배민 API

**우회 전략 (개발/테스트)**:
1. relay-server를 **로컬 PC**에서 실행
2. **Puppeteer 브라우저 자동화**로 실제 사용자 행동 시뮬레이션
3. AWS 서버 → ngrok → 로컬 relay-server → 배민

### 원인 분석

```
┌─────────────────────────────────────────────────────────────┐
│  relay-server (운영)                   배민 어드민          │
│                                                             │
│  /relay/partner/delivery-fee                                │
│       ↓                                                     │
│  Axios + BrightData Proxy (직접 API 호출)                   │
│       ↓                                                     │
│  협력사별 반복 호출: 59개 × 48회/일 = 2,832회/일            │
│       ↓                                                     │
│  배민 모니터링 시스템 감지                                   │
│       ↓                                                     │
│  배민: "자동화 접근이 너무 많아요!" 😡                       │
└─────────────────────────────────────────────────────────────┘
```

**근본 원인**:
- 운영 환경에서 `/relay/partner/delivery-fee` API가 **Axios로 직접 배민 API 호출**
- 협력사별로 배달비 엑셀 다운로드를 반복 호출
- **59개 협력사 × 30분 간격 배치(48회/일) = 하루 약 2,832회 호출**
- 자동화된 대량 접근이 배민 모니터링에 감지됨
- **버튼 클릭이 문제가 아님** - 직접 API 호출도 동일하게 감지됨

### 해결 방법

**임시 조치**: 협력사 배달비 API 비활성화

운영 환경에서는 `/relay/partner/delivery-fee` 엔드포인트가 있으며, 해당 API 호출을 일시 중단했습니다.

```javascript
// relay-server/server.js (prod)
// 협력사 배달비 조회 API - 배민 요청으로 임시 비활성화
app.get('/relay/partner/delivery-fee', async (req, res) => {
    console.log('[협력사 배달비] 임시 비활성화됨 - 배민 요청으로 인한 조치');
    res.status(503).json({
        success: false,
        skipped: true,
        message: '협력사 배달비 조회가 일시적으로 비활성화되었습니다'
    });
});
```

**참고**: 개발 환경(dev 브랜치)에서는 `/relay/fast/partner/delivery-fee` 엔드포인트도 동일하게 비활성화 처리되어 있습니다.

### 호출 방식을 바꾸면 해결될까?

**결론: 아니오. 운영 환경은 이미 직접 API 호출 방식(Axios)을 사용 중이며, 방식 변경으로는 해결되지 않습니다.**

| 관점 | 설명 |
|------|------|
| **현재 운영 방식** | 이미 Axios로 직접 API 호출 중 (Puppeteer 아님) |
| **호출 횟수가 문제** | API 호출 방식과 무관하게 **2,832회/일** 호출이 문제 |
| **패턴 탐지** | 30분 간격, 59개 협력사 순차 호출 패턴이 봇으로 감지됨 |
| **이용약관** | 방식과 관계없이 자동화된 대량 데이터 수집 자체가 문제 |

```
┌─────────────────────────────────────────────────────────────┐
│  배민 모니터링 시스템이 감지하는 것                          │
│                                                             │
│  ❌ 버튼 클릭 vs API 호출 (구분하지 않음)                    │
│  ✅ 호출 빈도 (분당/시간당/일당 요청 수)                     │
│  ✅ 호출 패턴 (동일 시간대 반복, 기계적 간격)                │
│  ✅ 접근 계정 (하나의 계정으로 전체 협력사 접근)             │
│  ✅ 데이터 양 (전체 협력사 배달비 일괄 다운로드)             │
└─────────────────────────────────────────────────────────────┘
```

### 근본적 해결 방안

| 방안 | 설명 | 실현 가능성 |
|------|------|-------------|
| **배민 공식 API 협의** | 배민과 협의하여 공식 데이터 연동 API 사용 | ⭐⭐⭐ 권장 |
| **호출 빈도 대폭 감소** | 30분 → 6시간 또는 1일 1회로 변경 | ⭐⭐ 가능 |
| **필요 협력사만 조회** | 59개 전체가 아닌 변경된 협력사만 조회 | ⭐⭐ 가능 |
| **수동 업로드 전환** | 자동화 대신 담당자가 직접 다운로드 후 업로드 | ⭐ 비효율적 |

### 예방 조치

| 권장 사항 | 설명 |
