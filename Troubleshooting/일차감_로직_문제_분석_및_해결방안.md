# 일차감(日次減) 로직 문제 분석 및 해결 방안

> 작성일: 2026-04-30
> 분석 대상: `RiderSettlementUseCase.java`, `PartnerSettlementUseCase.java`, `BatchHeadBalAmtUseCase.java`

---

## 1. 배경 및 도메인 개념

### 주요 테이블

| 테이블                    | 역할                                                  |
| ---------------------- | --------------------------------------------------- |
| `vr_deduct_mst`        | 일차감 규칙 정의 (DEDUCT_TYPE, CALC_TYPE, DEDUCT_TARGET 등) |
| `vr_rider_acct_daily`  | 선정산 결과 저장 (DEDUCT_VAL 포함)                           |
| `vr_rider_acct_weekly` | 라이더 주정산 결과 저장                                       |
| `vr_supp_acct_weekly`  | 협력사 주정산 결과 저장                                       |
| `vr_account_mst`       | 원장 (BAL_AMT, TRAN_AMT, UNPAID_AMT, ACCT_TYPE)       |

### 주요 enum 값

| 항목            | 값                           | 설명                            |
| ------------- | --------------------------- | ----------------------------- |
| ACCT_TYPE     | RW / RP / SW / RD / SD / RR | 라이더 주정산 / 라이더 선정산 / 협력사 주정산 등 |
| DEDUCT_TARGET | R / SRA / S                 | 특정 라이더 / 협력사 전체 라이더 / 협력사     |
| CALC_TYPE     | 1 / 2                       | 정액 / 정률                       |
| DEDUCT_TYPE   | DAILY / WORK_DAY            | 매일 / 배달 발생일                   |

### 정산 흐름 요약

```
[선정산 - 매일]
  라이더 배달료 발생
    → 일차감 적용 (DEDUCT_TYPE=DAILY or WORK_DAY)
    → vr_rider_acct_daily 저장
        - DEDUCT_VAL: 당일 일차감 금액
        - TOT_ACCT_VAL: 당일 최종 정산금액 (배달료에서 차감 등 적용 후)
    → vr_account_mst (ACCT_TYPE=RP) 원장 반영

[주정산 - 주 1회, 엑셀 업로드 기반]
  엑셀에 포함된 라이더만 처리 (선정산 전용 라이더는 미포함될 수 있음)
  주간 배달료 합산
    → preAcctVal = SUM(vr_rider_acct_daily.TOT_ACCT_VAL) for the week
      (이미 지급된 선정산 금액 합계 - 직접 컬럼이 아닌 집계값)
    → totalDeductAmt = vr_deduct_mst 규칙 기반 주간 전체 차감액 계산
      (주정산 엑셀의 배달료는 일차감 미적용 원본값이므로 주정산에서도 별도 차감)
    → 최종 지급액(totAcctVal) = finalDeliveryVal - preAcctVal - totalDeductAmt
    → vr_rider_acct_weekly 저장
    → vr_account_mst (ACCT_TYPE=RW) 원장 반영

[협력사 주정산]
  라이더 배달료 집계
    → 수수료, 공제(deductVal) 계산
    → actualPayment = paymentMgmtFee + deductVal - insuranceTotal
    → vr_supp_acct_weekly 저장
    → vr_account_mst (ACCT_TYPE=SW) 원장 반영

[Vroong 예치금 잔액 배치]
  예치금 잔액 = 이전잔액 - SW - RW - RP
```

## 2. 문제 분석

### 문제 1: 라이더 음수 정산 발생 (가장 근본적인 문제)

**위치**: `RiderSettlementUseCase.java:665`

```java
// 실지급 배달료 = 최종배달료 - 선정산배달료(절대값) - 주정산 일차감
BigDecimal totAcctVal = calc.finalDeliveryVal.subtract(preAcctVal).subtract(totalDeductAmt);
```

**문제**: `totAcctVal`의 음수 여부를 검사하지 않고 그대로 저장한다.

**발생 시나리오 (주정산)**:
- 라이더 주간 배달료: 50,000원
- 선정산 수령액(preAcctVal): 45,000원
- 주정산 일차감(totalDeductAmt): 20,000원
- `totAcctVal = 50,000 - 45,000 - 20,000 = **-15,000원**`

이 음수 값이 `vr_account_mst.TRAN_AMT`에 그대로 기록된다.

**선정산 전용 라이더의 경우 (주정산 엑셀에 포함되지 않는 라이더)**:

주정산 처리는 엑셀 업로드 기반이므로, **선정산만 받고 주정산 엑셀에 포함되지 않는 라이더**가 존재한다.
이 경우 음수 정산 문제는 `vr_rider_acct_daily.TOT_ACCT_VAL` 레벨에서만 발생하며, 주정산을 통한 사후 보정 경로가 없다.

- 일차감 > 당일 배달료이면 `TOT_ACCT_VAL` 자체가 음수로 기록됨
- 해당 라이더는 주정산 원장(RW)이 생성되지 않으므로, 미수금을 회수할 다음 정산 시점도 없음

**파급 효과**:
- 라이더의 `BAL_AMT` (잔액)이 음수가 됨
- 실제로 라이더가 Vroong에 돈을 **반납해야 하는** 상황이 발생
- 선정산 전용 라이더는 미수금 회수 경로 자체가 없어 손실이 확정됨

---

### 문제 2: UNPAID_AMT(미수금)가 계산되지만 다음 정산에서 활용되지 않음

**위치**: `RiderSettlementUseCase.java:1697~1714` (`createAccountEntityWithType`)

```java
BigDecimal tranAmt = settlement.getTotAcctVal() != null ? settlement.getTotAcctVal() : BigDecimal.ZERO;
BigDecimal newBalance = previousBalance.add(tranAmt);
BigDecimal unpaidAmt = newBalance.compareTo(BigDecimal.ZERO) < 0 ? newBalance : BigDecimal.ZERO;

return VrAccountMstEntity.builder()
    ...
    .unpaidAmt(unpaidAmt)  // 저장은 되지만...
    .build();
```

**문제**: `unpaidAmt`가 `vr_account_mst`에 저장되기는 하지만, 이후 선정산/주정산 계산 시 이 값을 읽어서 차감 보전에 활용하는 코드가 **존재하지 않는다**.

즉, **미수금이 쌓여도 다음 정산에서 자동으로 회수되지 않는다**.

**발생 시나리오**:
- 주차 1: 차감 20,000원 → 정산 -15,000원 → `UNPAID_AMT = -15,000`
- 주차 2: 배달료 충분 → 일차감 20,000원 그대로 적용
- **주차 1의 미수금 15,000원은 영구 손실**

---

> **설계 메모 - `preDeductVal` 미사용 (의도된 설계)**
>
> `RiderSettlementUseCase.java:636`에 아래 코드가 주석 처리되어 있다.
> ```java
> // BigDecimal actualDeductAmt = totalDeductAmt.subtract(preDeductVal).max(BigDecimal.ZERO);
> ```
> 이는 버그가 아닌 **의도된 설계**다. 주정산 엑셀의 배달 금액은 일차감이 적용되지 않은 원본 금액이다.
> 따라서 주정산에서 `totalDeductAmt`(전체 주간 차감)를 별도로 빼는 것이 올바른 동작이며,
> `preDeductVal`을 차감하면 오히려 일차감이 누락된다.

### 문제 4: 협력사 주정산 DEDUCT_VAL이 이론값 기반

**위치**: `PartnerSettlementUseCase.java:1097~1106`

```java
deductVal = calculateDeductionOptimized(
    supplyId,
    agg.deliveryMap,
    metadata.acctStDate,
    metadata.acctEdDate,
    batchData.deductsBySuppId.getOrDefault(supplyId, new ArrayList<>()),
    batchData.riderMap,
    batchData.workDaysMap,
    batchData.contractPeriodInfoMap);
```

**문제**: 협력사의 `deductVal`은 `vr_deduct_mst` 규칙에서 **이론적으로** 계산된 값이다. 실제로 라이더에게 차감이 적용됐는지(음수 정산으로 차감 실패했는지) 여부를 **반영하지 않는다**.

**발생 시나리오**:
- 라이더 A의 이론 차감액: 20,000원
- 라이더 A의 실제 배달료가 부족해서 차감 실패 → Vroong이 자체 예치금에서 20,000원 지급
- 협력사 정산 시에도 `deductVal` = 20,000원이 그대로 반영됨
- **Vroong이 이중으로 손실**: 라이더 미수금 + 협력사 과지급

### 문제 5: Vroong 예치금(헤드 잔액) 계산이 음수 TRAN_AMT를 정확히 반영하지 못함


**위치**: `BatchHeadBalAmtUseCase.java:63~98`

```java
BigDecimal swAmount = tranAmtSumByType.getOrDefault("SW", BigDecimal.ZERO);
BigDecimal rwAmount = tranAmtSumByType.getOrDefault("RW", BigDecimal.ZERO);
BigDecimal rpAmount = tranAmtSumByType.getOrDefault("RP", BigDecimal.ZERO);

// SW 처리
if (swAmount.compareTo(BigDecimal.ZERO) > 0) {
    currentBalance = currentBalance.subtract(swAmount);
}
// RW 처리
if (rwAmount.compareTo(BigDecimal.ZERO) > 0) {
    currentBalance = currentBalance.subtract(rwAmount);
}
```

**문제**: `tranAmtSumByType`은 해당 날짜의 ACCT_TYPE별 TRAN_AMT **합계**다. RW 정산에서 음수 totAcctVal이 포함되면 SUM이 줄어든다.

예시:
- 라이더 B: TRAN_AMT = +80,000원
- 라이더 C: TRAN_AMT = -15,000원 (음수 정산)
- RW 합계 = 65,000원
- 배치가 65,000원만 차감 → 실제로는 80,000원이 나갔어야 함
- **Vroong 예치금 15,000원 손실**이 정확히 기록되지 않음

## 3. 문제 전체 요약

```
vr_deduct_mst (차감 규칙)
        │
        ▼
  라이더 선정산 (RP)  ← 선정산 전용 라이더 → 주정산 없음 → 미수 회수 불가
  당일 배달료 < 일차감이면?
        │
        ▼
  TOT_ACCT_VAL = 음수 (vr_rider_acct_daily)
        │
        ▼

        │
        ▼
  라이더 주정산 (RW)

        │                                                  │

  totAcctVal = 음수                          협력사 주정산 (SW)
  vr_account_mst RW                         deductVal = 이론값 그대로

                                                    ▼
                                         Vroong 예치금 손실
                                      (배치 잔액 계산도 불정확)
```

## 4. 해결 방안 프로세스

### Step 1. 라이더 정산 시 음수 방지 - "차감 유보" 로직 적용

**수정 위치**: 선정산 일별 계산부 + `RiderSettlementUseCase.java` 주정산 계산부(665라인)

**변경 전**:
```java
BigDecimal totAcctVal = calc.finalDeliveryVal.subtract(preAcctVal).subtract(totalDeductAmt);
```

**변경 후 (개념)**:
```java
BigDecimal baseVal = calc.finalDeliveryVal.subtract(preAcctVal);

// 정산 가능 금액 내에서만 차감
BigDecimal actualDeductAmt;
BigDecimal deferredDeductAmt;

if (baseVal.compareTo(totalDeductAmt) >= 0) {
    // 정산액이 충분: 전액 차감
    actualDeductAmt = totalDeductAmt;
    deferredDeductAmt = BigDecimal.ZERO;
} else if (baseVal.compareTo(BigDecimal.ZERO) > 0) {
    // 정산액이 부족하지만 일부 차감 가능
    actualDeductAmt = baseVal;
    deferredDeductAmt = totalDeductAmt.subtract(baseVal);
} else {
    // 정산액이 0 이하: 차감 전액 유보
    actualDeductAmt = BigDecimal.ZERO;
    deferredDeductAmt = totalDeductAmt;
}

BigDecimal totAcctVal = baseVal.subtract(actualDeductAmt); // 항상 >= 0
```

**결과**:
- `totAcctVal`은 항상 `>= 0`이 됨
- `deferredDeductAmt`는 `UNPAID_AMT`(미수 차감액)로 저장

---

### Step 2. 미수금(UNPAID_AMT) 이월 및 회수 로직 구현

**수정 위치**: `RiderPreSettlementUseCase.java` (선정산), `RiderSettlementUseCase.java` (주정산)

**변경 내용**:

1. 정산 계산 전 `vr_account_mst`에서 해당 라이더의 누적 `UNPAID_AMT` 조회
2. 정산 가능 금액(`baseVal`)이 미수금을 초과하면 미수금 먼저 회수
3. 미수금 회수 후 남은 금액에 대해 당기 일차감 적용

**처리 우선순위**:
```
baseVal (정산 가능 금액)
  1순위: 이월 미수금(UNPAID_AMT) 회수
  2순위: 당기 일차감(totalDeductAmt)
  3순위: 실 지급 (totAcctVal)
```

**예시**:
- 이월 미수금: 15,000원
- 당기 배달료(baseVal): 60,000원
- 당기 일차감: 20,000원
- 미수금 회수: 15,000원
- 남은 금액: 60,000 - 15,000 = 45,000원
- 당기 차감 적용: 45,000 - 20,000 = 25,000원 지급
- UNPAID_AMT 갱신: 0

**DB 변경**:
- `vr_account_mst`에 `RECOVERED_AMT` (이번 정산에서 회수된 미수금) 컬럼 추가 고려
- 또는 별도 미수금 이력 테이블(`vr_unpaid_hist`) 관리

---

### Step 3. 협력사 DEDUCT_VAL을 실제 차감 합산으로 변경

**수정 위치**: `PartnerSettlementUseCase.java`의 `calculateDeductionOptimized`

**변경 방향**:
- `vr_deduct_mst` 기반 이론값 계산이 아닌
- 해당 정산 기간 내 `vr_rider_acct_daily.DEDUCT_VAL` + `vr_rider_acct_weekly.DEDUCT_AMT` 합산으로 변경

```java
// 변경 전: 이론값 계산
deductVal = calculateDeductionOptimized(supplyId, ...);

// 변경 후: 실제 적용된 차감 합산 조회
deductVal = riderSettlementRepository.sumActualDeductBySupplyIdAndPeriod(
    supplyId, acctStDate, acctEdDate);
```

**장점**:
- 협력사 정산이 실제 차감 반영 → Vroong 예치금 손실 차단
- 라이더 미수 발생 시 협력사도 동일하게 해당 금액 미수로 처리

---

### Step 4. 배치 헤드 잔액 계산 정확도 향상

**수정 위치**: `BatchHeadBalAmtUseCase.java`

**현재 문제**: `tranAmtSumByType`에 음수 TRAN_AMT가 섞이면 합계가 줄어들어 실제 지출보다 적게 차감됨.

**변경 방향**:
```java
// 변경 전: 합계가 양수일 때만 차감
if (rwAmount.compareTo(BigDecimal.ZERO) > 0) {
    currentBalance = currentBalance.subtract(rwAmount);
}

// 변경 후: 양수 지급분과 음수 환수분 분리 조회
BigDecimal rwPaidOut   = sumPositiveTranAmtByType("RW", acctDate);  // 실제 지급
BigDecimal rwRecovered = sumNegativeTranAmtByType("RW", acctDate);  // 미수로 인한 환수
currentBalance = currentBalance.subtract(rwPaidOut).add(rwRecovered.abs());
```

또는 Step 1~4가 완료되어 TRAN_AMT가 항상 `>= 0`이 보장되면 현재 배치 로직은 그대로 사용 가능.

## 6. 데이터 스키마 변경 제안

### vr_account_mst 추가 컬럼 (선택)

| 컬럼명 | 타입 | 설명 |
|---|---|---|
| `DEFERRED_DEDUCT_AMT` | DECIMAL | 이번 정산에서 유보된 차감액 |
| `RECOVERED_UNPAID_AMT` | DECIMAL | 이번 정산에서 회수된 미수금 |

### vr_rider_acct_daily / vr_rider_acct_weekly

- `DEDUCT_VAL`: 현재 이론값 저장 → **실제 차감액**으로 변경 필요
- 이론값도 보관하고 싶다면 `EXPECTED_DEDUCT_VAL` 별도 컬럼 추가

## 7. 테스트 케이스 제안

### 정상 케이스
- [x] 배달료 > 차감액: 전액 차감, totAcctVal > 0

### 음수 방지 케이스
- [ ] 배달료 < 차감액: 부분 차감, UNPAID_AMT 저장, totAcctVal = 0
- [ ] 배달료 = 0: 전액 유보, totAcctVal = 0
