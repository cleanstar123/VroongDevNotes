## 2. 자동 페이백 기능 테스트

### 2-1. 기능 설명

```
달성페이백 = vpay 등록 라이더 처리건수 × 페이백 금액 (예: 30원)

조건:
- 관리비(MAINTNC_VAL) > 0
- vr_fee_mst에 PAYBACK_USE_YN = 'Y', PAYBACK_AMT > 0 설정
- FEE_TYPE = 'C', SETTLEMENT_PERIOD = 'WEEKLY', FEE_TARGET = 'SRA'
```

### 2-2. 테스트 케이스

#### TC-2.1: 포탈에서 페이백 설정 저장

| 항목 | 내용 |
|------|------|
| **전제조건** | 포탈 로그인 |
| **테스트 방법** | 수수료 설정 > 페이백 금액 입력 > 저장 |
| **예상 결과** | 저장 성공, vr_fee_mst에 반영 |
| **확인 방법** | DB 쿼리 |

```sql
-- 페이백 설정 확인
SELECT SEQ_NO, SUPP_ID, FEE_TYPE, PAYBACK_AMT, PAYBACK_USE_YN
FROM vr_fee_mst
WHERE SUPP_ID = '{테스트 협력사 ID}'
  AND FEE_TYPE = 'C'
  AND SETTLEMENT_PERIOD = 'WEEKLY'
  AND FEE_TARGET = 'SRA';
```

#### TC-2.2: 협력사 주정산 업로드 시 달성페이백 계산

| 항목 | 내용 |
|------|------|
| **전제조건** | 페이백 설정 완료 (PAYBACK_AMT = 30, PAYBACK_USE_YN = 'Y') |
| **테스트 방법** | 협력사 주정산 업로드 실행 |
| **예상 결과** | 달성페이백 = vpay 라이더 처리건수 × 30원 |
| **확인 방법** | vr_supp_acct_weekly 테이블 확인 |

```sql
-- 주정산 업로드 후 달성페이백 확인
SELECT
  SUPP_ID,
  SETTLE_FROM,
  SETTLE_TO,
  MAINTNC_VAL,           -- 관리비
  ACHIEVEMENT_PAYBACK,   -- 달성페이백
  MAINTNC_FEE            -- 지급관리비
FROM vr_supp_acct_weekly
WHERE SUPP_ID = '{테스트 협력사 ID}'
  AND SETTLE_FROM = '{정산시작일}'
ORDER BY INS_DATE DESC
LIMIT 1;
```

#### TC-2.3: 관리비 없는 협력사는 페이백 0원

| 항목 | 내용 |
|------|------|
| **전제조건** | 협력사 관리비(MAINTNC_VAL) = 0 |
| **테스트 방법** | 협력사 주정산 업로드 |
| **예상 결과** | ACHIEVEMENT_PAYBACK = 0 |
| **확인 방법** | DB 쿼리 |

```sql
-- 관리비 0인 협력사 페이백 확인
SELECT SUPP_ID, MAINTNC_VAL, ACHIEVEMENT_PAYBACK
FROM vr_supp_acct_weekly
WHERE MAINTNC_VAL = 0
  AND SETTLE_FROM = '{정산시작일}';
-- 예상: ACHIEVEMENT_PAYBACK = 0
```

## 4. Race Condition 오류 수정 테스트

### 4-1. 수정 내용

- 잔액 조회 및 업데이트 메서드에 비관적 락(Pessimistic Lock) 추가
- 동시 요청 시 데이터 정합성 보장

### 4-2. 테스트 케이스

#### TC-4.1: 동시 잔액 조회/업데이트 요청

| 항목 | 내용 |
|------|------|
| **전제조건** | 동일 계좌에 대한 동시 요청 |
| **테스트 방법** | 여러 스레드에서 동시에 잔액 업데이트 API 호출 |
| **예상 결과** | 데이터 정합성 유지, 에러 없음 |
| **확인 방법** | 최종 잔액 = 초기잔액 + (입금액 합계) - (출금액 합계) |

```java
// 테스트 시나리오 예시
// 초기 잔액: 100,000원
// 동시 요청: +10,000원 × 5회, -5,000원 × 3회
// 예상 최종 잔액: 100,000 + 50,000 - 15,000 = 135,000원
```

#### TC-4.2: 에러 로그 없음 확인

| 항목 | 내용 |
|------|------|
| **테스트 방법** | 정산 배치 실행 중 로그 모니터링 |
| **예상 결과** | Race Condition 관련 에러 없음 |
| **확인 방법** | 로그 검색 |

```bash
# 로그에서 에러 검색
grep -i "race\|deadlock\|concurrent" application.log
```

## 6. 수수료 주 중간 변경 시 계산 오류 수정 테스트

### 6-1. 수정 내용

```
기존: 정산 기간 내 하나의 수수료만 적용 (최신 수수료)
변경: 날짜별로 해당 시점의 수수료 적용

예시 (정산 기간: 3/4 ~ 3/10)
- 3/4 ~ 3/8: 20원/건
- 3/9 ~ 3/10: 120원/건

기존 계산: 130건 × 120원 = 15,600원 (잘못됨)
수정 계산: 96건×20원 + 34건×120원 = 6,000원 (정확함)
```

### 6-2. 테스트 케이스

#### TC-6.1: 주 중간 수수료 변경 시 날짜별 계산

| 항목 | 내용 |
|------|------|
| **전제조건** | 라이더 수수료가 정산 기간 중간에 변경됨 |
| **테스트 방법** | 라이더 주정산 업로드 |
| **예상 결과** | 날짜별로 해당 시점 수수료 적용 |
| **확인 방법** | DB 쿼리 + 로그 확인 |

```sql
-- 1. 수수료 설정 확인
SELECT SEQ_NO, SUPP_ID, RIDER_ID, FEE_AMT, START_DAY, END_DAY, USE_YN
FROM vr_fee_mst
WHERE SUPP_ID = '{테스트 협력사 ID}'
  AND RIDER_ID = '{테스트 라이더 ID}'
  AND FEE_TYPE = 'S'
ORDER BY START_DAY DESC;

-- 2. 정산 결과 확인
SELECT
  SUPP_ID,
  RIDER_ID,
  SETTLE_FROM,
  SETTLE_TO,
  PARTNER_FEE  -- 협력사 수수료
FROM vr_rider_acct_weekly
WHERE SUPP_ID = '{테스트 협력사 ID}'
  AND RIDER_ID = '{테스트 라이더 ID}'
ORDER BY INS_DATE DESC
LIMIT 1;
```

#### TC-6.2: 수수료 변경 없는 라이더는 기존 로직 유지

| 항목 | 내용 |
|------|------|
| **전제조건** | 라이더 수수료가 정산 기간 동안 변경 없음 |
| **테스트 방법** | 라이더 주정산 업로드 |
| **예상 결과** | 기존 로직으로 계산 (추가 쿼리 없음) |
| **확인 방법** | 정산 결과 확인 |

#### TC-6.3: 수수료 3회 이상 변경 시 계산

| 항목 | 내용 |
|------|------|
| **전제조건** | 정산 기간 내 수수료 3회 변경 |
| **테스트 방법** | 라이더 주정산 업로드 |
| **예상 결과** | 모든 수수료가 날짜별로 정확히 적용 |
| **확인 방법** | 계산 검증 |

**테스트 시나리오:**
```
정산 기간: 3/4 ~ 3/10
수수료 변경:
- 3/4 ~ 3/5: 20원/건
- 3/6 ~ 3/8: 50원/건
- 3/9 ~ 3/10: 100원/건

배달 건수:
- 3/4: 10건, 3/5: 15건 → 25건 × 20원 = 500원
- 3/6: 20건, 3/7: 10건, 3/8: 15건 → 45건 × 50원 = 2,250원
- 3/9: 12건, 3/10: 8건 → 20건 × 100원 = 2,000원

예상 총 수수료: 500 + 2,250 + 2,000 = 4,750원
```

```sql
-- 협력사 잔액 원복
UPDATE vr_account_mst

UPDATE vr_rider_info

  AND RIDER_ID = '{테스트 라이더 ID}';

UPDATE vr_account_mst
SET BAL_AMT = {원래값}

  AND RIDER_ID = '{테스트 라이더 ID}'
  AND ACCT_TARGET = 'R';
```
