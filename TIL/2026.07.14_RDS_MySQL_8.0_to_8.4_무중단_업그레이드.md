## 핵심 개념: 왜 새 인스턴스 방식인가?

```
[in-place 업그레이드]
vraccountdb (8.0) → 업그레이드 → vraccountdb (8.4)
→ 업그레이드 중 DB 재시작 (5~15분 다운타임)
→ 문제 발생 시 롤백 = 스냅샷 복원 (30분 이상 소요)

[새 인스턴스 방식 (권장)]
vraccountdb (8.0) ──────────────────────────────── 그대로 유지
         ↓ 스냅샷
vraccountdb-84 (8.4) ← 여기서 모든 검증 완료 후
         ↓ 엔드포인트만 전환 (ECS 무중단 배포)
앱 → vraccountdb-84 (8.4)  (vraccountdb-8.0 은 1주일 후 삭제)
```

**장점**: DB 자체 다운타임 없음 / 롤백이 엔드포인트 변경 하나로 끝남

## STEP 2. 테스트 인스턴스 생성 (MySQL 8.4)

> 스냅샷으로 새 인스턴스를 만들면서 MySQL 버전을 8.4로 변경

```
RDS → 스냅샷 → vraccountdb-snapshot-20260714 선택
→ 작업(Actions) → 스냅샷 복원(Restore snapshot)
```

복원 설정:

| 항목 | 값 |
|------|-----|
| DB 인스턴스 식별자 | `vraccountdb-84` |
| DB 엔진 버전 | **MySQL 8.4.x** (드롭다운에서 선택) |
| DB 인스턴스 클래스 | `db.t4g.micro` (운영과 동일) |
| VPC | 운영 VPC와 동일하게 선택 |
| 퍼블릭 액세스 | 아니요 (운영 설정 유지) |
| VPC 보안 그룹 | 운영 vraccountdb와 동일한 보안 그룹 선택 |

> **중요**: VPC 보안 그룹을 운영과 동일하게 설정해야 로컬/앱 서버에서 접근 가능

복원 완료 후 새 엔드포인트 기록:
```
vraccountdb-84.xxxxxxxxxx.ap-northeast-2.rds.amazonaws.com
```

## STEP 3. mysql_native_password 인증 플러그인 확인 및 수정

> MySQL 8.4에서 `mysql_native_password` 플러그인이 기본 비활성화됨  
> 미처리 시 앱에서 **"Access denied"** 오류 발생

**테스트 인스턴스에서 확인 (MySQL Workbench 또는 DBeaver로 접속):**
```sql
SELECT user, plugin FROM mysql.user WHERE user = 'admin';
```

| 결과 | 조치 |
|------|------|
| `caching_sha2_password` | 이미 호환 → 조치 불필요 |
| `mysql_native_password` | 아래 쿼리로 변경 필요 |

**변경 쿼리 (mysql_native_password인 경우):**
```sql
-- 현재 비밀번호를 유지하면서 인증 방식만 변경
ALTER USER 'admin'@'%' IDENTIFIED WITH caching_sha2_password BY 'vraccount#$';
FLUSH PRIVILEGES;

-- 변경 확인
SELECT user, plugin FROM mysql.user WHERE user = 'admin';
-- → caching_sha2_password 로 변경되었는지 확인
```

> 테스트 인스턴스에서 먼저 수행하고, 운영 전환 당일 운영 인스턴스(vraccountdb-84)에 동일하게 적용

## STEP 4. 로컬 연결 테스트

> 앱을 로컬에서 테스트 인스턴스에 연결해 MySQL 8.4 호환성 검증

`application-local.yml`의 datasource url을 테스트 endpoint로 임시 변경:

```yaml
spring:
  datasource:
    url: jdbc:mysql://vraccountdb-84.xxxxxxxxxx.ap-northeast-2.rds.amazonaws.com:3306/vraccountdev?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=Asia/Seoul
```

> `vraccountdev` → 테스트 인스턴스에는 운영 DB의 스냅샷이 복원되어 있으므로  
> 실제로는 `vraccount` DB가 존재함. 테스트 시 url의 DB명을 `vraccount`로 변경

**앱 기동 후 확인 항목:**

| 항목 | 확인 방법 | 기대 결과 |
|------|----------|----------|
| Hibernate dialect 경고 | 앱 기동 로그 확인 | `MySQLDialect` 관련 ERROR/WARN 없음 |
| 로그인 API | Swagger 또는 프론트 | 정상 응답 |
| 주요 조회 API | 정산 조회, 라이더 목록 등 | 정상 응답 |
| GROUP BY 쿼리 | 정산 업로드 후 계산 | 기존 결과와 동일 |
| 날짜/시간 처리 | LocalDate/LocalDateTime 반환 확인 | 시간대 오류 없음 |

### 5-3. ECS Task Definition 환경변수 변경

> API 서버(vroong-backend)는 ECS 컨테이너로 운영 중.  
> DATABASE_URL을 새 인스턴스 엔드포인트로 변경 후 무중단 롤링 배포

```
AWS Console
→ ECS → 태스크 정의(Task Definitions)
→ [vroong-backend 태스크 정의 이름] 클릭
→ 최신 리비전 선택 → 새 리비전 생성(Create new revision)
```

환경변수에서 DATABASE_URL 값 변경:
```
변경 전: jdbc:mysql://vraccountdb.c1acmo8sqyvw.ap-northeast-2.rds.amazonaws.com:3306/vraccount?...
변경 후: jdbc:mysql://vraccountdb-84.xxxxxxxxxx.ap-northeast-2.rds.amazonaws.com:3306/vraccount?...
```

```
→ 저장(Create) → 새 리비전 번호 기록 (예: 리비전 15)
```

### 5-4. ECS 서비스 무중단 배포

```
ECS → 클러스터(Clusters) → [vroong 클러스터]
→ 서비스(Services) → [vroong-backend 서비스] → 업데이트(Update)
→ 태스크 정의 리비전: 방금 생성한 번호 선택
→ 강제 새 배포(Force new deployment): 체크
→ 업데이트 서비스(Update Service)
```

ECS가 구 태스크를 종료하면서 새 태스크를 순차 실행 (롤링 업데이트).  
단일 태스크 구성이면 약 1~2분 다운타임 발생.

**배포 상태 확인:**
```
ECS → 클러스터 → 서비스 → [서비스명]
→ 이벤트(Events) 탭에서 "service ... has reached a steady state" 메시지 확인
→ 태스크(Tasks) 탭에서 실행 중(RUNNING) 상태 확인
```

### 5-5. 배치 서버 환경변수 변경 (EC2)

> 배치 서버는 EC2에서 .env 파일로 환경변수를 관리

```bash
# EC2에 SSH 접속
ssh -i [키페어.pem] ec2-user@[EC2 퍼블릭 IP]

# .env 파일 위치 확인 (보통 아래 중 하나)
cat /etc/environment
# 또는
cat ~/.env
# 또는 앱 실행 디렉토리의 .env

# DATABASE_URL 변경
sudo vi /etc/environment
# 또는
vi ~/.env
```

변경할 내용:
```bash
# 변경 전
DATABASE_URL=jdbc:mysql://vraccountdb.c1acmo8sqyvw.ap-northeast-2.rds.amazonaws.com:3306/vraccount?...

# 변경 후
DATABASE_URL=jdbc:mysql://vraccountdb-84.xxxxxxxxxx.ap-northeast-2.rds.amazonaws.com:3306/vraccount?...
```

배치 프로세스 재시작:
```bash
# systemctl로 관리되는 경우
sudo systemctl restart vroong-batch

# 또는 PID로 직접 재시작 (프로세스 이름 확인 후)
ps aux | grep java
kill [PID]
# 다시 기동 스크립트 실행
```

## 주의사항 및 FAQ

**Q. ECS Task Definition이 어디 있는지 모르겠다면?**
```
ECS → 태스크 정의(Task Definitions) 메뉴
→ 이름에 "vroong" 또는 "backend"가 포함된 항목 찾기
```

**Q. 배치 서버 .env 파일 위치를 모르겠다면?**
```bash
# 실행 중인 Java 프로세스의 환경변수 확인
cat /proc/$(pgrep -f 'vroong.*batch')/environ | tr '\0' '\n' | grep DATABASE
# 또는
sudo find / -name ".env" 2>/dev/null | grep -v proc
```

**Q. 새 인스턴스에 로컬에서 접속이 안 된다면?**
```
RDS → vraccountdb-84 → 연결 및 보안 탭
→ VPC 보안 그룹 확인
→ 해당 보안 그룹 → 인바운드 규칙 → 3306 포트가 허용되어 있는지 확인
→ 내 IP 또는 EC2 IP가 허용 범위에 있는지 확인
```

**Q. 롤백은 얼마나 걸리나?**
- ECS 롤백: 약 2~3분 (이전 Task Definition 리비전으로 업데이트)
- EC2 배치 롤백: SSH → .env 수정 → 재시작 (약 5분)
- DB 자체는 건드리지 않으므로 데이터 손실 없음
