# 2026.07.14 RDS MySQL 8.0 → 8.4 무중단 업그레이드

#aws #rds #mysql #upgrade #infra #production #ecs

## 개요

| 항목 | 내용 |
|------|------|
| **마감** | 2026-07-31 (AWS RDS MySQL 8.0 표준 지원 종료) |
| **대상 인스턴스** | `vraccountdb` (MySQL 8.0.44, db.t4g.micro, 서울 리전) |

| **예상 서비스 중단** | **10~20분** (인스턴스 재시작 발생, 새벽 배치 최소 시간대 진행) |

## 핵심 개념: 왜 In-place 업그레이드인가?

```
[새 인스턴스 방식 - 데이터 누락 문제]
스냅샷 시점 ───────────────────────── 전환 시점
     │                                    │
  vraccountdb (8.0) → 배치가 계속 쓰기 → vraccountdb-84 (8.4) 로 전환
                                               ↑
                               스냅샷 이후 데이터 없음 (누락 발생)

[In-place 업그레이드 (최종 선택)]
vraccountdb (8.0) ──── AWS Modify ────→ vraccountdb (8.4)
     URL 그대로                              URL 그대로
     데이터 그대로                          데이터 그대로
```

**장점**: 데이터 손실 없음 / 엔드포인트 변경 없음 / AWS가 업그레이드 전 자동 스냅샷 생성  
**단점**: 업그레이드 중 10~20분 다운타임 → 새벽 시간대 진행으로 영향 최소화  
**롤백**: AWS 자동 스냅샷에서 복원 (약 15~30분 소요)

## STEP 1. 운영 DB 스냅샷 생성 (오늘 즉시)

> 4월 스냅샷(2026-04-06)은 3개월 이상 지났으므로 최신 스냅샷 재생성 필요

```
AWS Console
→ RDS → 데이터베이스 → vraccountdb 클릭
→ 작업(Actions) → 스냅샷 생성(Take snapshot)
→ 스냅샷 이름: vraccountdb-snapshot-20260714
→ 스냅샷 생성 클릭
```

**확인 방법:**
```
RDS → 스냅샷 → 수동 탭
→ vraccountdb-snapshot-20260714 상태가 '사용 가능' 인지 확인
```

## STEP 2. 테스트 인스턴스 생성 (MySQL 8.4)

> 스냅샷으로 새 인스턴스를 만들면서 MySQL 버전을 8.4로 변경

```
RDS → 스냅샷 → vraccountdb-snapshot-20260714 선택
→ 작업(Actions) → 스냅샷 복원(Restore snapshot)
```

복원 설정:

| 항목 | 값 |
|------|-----|
| DB 인스턴스 식별자 | `vraccountdb-84` |
| DB 엔진 버전 | **MySQL 8.4.x** (드롭다운에서 선택) |
| DB 인스턴스 클래스 | `db.t4g.micro` (운영과 동일) |
| VPC | 운영 VPC와 동일하게 선택 |
| 퍼블릭 액세스 | 아니요 (운영 설정 유지) |
| VPC 보안 그룹 | 운영 vraccountdb와 동일한 보안 그룹 선택 |

> **중요**: VPC 보안 그룹을 운영과 동일하게 설정해야 로컬/앱 서버에서 접근 가능

복원 완료 후 새 엔드포인트 기록:
```
vraccountdb-84.xxxxxxxxxx.ap-northeast-2.rds.amazonaws.com
```

## STEP 3. mysql_native_password 인증 플러그인 확인 및 수정

> MySQL 8.4에서 `mysql_native_password` 플러그인이 기본 비활성화됨  
> 미처리 시 앱에서 **"Access denied"** 오류 발생

**테스트 인스턴스에서 확인 (MySQL Workbench 또는 DBeaver로 접속):**
```sql
SELECT user, plugin FROM mysql.user WHERE user = 'admin';
```

| 결과 | 조치 |
|------|------|
| `caching_sha2_password` | 이미 호환 → 조치 불필요 |
| `mysql_native_password` | 아래 쿼리로 변경 필요 |

**변경 쿼리 (mysql_native_password인 경우):**
```sql
-- 현재 비밀번호를 유지하면서 인증 방식만 변경
ALTER USER 'admin'@'%' IDENTIFIED WITH caching_sha2_password BY 'vraccount#$';
FLUSH PRIVILEGES;

-- 변경 확인
SELECT user, plugin FROM mysql.user WHERE user = 'admin';
-- → caching_sha2_password 로 변경되었는지 확인
```

> 테스트 인스턴스에서 먼저 수행하고, 운영 전환 당일 운영 인스턴스(vraccountdb-84)에 동일하게 적용

## STEP 4. 로컬 연결 테스트

> 앱을 로컬에서 테스트 인스턴스에 연결해 MySQL 8.4 호환성 검증

`application-local.yml`의 datasource url을 테스트 endpoint로 임시 변경:

```yaml
spring:
  datasource:
    url: jdbc:mysql://vraccountdb-84.xxxxxxxxxx.ap-northeast-2.rds.amazonaws.com:3306/vraccountdev?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=Asia/Seoul
```

> `vraccountdev` → 테스트 인스턴스에는 운영 DB의 스냅샷이 복원되어 있으므로  
> 실제로는 `vraccount` DB가 존재함. 테스트 시 url의 DB명을 `vraccount`로 변경

**앱 기동 후 확인 항목:**

| 항목 | 확인 방법 | 기대 결과 |
|------|----------|----------|
| Hibernate dialect 경고 | 앱 기동 로그 확인 | `MySQLDialect` 관련 ERROR/WARN 없음 |
| 로그인 API | Swagger 또는 프론트 | 정상 응답 |
| 주요 조회 API | 정산 조회, 라이더 목록 등 | 정상 응답 |
| GROUP BY 쿼리 | 정산 업로드 후 계산 | 기존 결과와 동일 |
| 날짜/시간 처리 | LocalDate/LocalDateTime 반환 확인 | 시간대 오류 없음 |

## STEP 5. 운영 전환 - In-place 업그레이드 (새벽 시간대)

> ECS/EC2 설정 변경 없음. vraccountdb 인스턴스 자체를 8.4로 업그레이드.  
> URL은 그대로 유지되므로 앱 재배포 불필요.

### 5-1. 전환 전 수동 스냅샷 생성 (AWS 자동 스냅샷과 별개로 보험용)

```
RDS → vraccountdb → 작업(Actions) → 스냅샷 생성(Take snapshot)
→ 스냅샷 이름: vraccountdb-snapshot-before-84-upgrade
→ 생성 클릭 → Available 될 때까지 대기 (약 5분)
```

### 5-2. vraccountdb 인스턴스 In-place 업그레이드

```
RDS → 데이터베이스 → vraccountdb 클릭
→ 수정(Modify) 클릭
```

변경 항목:

| 항목 | 변경 값 |
|------|---------|
| DB 엔진 버전 | `8.4.x` (드롭다운에서 선택) |

```
→ 계속(Continue) 클릭
→ 수정 예약(Schedule modifications): "즉시 적용(Apply immediately)" 선택
→ DB 인스턴스 수정(Modify DB instance) 클릭
```

> AWS가 업그레이드 직전 자동 스냅샷을 생성한 후 업그레이드 진행.  
> 인스턴스 상태가 `Modifying` → `Available`로 바뀔 때까지 **10~20분** 대기.

### 5-3. 업그레이드 완료 확인

```
RDS → vraccountdb → 구성(Configuration) 탭
→ 엔진 버전: 8.4.x 확인
```

DBeaver로 직접 확인:
```sql
SELECT VERSION();
-- → 8.4.x 가 나와야 함
```

### 5-4. 앱 정상 동작 확인

URL이 바뀌지 않으므로 ECS/EC2 설정 변경 없음.  
이미 연결 중인 HikariPool은 DB 재시작으로 끊어지지만 자동으로 재연결됨.

```
ECS → 클러스터 → 서비스 → 태스크(Tasks) 탭
→ 태스크 상태 RUNNING 확인
→ 로그(Logs) 탭에서 DB 연결 에러 없는지 확인
```

ECS 태스크가 DB 재시작 중 ConnectionException으로 재시작될 수 있음.  
→ 정상: ECS가 자동으로 재시작하고 `HikariPool-1 - Start completed` 로그 뜨면 OK.

## STEP 6. 전환 후 확인 (30분 이내)

```sql
-- DB 버전 확인 (DBeaver로 vraccountdb에 접속)
SELECT VERSION();
-- → 8.4.x 확인

-- 최근 데이터 정상 여부 (배치가 정상적으로 써졌는지)
SELECT COUNT(*) FROM vr_rider_acct_weekly
WHERE INS_DATE >= DATE_SUB(NOW(), INTERVAL 1 HOUR);

-- 앱 에러 로그 확인
-- ECS → 클러스터 → 태스크 → 로그 탭
```

**30분 내 심각한 오류 발생 시 롤백:**
```
RDS → 스냅샷 → vraccountdb-snapshot-before-84-upgrade
→ 작업 → 스냅샷 복원 (새 인스턴스로 복원)
→ 복원된 인스턴스 이름: vraccountdb-rollback
→ ECS Task Definition의 DATABASE_URL을 vraccountdb-rollback 엔드포인트로 변경
→ ECS 서비스 강제 재배포
```

> 롤백은 약 15~30분 소요. In-place 롤백은 불가능하므로 스냅샷 복원 후 엔드포인트 전환.

## STEP 7. 테스트 인스턴스 삭제 (전환 후 1주일)

> 테스트용으로 만든 `vraccountdb-84`는 운영 전환에 사용되지 않음. 1주일 모니터링 후 삭제.

```
RDS → vraccountdb-84 → 작업 → 삭제
→ "최종 스냅샷 생성": 체크 해제 (이미 여러 스냅샷 있음)
→ 삭제
```

## 주의사항 및 FAQ

**Q. ECS Task Definition이 어디 있는지 모르겠다면?**
```
ECS → 태스크 정의(Task Definitions) 메뉴
→ 이름에 "vroong" 또는 "backend"가 포함된 항목 찾기
```

**Q. 배치 서버 .env 파일 위치를 모르겠다면?**
```bash
# 실행 중인 Java 프로세스의 환경변수 확인
cat /proc/$(pgrep -f 'vroong.*batch')/environ | tr '\0' '\n' | grep DATABASE
# 또는
sudo find / -name ".env" 2>/dev/null | grep -v proc
```

**Q. 새 인스턴스에 로컬에서 접속이 안 된다면?**
```
RDS → vraccountdb-84 → 연결 및 보안 탭
→ VPC 보안 그룹 확인
→ 해당 보안 그룹 → 인바운드 규칙 → 3306 포트가 허용되어 있는지 확인
→ 내 IP 또는 EC2 IP가 허용 범위에 있는지 확인
```

**Q. 롤백은 얼마나 걸리나?**
- ECS 롤백: 약 2~3분 (이전 Task Definition 리비전으로 업데이트)
- EC2 배치 롤백: SSH → .env 수정 → 재시작 (약 5분)
- DB 자체는 건드리지 않으므로 데이터 손실 없음
