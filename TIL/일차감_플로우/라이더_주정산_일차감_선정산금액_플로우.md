## 1. 개요 및 도메인 개념

### 일차감이란?

라이더와의 계약에 따라 **배달료에서 매일(또는 근무일마다) 차감되는 금액**. 장비 대여료, 교육비, 보험 추가분 등이 해당된다. `vr_deduct_mst` 테이블에 규칙으로 정의되어 있다.

### 주요 enum 값

| 항목 | 값 | 설명 |
|---|---|---|
| `ACCT_TYPE` | `RP` | 라이더 선정산 원장 |
| `ACCT_TYPE` | `RW` | 라이더 주정산 원장 |
| `ACCT_TYPE` | `SW` | 협력사 주정산 원장 |
| `DEDUCT_TARGET` | `R` | 특정 라이더 대상 일차감 |
| `DEDUCT_TARGET` | `SRA` | 협력사 소속 전체 라이더 대상 일차감 |
| `DEDUCT_TARGET` | `S` | 협력사 자체 대상 (라이더 무관) |
| `DEDUCT_TYPE` | `DAILY` | 배달 발생 여부와 무관하게 매일 차감 |
| `DEDUCT_TYPE` | `WORK_DAY` | 배달이 발생한 날(근무일)에만 차감 |
| `CALC_TYPE` | `1` | 정액 (고정 금액) |
| `CALC_TYPE` | `2` | 정률 (배달료 대비 비율) |

## 2. 관련 테이블

| 테이블 | 역할 |
|---|---|
| `vr_deduct_mst` | 일차감 규칙 정의 (금액, 타입, 기간, 대상) |
| `vr_deduct_daily` | 선정산/주정산에서 **실제 적용된** 일차감 내역 (규칙별 1행) |
| `vr_rider_acct_daily` | 선정산 일별 결과 저장 (TOT_ACCT_VAL, DEDUCT_VAL, RECOVER_AMT 등) |

| `vr_supp_acct_weekly` | 협력사 주정산 결과 저장 |
| `vr_account_mst` | 원장 (BAL_AMT, TRAN_AMT, UNPAID_AMT, ACCT_TYPE) |

### vr_rider_acct_daily 주요 컬럼

| 컬럼 | 의미 |
|---|---|
| `TOT_ACCT_VAL` | 선정산 실지급액 (항상 >= 0, 미수금 회수 포함하여 실제 이체된 금액) |
| `DEDUCT_VAL` | 당일 적용된 일차감 합계 (실적값) |
| `RECOVER_AMT` | 이전 미수금 회수액 합계 (배달료 회수 + 잔액 회수) |
| `BALANCE_DEDUCT_AMT` | 잔액에서 차감된 일차감 (배달료가 아닌 잔액에서 처리) |
| `BAL_RECOVER_AMT` | 잔액으로 회수한 이전 미수금 |

### vr_rider_acct_weekly 주요 컬럼 (선정산 라이더 기준)

| 컬럼 | 의미 |
|---|---|
| `PRE_ACCT_VAL` | 집계된 선정산 금액 (= 조정된 선정산 지급액) |
| `DEDUCT_AMT` | 주정산에서 확정된 일차감 총액 (= weeklyDeductTarget) |
| `TOT_ACCT_VAL` | 주정산 실지급액 (대부분 0원) |
| `RECOVER_AMT` | 이월 미수금 회수액 (이번 주 선정산에서 회수한 이전 미수금) |

### vr_deduct_daily 주요 컬럼

| 컬럼 | 의미 |
|---|---|
| `TRAN_DAY` | 적용 날짜 (선정산: 해당 일, 주정산: 정산 종료일) |
| `DEDUCT_AMT` | 실제 적용된 차감액 (배달료 부족 시 0, 미수금 회수 시 소급 업데이트) |
| `UNPAID_AMT` | 미납금 (이번에 못 차감한 금액, 다음 정산으로 이월) |

## 4. 선정산(매일) 일차감 플로우

> 관련 클래스: `BaeminToVroongIfUseCase.java`

### 4-1. Largest Fits First 알고리즘 (v2 도입)

선정산 시 일차감 규칙이 여러 개인 경우 **금액 내림차순**으로 정렬 후 **전부 or 전무(Full-or-Nothing)** 방식으로 적용한다. 배달료가 부족할 때 가장 큰 규칙부터 건너뛰고, 남은 금액으로 처리 가능한 규칙을 적용한다.

```
baseAcctVal = 배달료 - 수수료 - 산재보험 - 고용보험 - 원천징수 - 시간제보험
remaining   = MAX(baseAcctVal, 0)

일차감 규칙 목록을 금액 내림차순 정렬

for each 규칙:
    if remaining >= 규칙.금액:
        DEDUCT_AMT = 규칙.금액   (적용)
        remaining -= 규칙.금액
    else:
        DEDUCT_AMT = 0
        UNPAID_AMT = 규칙.금액   (미납, 이월)

TOT_ACCT_VAL = baseAcctVal - SUM(DEDUCT_AMT)  ← 항상 >= 0
```

> **왜 Largest Fits First인가?**  
> 부분 차감을 허용하면 규칙 하나가 여러 날에 걸쳐 조금씩 차감되어 계산이 복잡해지고 라이더가 혼란스러워한다. 큰 규칙부터 전액 적용 or 전액 보류함으로써 매일 결과가 명확해진다.

### 4-2. 선정산 일차감 적용 4단계 프로세스

```
① 이월 미수금 회수 (오늘 배달료에서 먼저)
   → YES (배달료 >= 미수금): 전액 회수
   → YES (배달료 < 미수금): 가능한 만큼만 회수, 나머지 잔여
   → NO: 생략

② 회수 후 남은 배달료에서 오늘 일차감 부분차감
   → 잔여 배달료 >= 일차감: 전액 차감
   → 잔여 배달료 < 일차감: Largest Fits First로 가능한 것만 차감, 나머지 신규 미수금

= 실지급액 (TOT_ACCT_VAL, 항상 >= 0)

③ 오늘 신규 미수금이 있는 경우:
   Vroong 계좌 잔액 >= 신규 미수금 전액 → 잔액에서 전액 차감 (Full-or-Nothing)
   Vroong 계좌 잔액 < 신규 미수금 전액 → 다음날로 전액 이월 (부분 차감 없음)

④ ③ 처리 후 잔액이 남고, ①에서 미회수 이월 미수금이 있는 경우:
   남은 잔액 범위 내에서 이월 미수금 잔여분 부분 회수 (부분 회수 허용)
```

### 4-3. 선정산 DB 저장 (vr_deduct_daily)

미수금 회수가 발생하면 **원래 미납 발생일의 `vr_deduct_daily` 레코드를 소급 수정**한다 (`updateDeductDailyForRecovery`).

```
가장 최근 UNPAID_AMT > 0 레코드부터 순서대로 (tranDay DESC, seqNo DESC):
  해당 레코드의 DEDUCT_AMT += 회수분
                UNPAID_AMT -= 회수분
```

**이 소급 수정 덕분에** 나중에 주정산에서 `SUM(DEDUCT_AMT)`를 집계할 때 회수분이 자동으로 포함된다.

## 5. 주정산에서 선정산 금액 집계 방식

> 관련 메서드: `sumPreAcctValByPeriodBatch()` in `VrRiderAcctDailyQueryDslRepository.java`

### 집계 공식

```
preAcctVal = SUM(TOT_ACCT_VAL)
           - SUM(BALANCE_DEDUCT_AMT)
           - SUM(BAL_RECOVER_AMT)
```

| 컬럼 | 포함 이유 |
|---|---|
| `SUM(TOT_ACCT_VAL)` | 선정산에서 라이더에게 실제 지급된 금액 합계 |
| `- SUM(BALANCE_DEDUCT_AMT)` | 잔액에서 차감된 일차감은 배달료 지급이 아님 → 제외 |
| `- SUM(BAL_RECOVER_AMT)` | 잔액으로 회수된 미수금도 배달료 지급이 아님 → 제외 |

### 예시

| 날짜 | TOT_ACCT_VAL | BALANCE_DEDUCT_AMT | BAL_RECOVER_AMT |
|---|---|---|---|
| 월 (미수금 7,000 배달료 회수) | 66,000 | 0 | 0 |
| 화 | 73,000 | 0 | 0 |
| 수 | 73,000 | 0 | 0 |
| 목 | 73,000 | 0 | 0 |
| 금 | 73,000 | 0 | 0 |
| **합계** | **358,000** | **0** | **0** |

→ `preAcctVal = 358,000 - 0 - 0 = 358,000원`

> **TOT_ACCT_VAL이 이미 미수금 회수를 반영한 값인 이유**  
> 월요일에 이전 주 미수금 7,000원을 배달료에서 먼저 회수했으므로, 라이더 실수령액은 `80,000 - 7,000(미수금 회수) - 7,000(일차감) = 66,000원`이다.  
> `TOT_ACCT_VAL = 66,000`으로 저장되어 있어 실제 지급액을 정확히 반영한다.

## 6. 주정산에서 일차감 금액 집계 방식

> 관련 메서드: `sumDeductAmtByPeriodBatch()` in `VrDeductDailyQueryDslRepository.java`

### v3 이전 vs v3 이후 비교

| 구분 | v3 이전 | v3 이후 (현재) |
|---|---|---|
| 선정산 라이더 일차감 소스 | `vr_deduct_mst` 이론값 계산 | `vr_deduct_daily.DEDUCT_AMT` 실적값 합산 |
| 주정산 라이더 일차감 소스 | `vr_deduct_mst` 이론값 계산 | 동일 (이론값 유지) |

### v3 현재 — 선정산 라이더 일차감 집계

```java
// RiderSettlementUseCase.java (v3)
if (batchData.preAcctValMap.containsKey(mapKey)) {
    // 선정산 라이더 → vr_deduct_daily 실적값
    totalDeductAmt = batchData.deductDailyAmtMap.getOrDefault(mapKey, BigDecimal.ZERO);
} else {
    // 주정산 라이더 → 이론값 계산
    totalDeductAmt = calculateDeductAmountOptimized(...);
}
```

```sql
-- sumDeductAmtByPeriodBatch 내부 쿼리
SELECT SUPP_ID, RIDER_ID, SUM(DEDUCT_AMT)
FROM vr_deduct_daily
WHERE SUPP_ID IN (...)
  AND RIDER_ID IN (...)
  AND TRAN_DAY BETWEEN acctStDate AND acctEdDate
GROUP BY SUPP_ID, RIDER_ID
```

### 왜 이론값에서 실적값으로 바꿨나?

선정산 라이더는 이미 매일 선정산에서 일차감이 적용된다. 그런데 v3 이전에는 주정산에서도 `vr_deduct_mst` 규칙을 다시 계산했기 때문에, 배달료 부족으로 **실제로 차감에 실패한 날도 이론값 그대로 계산**되었다.  

예를 들어 화요일 배달료 부족으로 장비대여비 30,000원을 못 받았더라도, 주정산 시 이론값 30,000원을 포함해 계산하면 **과도한 차감**이 발생한다.  

v3부터는 `vr_deduct_daily`에 **실제 차감된 값(DEDUCT_AMT)**만 기록되므로, 이를 합산하면 실제 수취한 일차감 합계가 정확하게 나온다.

## 7. 주정산 최종 계산 공식

> 관련 메서드: `buildSingleSettlement()` in `RiderSettlementUseCase.java`

### 선정산 라이더의 경우

```
basePay            = finalDeliveryVal - preAcctVal
                   (주간 총 배달료 - 조정된 선정산 지급액)

weeklyDeductTarget = totalDeductAmt + preRecoverAmt
                   (선정산 실적 일차감 + 이번 주 선정산에서 회수된 이월 미수금)

deductFromPay      = MIN(basePay, weeklyDeductTarget)

totAcctVal         = basePay - deductFromPay   ← 항상 >= 0
```

- `preRecoverAmt` = `SUM(vr_rider_acct_daily.RECOVER_AMT)` — 이번 주 선정산 과정에서 배달료/잔액으로 회수한 이전 미수금 합계
- `weeklyDeductTarget`에 `preRecoverAmt`를 더하는 이유: 이번 주 선정산에서 이전 미수금을 회수했다면, 그 회수액이 **협력사 정산의 일차감**에 정확히 반영되어야 하기 때문이다.

> **preAcctVal과 weeklyDeductTarget의 관계**  
> `TOT_ACCT_VAL`이 미수금 회수분을 이미 제외한 실지급액이라면, `preAcctVal`도 그 값을 SUM하므로 회수분만큼 preAcctVal이 줄어든다.  
> 줄어든 만큼 basePay가 늘어나고, 동시에 weeklyDeductTarget에 preRecoverAmt를 더해 균형을 맞춘다.

## 8. 케이스별 예시 — 선정산 라이더의 주정산

> 공통 기준: 일차감 단가 7,000원/근무일

---

### Case A-1. 모든 선정산에서 정상 차감 → 주정산 실지급 0원

**선정산 내역**

| 요일 | 배달료 | 일차감 | 실지급 | 미수금 |
|---|---|---|---|---|
| 월 | 80,000 | 7,000 | 73,000 | 0 |
| 화 | 80,000 | 7,000 | 73,000 | 0 |
| 수 | 80,000 | 7,000 | 73,000 | 0 |
| 목 | 80,000 | 7,000 | 73,000 | 0 |
| 금 | 80,000 | 7,000 | 73,000 | 0 |
| **합계** | **400,000** | **35,000** | **365,000** | **0** |

**주정산 계산**

```
preAcctVal            = SUM(TOT_ACCT_VAL) = 365,000원
basePay               = 400,000 - 365,000 = 35,000원
totalDeductAmt        = SUM(vr_deduct_daily.DEDUCT_AMT) = 35,000원
preRecoverAmt         = 0원
weeklyDeductTarget    = 35,000 + 0 = 35,000원
deductFromPay         = MIN(35,000, 35,000) = 35,000원
totAcctVal            = 35,000 - 35,000 = 0원
```

→ 선정산에서 모든 일차감이 처리되었으므로 주정산 실지급 **0원**.

---

### Case A-2. 금요일 배달료 부족 → 일부 유보

**선정산 내역**

| 요일 | 배달료 | 일차감 적용 | 실지급 | 미수금 |
|---|---|---|---|---|
| 월~목 | 80,000×4 | 7,000×4 | 73,000×4 | 0 |
| 금 | 2,000 | 2,000 (부분) | 0 | **5,000** |
| **합계** | **322,000** | **30,000** | **292,000** | **5,000** |

> `vr_deduct_daily` 금요일: `DEDUCT_AMT = 2,000, UNPAID_AMT = 5,000`

**주정산 계산**

```
preAcctVal            = 292,000원
basePay               = 322,000 - 292,000 = 30,000원
totalDeductAmt        = SUM(vr_deduct_daily.DEDUCT_AMT) = 30,000원
preRecoverAmt         = 0원
weeklyDeductTarget    = 30,000원
deductFromPay         = 30,000원
totAcctVal            = 0원
```

→ 주정산 실지급 0원. 금요일 유보 **5,000원은 이월 미수금**으로 다음 주 선정산에서 처리.

---

### Case A-3. 배달 없는 날(금요일) → 일차감 전액 유보

**선정산 내역**

| 요일 | 배달료 | 일차감 적용 | 실지급 | 미수금 |
|---|---|---|---|---|
| 월~목 | 80,000×4 | 7,000×4 | 73,000×4 | 0 |
| 금 | 0 | 0 | 0 | **7,000** |
| **합계** | **320,000** | **28,000** | **292,000** | **7,000** |

> 금요일은 배달이 없으므로 `DEDUCT_TYPE=DAILY`이면 7,000원 전액이 미수금으로 이월된다.  
> `vr_deduct_daily` 금요일: `DEDUCT_AMT = 0, UNPAID_AMT = 7,000`

**주정산 계산**

```
preAcctVal            = 292,000원
basePay               = 320,000 - 292,000 = 28,000원
totalDeductAmt        = SUM(vr_deduct_daily.DEDUCT_AMT) = 28,000원
weeklyDeductTarget    = 28,000원
totAcctVal            = 0원
```

→ 미수금 **7,000원**은 다음 주 선정산에서 잔액 Full-or-Nothing 또는 배달료로 회수.

---

### Case A-4. 이번 주 선정산에서 이월 미수금 회수 발생 → 주정산 DEDUCT_AMT 증가

> 지난 주 금요일 유보 미수금 7,000원을 이번 주 월요일 배달료에서 회수한 케이스

**선정산 내역**

| 요일 | 배달료 | 미수금 회수 | 일차감 | 실지급 | 미수금 |
|---|---|---|---|---|---|
| 월 | 80,000 | **7,000 회수** | 7,000 | **66,000** | 0 |
| 화~금 | 80,000×4 | 0 | 7,000×4 | 73,000×4 | 0 |
| **합계** | **400,000** | **7,000** | **35,000** | **358,000** | **0** |

> 월요일 `vr_rider_acct_daily.RECOVER_AMT = 7,000`  
> 지난 주 금요일 `vr_deduct_daily.DEDUCT_AMT`가 0 → 7,000으로 **소급 업데이트** 됨

**주정산 계산**

```
preAcctVal            = SUM(TOT_ACCT_VAL) = 358,000원
basePay               = 400,000 - 358,000 = 42,000원

totalDeductAmt        = SUM(vr_deduct_daily.DEDUCT_AMT) = 35,000원
                        (이번 주 실적 일차감)
preRecoverAmt         = SUM(RECOVER_AMT) = 7,000원
                        (이번 주 미수금 회수액)

weeklyDeductTarget    = 35,000 + 7,000 = 42,000원
deductFromPay         = MIN(42,000, 42,000) = 42,000원
totAcctVal            = 42,000 - 42,000 = 0원
```

→ 주정산 `DEDUCT_AMT = 42,000원`. 이번 주 회수된 미수금 7,000원이 포함되어 **협력사 정산에 정확히 반영**됨.

---

### Case A-5. 이월 미수금이 있으나 주정산 실지급도 0원 → 재이월

> 지난 주 미수금 7,000원이 있고, 이번 주 선정산에서도 회수 못 한 경우 (잔액도 부족)

**주정산 계산**

```
preAcctVal            = 365,000원 (정상 선정산)
basePay               = 400,000 - 365,000 = 35,000원
weeklyDeductTarget    = 35,000 + 0 = 35,000원
totAcctVal            = 0원

이월 미수금           = 7,000원  ← 주정산 실지급이 0원이라 회수 불가
```

→ 선정산 배치에서 잔액 기반 Full-or-Nothing 회수가 성공해야 해소됨.

## 9. 주정산 라이더의 주정산 (비교용)

선정산 라이더와 **동일한 4단계 프로세스**가 적용되지만, 일차감 계산 방식이 다르다.

| 항목 | 선정산 라이더 | 주정산 라이더 |
|---|---|---|
| 일차감 계산 기준 | `vr_deduct_daily` **실적값** | `vr_deduct_mst` 규칙 기반 **이론값** |
| 정산 가능 금액 (`basePay`) | 주간 배달료 - 선정산 지급액 (소액) | 주간 배달료 전액 |
| 주정산 실지급액 | 대부분 **0원** | 배달료 - 일차감 잔여분 |
| 잔액 기반 차감 ③ | 선정산(매일) 단계에서 처리 | 주정산 시 처리 (full-or-nothing) |
| 잔액 기반 회수 ④ | 선정산(매일) 단계에서 처리 | 주정산 시 처리 (부분 회수 허용) |

### 주정산 라이더 계산 예시

> 일차감 10,000원/근무일, 5일 근무

```
basePay               = 250,000 - 0 = 250,000원
totalDeductAmt        = calculateDeductAmountOptimized() = 50,000원  ← 이론값
weeklyDeductTarget    = 50,000원
deductFromPay         = 50,000원
totAcctVal            = 200,000원
```

### 주정산 라이더 부분차감 예시 (배달료 부족)

```
basePay               = 30,000원
totalDeductAmt        = 50,000원
deductFromPay         = MIN(30,000, 50,000) = 30,000원  ← 전액 차감 불가
deferredDeductAmt     = 50,000 - 30,000 = 20,000원    ← 미수금 이월
totAcctVal            = 0원
```

→ `vr_deduct_daily` 삽입: `DEDUCT_AMT=30,000, UNPAID_AMT=20,000`  
→ `vr_account_mst(RW)`: `UNPAID_AMT = -20,000` (다음 주 자동 회수)

## 10. 협력사 주정산 일차감 집계

> 관련 클래스: `PartnerSettlementUseCase.java`  
> 실행 순서: 라이더 주정산 완료 **이후** 실행

### 집계 소스 분류

```
등록 라이더 전체 (registeredRiderIds)
    │
    ├─ vr_rider_acct_weekly 레코드 있음?
    │       YES → vr_rider_acct_weekly.DEDUCT_AMT 합산  ①
    │       NO  → vr_deduct_daily.DEDUCT_AMT 합산       ②
    │
    └─ deductVal = ① + ②
```

**① 주정산 레코드가 있는 라이더** (주정산 엑셀 포함):

```sql
SELECT SUM(DEDUCT_AMT)
FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = ?
  AND ACCT_ST_DATE >= ? AND ACCT_ED_DATE <= ?
  AND RIDER_ID IN (...)
```

**② 주정산 레코드가 없는 선정산 라이더** (주정산 엑셀 미포함):

```sql
SELECT SUM(DEDUCT_AMT)
FROM vr_deduct_daily
WHERE SUPP_ID = ?
  AND TRAN_DAY BETWEEN ? AND ?
  AND RIDER_ID IN (...)
```

→ 선정산 전용 라이더가 주정산 엑셀에 누락되어도 `vr_deduct_daily`에서 직접 읽어 **일차감 누락을 방지**한다.

### 협력사 정산액 공식

```
정산액 = 지급관리비 (VAT 포함) + 일차감(deductVal) - 보험료합계
```

### 협력사 일차감 예시

> 협력사 SUPP001, 선정산 라이더 박철수 + 주정산 라이더 이영희

| 라이더 | 유형 | 소스 | DEDUCT_AMT |
|---|---|---|---|
| 박철수 | 선정산, 주정산 엑셀 포함 | `vr_rider_acct_weekly` | 35,000원 (5일 × 7,000) |
| 이영희 | 주정산, 주정산 엑셀 포함 | `vr_rider_acct_weekly` | 35,000원 (배달료에서 전액 차감) |
| **합계** | | | **70,000원** |

> 박철수가 주정산 엑셀에 없는 경우: `vr_deduct_daily.DEDUCT_AMT`에서 35,000원 읽어 동일하게 반영됨.

### 기존 버전의 문제점 요약

```
[기존 공식]
totAcctVal = finalDeliveryVal - preAcctVal - totalDeductAmt  ← 음수 가능

문제 1: 음수 totAcctVal → vr_account_mst에 음수 TRAN_AMT 기록
문제 2: UNPAID_AMT 저장은 하지만 다음 정산에서 활용하지 않음 (영구 손실)
문제 3: 선정산 라이더도 이론값으로 일차감 계산 → 실제 차감과 불일치
문제 4: 협력사도 이론값 사용 → 라이더가 못 받은 일차감도 협력사에 청구
```

### v3 이후 현재 핵심 원칙

1. **실지급액은 절대 음수가 되지 않는다** — `MIN(basePay, weeklyDeductTarget)`으로 cap
2. **선정산 라이더 일차감 = 실적값** — `vr_deduct_daily.DEDUCT_AMT` 합산
3. **미수금 회수분 소급 수정** — 원래 미납일 레코드 `DEDUCT_AMT` 업데이트
4. **이월 미수금은 다음 정산에서 자동 회수** — `vr_account_mst.UNPAID_AMT` 활용
5. **협력사도 실적값** — `vr_rider_acct_weekly.DEDUCT_AMT` 또는 `vr_deduct_daily.DEDUCT_AMT` 합산
