# 프로모션 테스트 — FIXED (목표 달성 고정 지급)

> 협력사: `DP2503071840` | 정산 기간: `20260617 ~ 20260623`

---

## 계산 공식

```
목표 건수(targetCnt) 이상 배달 시:
  지급액 = rewardAmt (고정 금액, 건수 무관)

목표 미달 시: 0원

※ maxCnt, OVERAGE 개념 없음. 달성/미달만 판단.
```

---

## STEP 0 — 사전 확인: 실제 배달건수 조회

```sql
SELECT
    RIDER_ID,
    SUM(FINISH_CNT) AS total_finish_cnt
FROM vr_delivery_list
WHERE SUPPLY_ID = 'DP2503071840'
  AND DELIVERY_DATE BETWEEN '20260617' AND '20260623'
  AND FINISH_CNT > 0
GROUP BY RIDER_ID
ORDER BY total_finish_cnt DESC;
```

> 달성 기준(targetCnt)을 실제 라이더 건수를 보고 설정하면 테스트하기 좋습니다.
> 예: 라이더 건수 중간값 기준으로 절반은 달성, 절반은 미달이 되도록 설정.

## STEP 3 — 라이더 주정산 엑셀 업로드

1. 20260617~20260623 기간 라이더 주정산 엑셀 업로드

### 검증 쿼리 ①: 라이더별 PROMOTION 컬럼

```sql
SELECT
    RIDER_ID,
    DELIVERY_CNT,
    DELIVERY_VAL,
    PROMOTION,                -- ← 달성 라이더: 10000, 미달 라이더: 0
    TAXABLE_DELIVERY_VAL,     -- ← PROMOTION 포함된 과세배달료
    TOT_ACCT_VAL
FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '20260617'
ORDER BY DELIVERY_CNT DESC;
```

**확인 포인트:**
- `DELIVERY_CNT >= 30` 인 라이더: `PROMOTION = 10000`
- `DELIVERY_CNT < 30` 인 라이더: `PROMOTION = 0`
- `TAXABLE_DELIVERY_VAL`에 PROMOTION이 포함되어 세금 계산에 반영됐는지 확인

### 검증 쿼리 ②: promo_supp_settlement PENDING 상태

```sql
SELECT
    SUPP_ID,
    TOTAL_PROMO_COST,   -- ← 달성 라이더 수 × 10,000원
    HOLD_STATUS         -- ← 'PENDING'
FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840';
```

## STEP 4 — 협력사 주정산 엑셀 업로드

### 협력사 주정산비용 확인

```sql
SELECT SUPPLY_ID, ACCT_ST_DATE, ACCT_VAL
FROM vr_supp_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_ST_DATE = '20260617';
```

### 검증 쿼리 ③: HELD vs PENDING_DEPOSIT

```sql
SELECT
    TOTAL_PROMO_COST,
    PARTNER_ACCT_VAL,
    HOLD_STATUS,
    NOTICE_ID
FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840';
```

**판단 기준:**
- `PARTNER_ACCT_VAL >= TOTAL_PROMO_COST` → `HOLD_STATUS = HELD`
- `PARTNER_ACCT_VAL < TOTAL_PROMO_COST` → `HOLD_STATUS = PENDING_DEPOSIT`, `NOTICE_ID` 존재

**FIXED 미션 특이사항:**
- TOTAL_PROMO_COST = `달성 라이더 수 × 10,000원`
- 달성 라이더가 많을수록 PENDING_DEPOSIT 가능성 증가
- 테스트 시 HELD/PENDING_DEPOSIT 두 경우 모두 확인 권장
  - HELD 확인: targetCnt를 낮게 설정 (거의 모든 라이더 달성)
  - PENDING_DEPOSIT 확인: targetCnt를 낮게 + rewardAmt를 높게 설정

## STEP 5 — 정산 Apply

### 검증 쿼리 ④: vr_account_mst PD 레코드 (HELD인 경우)

```sql
SELECT
    ACCT_TYPE,
    ACCT_DATE,
    SUPPLY_ID,
    TRAN_AMT,   -- ← -(달성 라이더 수 × 10,000원), 음수
    BAL_AMT
FROM vr_account_mst
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_TYPE = 'PD'
ORDER BY SEQ_NO DESC
LIMIT 1;
```

### 검증 쿼리 ⑤: PROCESSED 상태

```sql
SELECT HOLD_STATUS FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840';
-- 'PROCESSED'
```
