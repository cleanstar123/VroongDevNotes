# 프로모션 테스트 — 엣지케이스

> 협력사: `DP2503071840` | 기준 정산 기간: `20260624 ~ 20260630`
>
> 기존 OVERAGE/FIXED/FULL_UNIT 테스트에서 다루지 않은 경계 시나리오를 검증합니다.

---

## 코드 동작 요약 (이해 전제)

| 함수 | 동작 |
|------|------|
| `findActiveMissionsOverlapping(suppId, from, to)` | `STATUS='ACTIVE' AND startDate <= to AND endDate >= from` 인 미션 전체 반환 |
| `computeRiderPromotion(missions, ...)` | 반환된 미션 목록을 **루프 돌며 누적 합산** → PROMOTION = Σ(미션별 지급액) |
| `buildPeriods(mission)` | 미션 startDate~endDate를 periodType 단위로 분할. WEEKLY면 7일씩 자름 (마지막 period는 endDate에서 잘림) |
| 기간 필터 | `period[1](period 끝)이 acctStDate~acctEdDate 내` 에 있는 주기만 집계 |

## EDGE CASE 1 — 복수 미션 동시 활성화 (OVERAGE + FIXED 겹침)

> 같은 협력사, 같은 정산 주에 2개의 미션이 동시에 ACTIVE인 경우
> `computeRiderPromotion`이 두 미션을 합산하는지 확인합니다.

### 기대 동작

```
PROMOTION = 미션A 지급액 + 미션B 지급액
vr_promo_supp_settlement.TOTAL_PROMO_COST = Σ(모든 라이더 PROMOTION)
```

```sql
-- 미션 B: FIXED (TARGET_CNT=30, REWARD_AMT=5000)
```

### STEP 2 — 기대값 계산

> 라이더 rpx3060 기준 (실측 배달건수 302건, 목표 30건 달성)
>
> - 미션A(OVERAGE): min(302-30, 20) × 500 = 20건 × 500 = **10,000원**
> - 미션B(FIXED): 30건 달성 → **5,000원**
> - 합계 PROMOTION = **15,000원**

| 조건 | 미션A (OVERAGE) | 미션B (FIXED) | 합계 PROMOTION |
|------|:--------------:|:-------------:|:--------------:|
| 30건 이상 (달성) | min(건수-30,20)×500 | 5,000원 | 두 미션 합산 |
| 30건 미만 (미달) | 0원 | 0원 | 0원 |
| chansu0519 (30건) | 0원 (초과 없음) | 5,000원 | **5,000원** ← FIXED는 달성만 확인 |

> ⚠️ **포인트**: chansu0519처럼 target과 정확히 같은 건수인 라이더는
> OVERAGE=0원이지만 FIXED=5,000원으로 두 미션의 계산 방식 차이가 드러납니다.

### STEP 3 — 라이더 주정산 업로드 후 검증

```sql
-- 복수 미션 합산 확인 (달성 라이더의 PROMOTION이 두 미션 합산값인지)
SELECT
    RIDER_ID,
    DELIVERY_CNT,
    PROMOTION,
    -- OVERAGE 기대값: min(max(DELIVERY_CNT-30,0), 20)*500
    -- FIXED 기대값: CASE WHEN DELIVERY_CNT>=30 THEN 5000 ELSE 0 END
    -- 합계 기대값:
    (CASE WHEN DELIVERY_CNT >= 30
          THEN LEAST(DELIVERY_CNT - 30, 20) * 500 + 5000
          ELSE 0 END) AS expected_promotion
FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '20260624'
ORDER BY DELIVERY_CNT DESC;

-- PROMOTION = expected_promotion 인지 모든 라이더 확인
SELECT COUNT(*) AS mismatch_cnt
FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '20260624'
  AND PROMOTION != (
    CASE WHEN DELIVERY_CNT >= 30
         THEN LEAST(DELIVERY_CNT - 30, 20) * 500 + 5000
         ELSE 0 END
```

## EDGE CASE 3 — 업로드 순서 (미션 등록 전 업로드)

> 라이더 주정산을 **먼저 업로드한 다음** 미션을 등록한 경우.
>
> **코드 동작**: `findActiveMissionsOverlapping`은 업로드 시점에 호출됨.
> 업로드 당시 미션이 없으면 PROMOTION=0 으로 저장되고, 이후 미션을 등록해도 재계산되지 않음.
> → 미션 등록 순서가 정산 결과에 영향을 미칩니다.

### 시나리오 A — 미션 등록 전 업로드 (의도적 누락 확인)

```
1. 미션 없이 라이더 주정산 업로드
2. 업로드 후 미션 등록
3. vr_rider_acct_weekly.PROMOTION 확인 → 모두 0원 (미션 소급 적용 안 됨)
4. vr_promo_supp_settlement 확인 → 레코드 없음 (미션 없었으므로)
```

```sql
-- 미션 없는 상태에서 업로드 후 확인
SELECT COUNT(*) AS promo_settlement_cnt
FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-24';
-- 0 이어야 함 (미션이 없었으므로)

SELECT COUNT(*) AS non_zero_promo
FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '20260624'
  AND PROMOTION > 0;
-- 0 이어야 함
```

### 시나리오 B — 재업로드로 미션 반영 (비교용)

```
1. 업로드된 데이터 전체 삭제
2. 미션 등록
3. 라이더 주정산 재업로드
4. PROMOTION 확인 → 미션 기반 계산값 존재
```

> **결론**: 미션은 라이더 주정산 업로드 전에 반드시 등록되어야 합니다.
> 등록 후 업로드하면 PROMOTION 적용, 등록 전 업로드하면 PROMOTION=0 (소급 불가).

```sql
-- 재업로드 전 데이터 정리
DELETE FROM vr_rider_acct_weekly WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_ST_DATE = '20260624';
DELETE FROM vr_supp_acct_weekly WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_ST_DATE = '20260624';
-- 미션 등록 후 재업로드 → PROMOTION 값이 채워지는지 확인
```

## EDGE CASE 4 — 미션 INACTIVE 처리 시점별 영향

> 라이더 업로드 후, 협력사 업로드 전에 미션을 INACTIVE로 변경한 경우.

### 코드 동작

- `findActiveMissionsOverlapping`은 **라이더 업로드 시점**에 호출
- INACTIVE 처리는 이미 저장된 `vr_rider_acct_weekly.PROMOTION`과 `vr_promo_supp_settlement.TOTAL_PROMO_COST`에 영향 없음
- 협력사 업로드 시 `vr_promo_supp_settlement`의 기존 TOTAL_PROMO_COST와 PARTNER_ACCT_VAL 비교 → 정상 처리됨

### STEP 1 — 미션 등록 → 라이더 업로드 → 미션 INACTIVE → 협력사 업로드

```sql
-- 미션 등록

  (SUPP_ID, TITLE, MISSION_TYPE, PERIOD_TYPE, TARGET_CNT, REWARD_AMT, MAX_CNT,
   START_DATE, END_DATE, DESCRIPTION, TARGET_TYPE, STATUS_CODE, INS_DATE, INS_USER)
VALUES
  ('DP2503071840', '[TEST] 엣지-INACTIVE테스트', 'FIXED', 'WEEKLY', 30, 5000, 0,
   '2026-06-24', '2026-06-30', 'INACTIVE 타이밍 테스트', 'ALL', 'ACTIVE', NOW(), 'TEST');
```

```
1. 라이더 주정산 업로드 (PROMOTION 계산됨)
2. 미션을 INACTIVE로 변경
```



> **기대 결과**: 라이더 업로드 시 계산된 PROMOTION과 TOTAL_PROMO_COST는
> 이후 미션 INACTIVE 처리와 무관하게 유지됩니다.
>
> ⚠️ **운영 주의**: 미션을 취소해야 한다면 라이더 업로드 전에 INACTIVE 처리해야 합니다.

### STEP 1 — 단일 날짜 미션 등록

```sql
  (SUPP_ID, TITLE, MISSION_TYPE, PERIOD_TYPE, TARGET_CNT, REWARD_AMT, MAX_CNT,
```
