# 프로모션 강제진행 / 주정산만 진행 테스트

> 협력사: `DP2503071840`
> 테스트 기간: `2026-06-17 ~ 2026-06-23` (이미 PROCESSED → 직접 INSERT로 상태 재설정)

---

## 테스트 목록

| # | 시나리오 | 초기 상태 | 실행 | 기대 결과 |
|---|----------|-----------|------|-----------|
| A | 강제진행 (PENDING_DEPOSIT) | PENDING_DEPOSIT | POST /force | SW + PD 생성, FORCED |
| B | 주정산만 진행 후 강제진행 (SW_ONLY) | SW_ONLY | POST /force | PD만 생성, FORCED |
| C | 주정산만 진행 (PENDING_DEPOSIT → SW_ONLY) | PENDING_DEPOSIT | POST /sw-only | SW만 생성, SW_ONLY |
| D | Apply 시 라이더 프로모션 차단 확인 | PENDING_DEPOSIT | Apply 버튼 | 해당 협력사 라이더 vr_account_mst 미생성 |

## 공통 사전 준비

### 현재 상태 확인

```sql
SELECT HOLD_STATUS FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';
```

→ `PROCESSED`면 아래 SQL로 상태 초기화 후 테스트

### 상태 초기화 (PROCESSED → PENDING_DEPOSIT)

```sql
UPDATE vr_promo_supp_settlement
SET HOLD_STATUS = 'PENDING_DEPOSIT',
    UPD_DATE = NOW(),
    UPD_USER = 'TEST'
WHERE SUPP_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '2026-06-17'
  AND ACCT_ED_DATE = '2026-06-23';
```

### 최신 잔액 확인

```sql
SELECT SEQ_NO, ACCT_TYPE, ACCT_DATE, BAL_AMT, TRAN_AMT, UNPAID_AMT
FROM vr_account_mst
WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_TARGET = 'S'
ORDER BY SEQ_NO DESC LIMIT 5;
```

## 테스트 A — 강제진행 (PENDING_DEPOSIT → FORCED)

### 초기 상태 설정

```sql
-- 상태가 PENDING_DEPOSIT인지 확인
SELECT HOLD_STATUS, TOTAL_PROMO_COST, PARTNER_ACCT_VAL
FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';
-- HOLD_STATUS = PENDING_DEPOSIT 이어야 함
```

### 실행 — API 호출

```
curl.exe -X POST "http://localhost:8080/api/v1/promotion/supp-settlement/DP2503071840/force?acctStDate=2026-06-17&acctEdDate=2026-06-23&userId=test123" -H "X-API-Secret: [시크릿키]"
```

또는 **웹 화면** → 본사 계정 로그인 → 협력사 → 라이더 프로모션 → 기간 조회 → 강제진행 버튼

### 결과 확인

```sql
-- 1. SW, PD 레코드 생성 확인
SELECT ACCT_TYPE, ACCT_DATE, TRAN_AMT, BAL_AMT, UNPAID_AMT
FROM vr_account_mst
WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_TARGET = 'S'
ORDER BY SEQ_NO DESC LIMIT 4;
```

**기대값:**

| ACCT_TYPE | TRAN_AMT | 설명 |
|-----------|----------|------|
| PD | -3,100,000 | 프로모션 차감 |
| SW | +1,801,793 | 주정산 수령 |

> BAL_AMT와 UNPAID_AMT는 이전 잔액에 따라 달라짐.
> 잔액이 부족하면 `UNPAID_AMT < 0` 발생 (마이너스 허용).

```sql
-- 2. holdStatus FORCED 전환 확인
SELECT HOLD_STATUS FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';
-- 기대: FORCED
```

### 테스트 후 정리

```sql
DELETE FROM vr_account_mst
WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_TARGET = 'S'
  AND ACCT_TYPE IN ('SW', 'PD') AND INS_USER = 'test123';

UPDATE vr_promo_supp_settlement
SET HOLD_STATUS = 'PENDING_DEPOSIT', UPD_DATE = NOW(), UPD_USER = 'TEST'
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';
```

## 테스트 B — 강제진행 (SW_ONLY → FORCED, PD만 생성)

### 초기 상태 설정

```sql
-- SW_ONLY 상태로 변경 + SW 레코드가 이미 존재하는 상황 시뮬레이션

SET HOLD_STATUS = 'SW_ONLY', UPD_DATE = NOW(), UPD_USER = 'TEST'
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';

-- SW 레코드가 이미 있어야 하므로 직접 삽입
-- (이전 잔액 기준으로 BAL_AMT 계산, 여기서는 이전 잔액을 0으로 가정)
INSERT INTO vr_account_mst
  (ACCT_TYPE, ACCT_DATE, ACCT_TARGET, SUPPLY_ID, RIDER_ID,
   BAL_AMT, TRAN_AMT, UNPAID_AMT, INS_USER, INS_DATE)
VALUES
  ('SW', '20260623', 'S', 'DP2503071840', '',
   1801793, 1801793, 0, 'TEST', NOW());
```

```sql
-- 현재 최신 잔액 확인 (SW INSERT 후)

WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_TARGET = 'S'
ORDER BY SEQ_NO DESC LIMIT 1;
-- 기대: 1,801,793
```

또는 **웹 화면** → 기간 조회 → 주정산만 진행 버튼

## 테스트 D — Apply 시 라이더 프로모션 차단 확인

> PENDING_DEPOSIT / SW_ONLY 협력사 소속 라이더는 Apply 버튼 클릭 시 vr_account_mst에 RW/RR 레코드가 **생성되지 않아야** 합니다.

### 초기 상태 확인

```sql
-- 1. PENDING_DEPOSIT 상태 확인
SELECT HOLD_STATUS FROM vr_promo_supp_settlement
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';
-- 기대: PENDING_DEPOSIT

-- 2. 해당 협력사 소속 라이더 목록 확인
SELECT RIDER_ID FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '2026-06-17'
LIMIT 5;
```

### 실행

웹 화면에서 해당 정산 기간 Apply 버튼 클릭.

### 결과 확인

```sql
-- 라이더 계정에 RW/RR 레코드가 생성되지 않았는지 확인
SELECT COUNT(*) AS cnt
FROM vr_account_mst
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_TARGET = 'R'
  AND ACCT_TYPE IN ('RW', 'RR')
  AND ACCT_DATE = '20260623';
-- 기대: 0 (라이더 지급 차단됨)
```

SW_ONLY 상태에서도 동일하게 확인:

```sql
UPDATE vr_promo_supp_settlement
SET HOLD_STATUS = 'SW_ONLY', UPD_DATE = NOW(), UPD_USER = 'TEST'
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';
-- Apply 재실행 후 위 COUNT 쿼리 반복
```

## 상태 전체 초기화 (모든 테스트 완료 후)

```sql
-- promo_supp_settlement 원복 (PROCESSED)
UPDATE vr_promo_supp_settlement
SET HOLD_STATUS = 'PROCESSED', UPD_DATE = NOW(), UPD_USER = 'TEST'
WHERE SUPP_ID = 'DP2503071840' AND ACCT_ST_DATE = '2026-06-17';

-- 테스트 중 삽입된 vr_account_mst 레코드 삭제
DELETE FROM vr_account_mst
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_TARGET = 'S'
  AND ACCT_TYPE IN ('SW', 'PD')
  AND INS_USER IN ('TEST', 'test123', 'SYSTEM');
```

## 상태 흐름 요약

```
PENDING_DEPOSIT
  ├─ 강제진행 (forceProcess) → SW + PD 생성 → FORCED
  └─ 주정산만 진행 (processSwOnly) → SW만 생성 → SW_ONLY
       └─ 강제진행 (forceProcess) → PD만 생성 → FORCED

PENDING_DEPOSIT / SW_ONLY
  └─ Apply 시 해당 협력사 라이더 vr_account_mst 미생성 (차단)
```
