# 프로모션 테스트 — PENDING_DEPOSIT & 공지사항

> 협력사: `DP2503071840` | 정산 기간: `20260617 ~ 20260623`

---

## 발생 조건

```
협력사 주정산비용(PARTNER_ACCT_VAL) < 미션 총비용(TOTAL_PROMO_COST)
→ HOLD_STATUS = PENDING_DEPOSIT
→ vr_notice + vr_notice_target 레코드 자동 생성
→ vr_promo_supp_settlement.NOTICE_ID 채워짐
```

---

## STEP 0 — 기존 PARTNER_ACCT_VAL 확인

PENDING_DEPOSIT을 만들려면 TOTAL_PROMO_COST > PARTNER_ACCT_VAL 이어야 합니다.

```sql
SELECT SUPPLY_ID, ACCT_ST_DATE, ACCT_VAL
FROM vr_supp_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840' AND ACCT_ST_DATE = '20260617';
```

> 이 테스트에서는 ACCT_VAL ≈ 1,801,793 기준으로 이를 초과하는 TOTAL_PROMO_COST를 설정합니다.

## STEP 3.5 — PENDING_DEPOSIT 결정 시 자동 실행된 동작 확인 ⭐ 신규

> 협력사 업로드 완료 시 PENDING_DEPOSIT으로 결정되면 `savePromoAndClearForPendingDeposit`이 자동 호출됩니다.
> 라이더가 받지 못할 프로모션 금액을 정산내역에서 미리 제거합니다.

### 검증 쿼리 ③-A: 라이더 PROMOTION null 클리어 확인

```sql
SELECT
    RIDER_ID,
    PROMOTION,           -- ← 달성 라이더: NULL, 미달 라이더: 0
    TAXABLE_DELIVERY_VAL -- ← PROMOTION=0 기준으로 재계산된 값
FROM vr_rider_acct_weekly
WHERE SUPPLY_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '20260617'
ORDER BY RIDER_ID
LIMIT 10;
```

**기대 동작 (실측 확인됨):**
- **달성 라이더** (원래 PROMOTION > 0): `NULL` → vr_promo_rider_result에 저장, TAXABLE_DELIVERY_VAL 재계산
- **미달 라이더** (원래 PROMOTION = 0): `0` 그대로 유지 → vr_promo_rider_result에 저장 안 됨
- 이 구분이 강제진행 시 복원 대상을 결정함 (null인 라이더만 복원)

### 검증 쿼리 ③-B: vr_promo_rider_result 저장 확인

```sql
SELECT
    RIDER_ID,
    PROMOTION_AMT,       -- ← STEP 2에서 계산한 각 라이더 지급액
    ACCT_ST_DATE,
    ACCT_ED_DATE
FROM vr_promo_rider_result
WHERE SUPP_ID = 'DP2503071840'
  AND ACCT_ST_DATE = '2026-06-17'
ORDER BY PROMOTION_AMT DESC;
-- 프로모션이 0이 아닌 라이더들의 금액이 저장되어 있어야 함
-- 총 SUM(PROMOTION_AMT) = 3,100,000원 이어야 함
```
