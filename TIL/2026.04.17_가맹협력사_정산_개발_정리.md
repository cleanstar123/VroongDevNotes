# 가맹 협력사 주정산 & 선정산 개발 정리

> 작성일: 2026-04-15
> 프로젝트: vroong-backend  
> 브랜치: dev

---

## 1. 개요

가맹 협력사(`suppType='F'`)의 **주정산(weekly settlement)** 과 **선정산(pre-settlement)** 정산 내역을 조회·관리하는 기능을 개발하였다.

| 구분              | 설명                                                                   | 기준 테이블                                       |
| --------------- | -------------------------------------------------------------------- | --------------------------------------------- |
| **주정산 (갑지 정산)** | 협력사 주 단위 정산. 엑셀 업로드로 데이터 등록 후 조회                                    | `vr_supp_acct_weekly`                         |
| **선정산**         | 가맹 협력사가 주정산 기간 중 매일 미리 지급받은 금액. 일별 집계 후 기간 합산하여 참고값으로 표시             | `vr_supp_acct_daily`                          |
| **가맹 정산**       | 가맹 협력사(`suppType='F'`)만 필터링하여 별도 공식으로 계산. 선정산금액 합계를 함께 조회하여 화면에 표시 | `vr_supp_acct_weekly` + `vr_supp_acct_daily`  |

    │

[Repository Layer]
    PartnerSettlementRepository           (interface)

## 4. 가맹 협력사 주정산 (FranchiseSettlement)

### 4-1. API

```
GET /api/v1/franchise-settlement
    파라미터: startDate, endDate (정산시작일 범위), supplyId (선택)
    반환: FranchiseSettlementDto 목록

GET /api/v1/franchise-settlement/latest-date
    반환: latestStartDate, latestEndDate

GET /api/v1/franchise-settlement/suppliers
    반환: 가맹 협력사 목록 [{id, name}]
```

### 4-2. 처리 흐름

```
1. 건당 공제 금액 조회
   └─ vr_comm_code WHERE GRP_CODE='FRANC' AND COM_CODE='FEE_P'
      기본값: 30원

2. 협력사 주정산 데이터 조회
   └─ vr_supp_acct_weekly WHERE ACCT_ST_DATE BETWEEN startDate AND endDate
      (supplyId 있으면 추가 필터)

3. 가맹 협력사만 필터링
   └─ vr_supplier_info WHERE SUPP_ID IN (조회된 협력사 목록) AND SUPP_TYPE = 'F'

4. 협력사명 일괄 조회 (N+1 방지)
   └─ vr_supplier_info WHERE SUPP_ID IN (가맹 협력사 ID 목록)

5. 선정산금액 일괄 조회
   └─ vr_supp_acct_daily WHERE SUPPLY_ID IN (...) AND DELIVERY_DATE BETWEEN fromDate AND toDate
      → SUM(PRE_SETTLEMENT_AMT) GROUP BY SUPPLY_ID

6. 가맹 정산 계산 (협력사별 루프)
   └─ 아래 계산식 적용

7. FranchiseSettlementDto 빌드 후 반환
```

### 4-3. 가맹 정산 계산식

가맹 협력사 주정산 계산은 **엑셀 업로드 시점** (`PartnerSettlementUseCase`)과 **조회 화면** (`FranchiseSettlementUseCase`) 두 곳에서 각각 다르게 처리된다.

#### 엑셀 업로드 시 실계산 (PartnerSettlementUseCase) — `vr_supp_acct_weekly.ACCT_VAL` 저장

```
[부릉수수료]
  부릉수수료 = 처리건수(PROC_CNT) × 건당공제금액(기본 30원)

[공급가액]
  공급가액 = 배달료(DELIVERY_VAL) + 후정산(ADD_ACCT_VAL) + 관리비(MAINTNC_VAL)

[적격증빙]
  적격증빙 = (공급가액 - 부릉수수료) × 1.1  (VAT 포함, 소수점 이하 내림)

[선정산지급액 합계]
  선정산합 = vr_account_mst WHERE ACCT_TYPE='SP' AND 정산기간 내 SUM(TRAN_AMT)
  (배치가 매일 입금 반영한 실제 지급액)

[최종 지급금액 — ACCT_VAL]
  지급금액 = 적격증빙
           - 선정산지급액 합계
           - 고용보험료(사업주)
           - 산재보험료(사업주)
           - 시간제보험료
```

#### 조회 화면 표시용 재계산 (FranchiseSettlementUseCase)

```
[공급가액]
  공급가액 = 배달료 + 후정산 + 관리비 - 부릉수수료

[부가세액]
  부가세액 = 공급가액 × 10%  (소수점 이하 내림)

[공급대가]
  공급대가 = 공급가액 + 부가세액

[보험료합계]
  보험료합계 = 고용보험(사업주) + 고용보험(라이더) + 산재보험(사업주) + 산재보험(라이더) + 시간제보험료

[화면 표시 지급금액 — paymentAmount]
  paymentAmount = 공급대가 - 보험료합계
  ※ 선정산 차감 없이 표시 (선정산금액은 별도 컬럼으로 나란히 표시)
```

> 실제 정산 확정금액은 엑셀 업로드 시 계산된 `ACCT_VAL`이다.  
> `FranchiseSettlementUseCase`의 `paymentAmount`는 선정산 차감 전 총액을 보여주고,  
> `preSettlementAmt`(선정산 합계)를 함께 표시하여 운영자가 두 값을 비교할 수 있도록 한다.

## 5. 협력사 주정산 (PartnerSettlement)

가맹을 포함한 **전체 협력사**의 주정산 내역 관리. 엑셀 업로드로 데이터를 등록하고, 원장(`vr_account_mst`) 적용 기능 포함.

### 5-1. API

```
GET  /api/v1/partner-settlement
     파라미터: startDate, endDate (정산시작일 범위) 또는 acctEdDate (하위호환)
              supplyId (선택)

GET  /api/v1/partner-settlement/latest-date
     반환: latestStartDate, latestEndDate

POST /api/v1/partner-settlement/upload   (multipart/form-data)
     파라미터: file (Excel), userId
     동작: 엑셀 파싱 → 기존 데이터 히스토리 백업 → 삭제 후 재저장

GET  /api/v1/partner-settlement/excel
     파라미터: acctStDate, acctEdDate, supplyId
     반환: .xlsx 파일 다운로드
```

### 5-2. 엑셀 업로드 흐름

```
1. 파일 유효성 검증 (확장자, 빈 파일 등)
2. 엑셀 파싱 → List<PartnerSettlement>
3. 파싱 데이터 유효성 검증
4. 동일 정산기간 기존 데이터 존재 여부 확인
   ├─ 존재하면: 히스토리 백업 (vr_supp_acct_weekly_his, reason='엑셀 재업로드') → 기존 삭제
   └─ 없으면: 최초 업로드
5. 새 데이터 저장 (vr_supp_acct_weekly)
※ 원장(vr_account_mst) 적용은 '적용' 버튼 클릭 시 별도 호출 (applyToAccountMst)
```

### 5-3. 협력사 주정산 계산식

```
[지급관리비 (VAT 포함)]
  지급관리비 = (관리비 + 지점운영수수료 + 달성페이백 + 프로모션) × 1.1

[실 지급 금액]
  실 지급 금액 = 지급관리비(VAT 포함) + 일차감 - 보험료합계
```

### 5-4. PartnerSettlement DTO 주요 필드

```
협력사명     → vr_supplier_info.SUPP_NAME (항상 DB 조회)
사업자번호   → vr_supplier_info.SUPP_NUM (항상 DB 조회)
대표자명     → vr_supplier_info.COMP_CEO_NAME (항상 DB 조회)
```
> `vr_supp_acct_weekly` 에 저장된 `BIZ_REG_NO`, `BIZ_NAME` 대신
> 항상 `vr_supplier_info` 에서 최신값을 조회하도록 구현.

### 5-5. 히스토리 테이블 활용 (`vr_supp_acct_weekly_his`)

- 엑셀 재업로드 시 기존 데이터를 자동 백업
- 원장 적용 시도 전에도 백업 (`reason='주정산 적용'`)
- `HIS_SEQ_NO`, `ARCHIVED_AT`, `RECALC_REASON` 컬럼 추가

## 6. 가맹 협력사 선정산 (FranchisePreSettlement)

가맹 협력사는 주정산 기간(1주일) 동안 **매일 배달 건수에 따라 선지급**을 받는다.  
이 일별 선지급 금액은 `vr_supp_acct_daily` 테이블에 날짜별로 집계되어 저장되며,  
주정산 화면에서 해당 기간의 합계를 **참고값(선정산금액)** 으로 함께 표시한다.

> **선정산금액은 이미 실제로 지급된 금액이다.**  
> 배치가 매일 선정산금액을 계산해 `vr_account_mst`에 입금 반영하므로, 협력사 잔액은 매일 증가한다.  
> 주정산 화면의 `paymentAmount`(지급금액)는 "이번 주 지급해야 할 총액"이고, `preSettlementAmt`는 "그 중 이미 선지급된 금액"이다.  
> 시스템은 `paymentAmount - preSettlementAmt`(실 추가 지급 필요액)를 자동 계산하지 않으므로, 운영자가 두 값을 보고 직접 판단한다.

### 6-1. 선정산 계산 공식

**위치**: `BaeminToVroongIfUseCase.java` → `processFranchisePreSettlement()` (일별 배치)

```
선정산금액 = 소속 라이더의 일별 배달처리비 합계 × 0.95 × 1.1  (소수점 이하 내림)
```

| 구성 요소 | 설명 |
|-----------|------|
| `배달처리비 합계 (TOT_DELIVERY_AMT)` | 해당 가맹 협력사 소속 라이더 전원의 당일 배달료 합산 |
| `× 0.95` | 배달처리비의 95%만 선지급 (5%는 주정산 시 정산) |
| `× 1.1` | 부가세(VAT) 10% 포함 |

```java
// BaeminToVroongIfUseCase.java
BigDecimal preSettlementAmt =
    totDeliveryAmt
        .multiply(new BigDecimal("0.95"))
        .multiply(VAT_RATE)           // 1.1
        .setScale(0, RoundingMode.DOWN);
```

### 6-2. 선정산 처리 흐름 (일별 배치)

```
[매일 배치 실행 (전일 기준)]
        ↓
1. 가맹 협력사 소속 라이더의 전일 배달처리비 합계 조회
   └─ vrRiderAcctDailyQueryDslRepository.findFranchisePreSettlementData(deliveryDate)

2. vr_user_info 등록 여부 확인 (USE_YN='Y' 협력사만 처리)

3. 선정산금액 계산
   └─ preSettlementAmt = totDeliveryAmt × 0.95 × 1.1 (소수점 내림)

4. vr_supp_acct_daily 저장 (upsert)
   └─ DELIVERY_DATE, SUPPLY_ID, TOT_DELIVERY_CNT, TOT_DELIVERY_AMT, PRE_SETTLEMENT_AMT

5. vr_account_mst 잔액 반영
   └─ 협력사 잔액 원장에 선정산금액 입금 처리

        ↓ (주정산 화면 조회 시)

[VrSuppAcctDailyQueryDslRepository.sumPreSettlementAmountBySupplyIds()]
    → SUM(PRE_SETTLEMENT_AMT) GROUP BY SUPPLY_ID (정산기간 내)

[FranchiseSettlementDto.preSettlementAmt 에 포함하여 참고 표시]
```

### 6-3. 선정산금액 조회 로직 (FranchiseSettlementUseCase.java)

```java
// 정산기간을 YYYYMMDD 형식으로 변환
String fromDate = acctStDate.format(DateTimeFormatter.ofPattern("yyyyMMdd"));
String toDate   = acctEdDate.format(DateTimeFormatter.ofPattern("yyyyMMdd"));

// 가맹 협력사 ID 목록에 대해 기간 내 선정산금액 합계 일괄 조회
Map<String, BigDecimal> preSettlementAmtBySupplyId =
    vrSuppAcctDailyQueryDslRepository.sumPreSettlementAmountBySupplyIds(
        supplyIds, fromDate, toDate);

// 협력사별 선정산금액 적용
BigDecimal preSettlementAmt =
    preSettlementAmtBySupplyId.getOrDefault(currentSupplyId, BigDecimal.ZERO);
```

## 7. 주요 구현 포인트

### 7-1. 가맹 협력사 필터링

`vr_supplier_info.SUPP_TYPE = 'F'` 로 가맹 협력사를 식별.

```java
// VrSupplierInfoQueryDslRepository.java
public Set<String> findFranchiseSupplyIds(List<String> supplyIds) {
    return queryFactory
        .select(vsi.suppId)
        .from(vsi)
        .where(vsi.suppId.in(supplyIds).and(vsi.suppType.eq("F")))
        .fetch();
}
```

### 7-2. N+1 문제 방지 (배치 조회)

협력사명과 선정산금액을 협력사별 개별 조회하지 않고 ID 목록으로 한 번에 조회.

```java
// 협력사명 일괄 조회
Map<String, String> suppNameBySuppId =
    vrSupplierInfoQueryDslRepository.findSuppNamesBySuppIds(supplyIds);

// 선정산금액 일괄 조회 (SUM GROUP BY SUPPLY_ID)
Map<String, BigDecimal> preSettlementAmtBySupplyId =
    vrSuppAcctDailyQueryDslRepository.sumPreSettlementAmountBySupplyIds(
        supplyIds, fromDate, toDate);
```

### 7-3. 비동기 처리 (CompletableFuture)

`getFranchiseSettlementList()` 전체를 `CompletableFuture.supplyAsync()` 로 감싸 비동기 실행.  
컨트롤러 반환 타입도 `CompletableFuture<ResponseEntity<...>>` 로 통일.

```java
return CompletableFuture.supplyAsync(() -> {
    // 1. 건당 공제 금액 조회
    // 2. 주정산 데이터 조회
    // 3. 가맹 협력사 필터링
    // 4. 협력사명 배치 조회
    // 5. 선정산금액 배치 조회
    // 6. 계산 및 DTO 빌드
    return result;
});
```

## 10. 회고 및 주의사항

1. **가맹 협력사 필터링 위치**: `vr_supp_acct_weekly` 에는 모든 협력사 데이터가 저장되므로, 조회 후 `vr_supplier_info.SUPP_TYPE='F'` 로 인메모리 필터링한다. 데이터 볼륨이 커지면 쿼리 단에서 JOIN으로 처리하는 것도 고려.

2. **협력사명/사업자번호 조회**: `vr_supp_acct_weekly` 에 저장된 값 대신 항상 `vr_supplier_info` 에서 최신 정보를 조회함. 마스터 데이터 변경 시 일관성 유지.

3. **선정산금액 차감**: 엑셀 업로드 시 `PartnerSettlementUseCase`가 `vr_account_mst`(ACCT_TYPE='SP')에서 정산기간 내 실지급 선정산 합계를 조회하여 `ACCT_VAL`(최종 지급금액) 계산 시 차감한다. `FranchiseSettlementUseCase`(조회 전용)의 `paymentAmount`는 선정산 차감 전 총액이며, `preSettlementAmt`를 별도 컬럼으로 나란히 표시하는 구조다.

4. **비동기 처리**: `CompletableFuture` 를 전 레이어에 걸쳐 사용. 컨트롤러 반환 타입도 `CompletableFuture<ResponseEntity<...>>` 로 통일.

5. **히스토리 보존**: 재업로드/원장 적용 전에 반드시 히스토리 백업 → 데이터 추적 가능.
