## 1. 라이더 상세정보 개별 트랜잭션 처리

> [!info] 배경
> - 라이더 상세 정보 크롤링 배치 실행 시 "저장 완료" 로그가 출력되지만 DB에 데이터가 없는 현상 발생
> - 배민 API에서 간헐적으로 403(봇 감지/Cloudflare) 에러 발생

> [!warning] 문제
> - `RiderDetailInfoUseCase.getRiderDetailInfo()`에 `@Transactional`이 적용되어 있음
> - 한 라이더라도 API 호출 실패 시 전체 트랜잭션 롤백 → 이전에 저장한 모든 데이터 소실

> [!success] 해결
> - `RiderDetailInfoSaveService` 신규 생성
> - `@Transactional(propagation = Propagation.REQUIRES_NEW)` 적용으로 개별 트랜잭션 처리
> - 각 라이더별 try-catch로 실패 시 스킵하고 계속 진행

```java
// RiderDetailInfoSaveService.java
@Service
@RequiredArgsConstructor
public class RiderDetailInfoSaveService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveRiderDetailInfo(RiderDetailInfoParse riderDetailInfo,
                                    String centerId, String ifDate, Integer ifDateSeq) {
        // 개별 저장 로직
    }
}
```

> [!tip] 배운 점
> - 배치 작업에서 외부 API 호출 시 개별 트랜잭션 분리 필수
> - 403 에러는 봇 감지로 랜덤 발생 → 재시도 시 성공 가능

## 3. vr_rider_info 배치 흐름 분석

> [!info] 분석 내용
> - vr_rider_info 테이블에 데이터가 적재되기까지의 전체 흐름 파악

**데이터 흐름:**
```
[배민 크롤링]
     │
     ▼
┌─────────────────┐     ┌──────────────────────┐
│  rider_info_if  │ ──▶ │                      │
└─────────────────┘     │    vr_rider_info     │
┌──────────────────────┐│                      │
│ rider_detail_info_if │──▶                    │
└──────────────────────┘└──────────────────────┘
```

**1단계: 크롤링**
- `GET /api/v1/rider/info` → rider_info_if
- `GET /api/v1/rider/info/detail` → rider_detail_info_if

**2단계: 마이그레이션**
- `POST /api/v1/migration/migrate` (tableName: "rider_info_if")
- 두 IF 테이블 JOIN 후 vr_rider_info에 INSERT/UPDATE
- transfer_yn='Y'로 마킹

**주요 파일:**
- `BaeminToVroongIfUseCase.java` (migrateRiderInfoIf 메서드)
- `VrRiderInfoQueryDslRepository.java` (mergeRiderInfo, batchMergeRiderInfo)
