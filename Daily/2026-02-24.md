## 1. 프로젝트 폴더 구조 대규모 리팩토링

> [!info] 배경
> - 기존 프로젝트 폴더 구조가 혼재되어 어떤 파일이 어떤 목적으로 만들어졌는지 파악이 어려움
> - 크롤링 관련 코드와 웹/앱 백엔드 코드가 섞여 있음
> - domain/migration 폴더에 크롤링과 배치 처리 코드가 혼합됨

> [!success] 해결
> - **domain 계층 리팩토링**
>   - `domain/crawling/baemin/` - 배민 크롤링 관련 코드 통합 (auth, center, delivery, migration, partner, rider)
>   - `domain/batch/` - 배치 처리 코드 분리 (backup, headbalamt, withdrawal)
>   - `domain/portal/` - 포털 비즈니스 로직 (balance, supplier 등 추가)
>   - `domain/mobile/` - 모바일 앱 비즈니스 로직
>
> - **infrastructure 계층 리팩토링**
>   - `infrastructure/crawling/` - 크롤링 인프라 (webclient, crypto, http)
>   - `infrastructure/payment/hecto/` - Hecto 결제 관련 분리
>   - `infrastructure/crypto/` - 일반 암호화 유틸만 유지
>
> - **app 계층 리팩토링**
>   - `app/controller/crawling/` - 크롤링 API (auth, center, delivery, migration, partner, rider)
>   - `app/controller/batch/` - 배치 처리 API
>   - `app/controller/portal/` - 포털 API (balance, supplier 이동)
>   - `app/request/crawling/` - 크롤링 요청 DTO
>   - `app/response/crawling/` - 크롤링 응답 DTO
>   - `app/response/batch/` - 배치 응답 DTO

> [!tip] 배운 점
> - Clean Architecture에서 계층 간 의존성 방향이 중요함
> - 크롤링 모델을 사용하는 Converter는 크롤링 도메인에 위치해야 함
> - Controller는 절대 domain 계층에 있으면 안 됨 (app 계층에 위치)
> - JPA Repository는 Entity 기준으로 분류해도 무방하나, 도메인 인터페이스 구현체는 도메인 구조를 따라야 함

## 3. 배치 실행 오류 해결 (relay-server)

> [!bug] 증상
> - relay-server 재배포 후 배치가 실행되지 않음
> - 배치 로그에 `400 Bad Request` 오류 다수 발생
> - `/api/v1/rider/info`, `/api/v1/delivery/rider-delivery-history` 등 API 호출 실패

> [!info] 원인 분석
> - 백엔드 API의 `SessionHelper.getActiveSessionId()`가 Redis `active_sessions` Set 조회
> - Redis 확인 결과: `SMEMBERS vroong:baemin:dev:active_sessions` → `(empty array)`
> - 세션 데이터는 존재하지만 `active_sessions` Set이 비어있어서 세션을 찾지 못함
>
> **문제 코드 흐름:**
> 1. relay-server 시작 시 `startSessionKeepAlive()` 즉시 실행 (1분 간격)
> 2. `keepAllSessionsAlive()` → `getSessionFromRedis(sessionId)` 호출
> 3. Puppeteer 모드에서 `browserManager.getSession()`이 세션을 찾지 못함 (아직 초기화 안됨)
> 4. `dualSessionManager.state.isInitialized`도 false → null 반환
> 5. **세션이 null이면 `active_sessions`에서 제거** ← 핵심 문제

> [!success] 해결
> `relay-server/server.js` - `keepAllSessionsAlive()` 함수 수정
>
> ```javascript
> if (!session) {
>     // Puppeteer 모드에서는 Redis에 세션 데이터가 있는지 한번 더 확인
>     // browserManager 상태와 Redis 상태가 불일치할 수 있음
>     if (USE_PUPPETEER) {
>         const sessionKey = `${REDIS_KEY_PREFIX}session:${sessionId}`;
>         const redisSessionData = await redisClient.get(sessionKey);
>
>         if (redisSessionData) {
>             const parsedSession = JSON.parse(redisSessionData);
>             if (parsedSession.loginStatus === 'AUTHENTICATED') {
>                 console.log(`[Keep-Alive] 세션 ${sessionId.substring(0, 8)}... - browserManager에 없지만 Redis에 유효한 세션 존재, 유지`);
>                 continue; // 삭제하지 않고 다음 세션으로
>             }
>         }
>     }
>     // ... 삭제 진행
> }
> ```

> [!tip] 배운 점
> - Puppeteer 모드에서 `browserManager`(메모리)와 Redis 상태가 불일치할 수 있음
> - 타이밍 이슈: keep-alive가 세션 생성보다 먼저 실행될 수 있음
> - 세션 삭제 전 Redis에 유효한 데이터가 있는지 이중 확인 필요
