## 1. 수수료/페이백 저장 로직 버그 수정

> [!info] 배경
> - 페이백을 "미사용"으로 변경 시 END_DAY가 2026-03-24로 변경되는 문제 발생
> - 기대 동작: END_DAY는 9999-12-31 유지, 새 레코드 생성
> - 추가로 UPD_DATE가 저장되지 않는 문제 발견

> [!warning] 문제 1: START_DAY > END_DAY 발생
> - 예정 수수료(아직 시작 안된)의 END_DAY도 업데이트되어 논리적으로 잘못된 데이터 발생
> - 예: START_DAY=2026-03-25, END_DAY=2026-03-24 (불가능한 조합)

**문제 상황:**
```
목요일에 수수료 생성 → START_DAY=4/1, END_DAY=9999-12-31
금요일에 페이백 변경 →
  - 기존 레코드: END_DAY=3/31로 변경됨 (START_DAY=4/1 > END_DAY=3/31 ❌)
  - 새 레코드: START_DAY=4/1, END_DAY=9999-12-31
```

> [!success] 해결
> - 예정 수수료(START_DAY > newEndDay)는 END_DAY 변경 대신 USE_YN='N'으로 무효화
> - 새 레코드의 END_DAY는 항상 9999-12-31 (DEFAULT_END_DATE) 사용

## 2. 예정 수수료 무효화 로직 추가

> [!info] 구현 내용
> - 새 메서드 `invalidateScheduledSupplierCommission` 추가
> - 아직 시작되지 않은 예정 수수료를 USE_YN='N'으로 무효화
> - 이력은 보존하면서 새 수수료가 적용되도록 함

### 변경된 파일

**FeeMstJpaRepository.java:**
```java
// 기존: 모든 레코드의 END_DAY 업데이트 (문제 발생)
// 변경: 이미 시작된 수수료만 END_DAY 업데이트
@Modifying(flushAutomatically = true, clearAutomatically = true)
@Query(
    "UPDATE FeeMstEntity f "
        + "SET f.endDay = :newEndDay, f.updUser = :updUser, f.updDate = CURRENT_TIMESTAMP "
        + "WHERE f.suppId = :suppId "
        + "AND f.feeType = 'C' "
        + "AND f.feeTarget = 'SRA' "
        + "AND f.settlementPeriod = :settlementPeriod "
        + "AND f.useYn = 'Y' "
        + "AND f.endDay > :newEndDay "
        + "AND f.startDay <= :newEndDay")  // 추가: 이미 시작된 수수료만
int updatePreviousSupplierCommissionEndDay(...);

// 신규: 예정 수수료 무효화
@Modifying(flushAutomatically = true, clearAutomatically = true)
@Query(
    "UPDATE FeeMstEntity f "
        + "SET f.useYn = 'N', f.updUser = :updUser, f.updDate = CURRENT_TIMESTAMP "
        + "WHERE f.suppId = :suppId "
        + "AND f.feeType = 'C' "
        + "AND f.feeTarget = 'SRA' "
        + "AND f.settlementPeriod = :settlementPeriod "
        + "AND f.useYn = 'Y' "
        + "AND f.startDay > :newEndDay")  // 아직 시작 안된 예정 수수료
int invalidateScheduledSupplierCommission(...);
```

**FeeUseCase.java:**
```java
// 1. 이미 시작된 수수료의 END_DAY 업데이트
// 2. 아직 시작되지 않은 예정 수수료 무효화 (신규)
// 3. 새로운 수수료 저장 (END_DAY는 항상 9999-12-31)
return feeMstRepository
    .updatePreviousSupplierCommissionEndDay(...)
    .thenCompose(updatedCount -> {
        return feeMstRepository.invalidateScheduledSupplierCommission(...);
    })
    .thenCompose(invalidatedCount -> {
        FeeMst feeMst = FeeMst.builder()
            ...
            .endDay(DEFAULT_END_DATE)  // 항상 9999-12-31
            ...
            .build();
        return feeMstRepository.save(feeMst);
    });
```

## 3. UPD_DATE 저장 안되는 문제 해결

> [!warning] 문제
> - `@Modifying @Query`를 사용한 벌크 업데이트 시 UPD_DATE가 저장되지 않음
> - 쿼리에 `f.updDate = CURRENT_TIMESTAMP`가 있지만 실제 저장 안됨

> [!success] 해결
> - `@Modifying` 어노테이션에 `flushAutomatically = true, clearAutomatically = true` 추가
> - 영속성 컨텍스트를 강제로 플러시하여 업데이트 반영

```java
// 변경 전
@Modifying
@Query("UPDATE FeeMstEntity f SET ...")

// 변경 후
@Modifying(flushAutomatically = true, clearAutomatically = true)
@Query("UPDATE FeeMstEntity f SET ...")
```

## 4. 수정 후 데이터 검증

**수정 전 (문제 상황):**
| SEQ | START_DAY | END_DAY | USE_YN | 문제 |
|-----|-----------|---------|--------|------|
| 1975 | 2026-03-25 | 2026-03-24 | Y | START_DAY > END_DAY |
| 1976 | 2026-03-25 | 2026-03-24 | Y | START_DAY > END_DAY |

**수정 후 (정상):**
| SEQ | START_DAY | END_DAY | USE_YN | 상태 |
|-----|-----------|---------|--------|------|
| 1975 | 2026-03-25 | 9999-12-31 | N | 무효화됨 (이력 보존) |
| 1976 | 2026-03-25 | 9999-12-31 | N | 무효화됨 (이력 보존) |
| 1981 | 2026-03-25 | 9999-12-31 | Y | 새 활성 수수료 |
| 1982 | 2026-03-25 | 9999-12-31 | Y | 새 활성 수수료 |

## 5. 관련 필드 설명

| 필드 | 설명 |
|------|------|
| `USE_YN` | Y: 활성 (정산에 적용), N: 무효화 (이력만 보존) |
| `APPLY_TIMING` | IMMEDIATE: 즉시 적용, NEXT_WEDNESDAY: 다음 수요일 적용 예정 |
| `START_DAY` / `END_DAY` | 수수료 적용 기간 (정산 시 날짜 기준으로 조회) |

**날짜별 수수료 적용:**
```
2025-12-24 ~ 2026-03-24: SEQ 1054, 1055 적용
2026-03-25 ~ 무기한:     SEQ 1981, 1982 적용

* SEQ 1975, 1976: USE_YN='N'으로 무효화 (이력만 보존)
```

## 배운 점

> [!tip] JPA @Modifying 쿼리 주의사항
> - 벌크 업데이트 시 `flushAutomatically = true, clearAutomatically = true` 옵션 필요
> - 영속성 컨텍스트를 거치지 않으므로 `@PreUpdate` 콜백이 동작하지 않음

> [!tip] 예정 데이터 처리
> - 미래 적용 예정 데이터 수정 시 END_DAY 변경 대신 USE_YN='N'으로 무효화
> - START_DAY > END_DAY 같은 논리적 오류 방지
