## 1. DB 커넥션 풀(HikariCP) 장애 분석 및 해결

> [!info] 배경
> - 2026-02-20 12:23경 운영 서버에서 대규모 장애 발생
> - 배치 작업 실패, API 500 에러, DB 커넥션 타임아웃 다수 발생
> - 주요 에러 로그:
>   - `Communications link failure during rollback()`
>   - `HikariPool - Connection is not available, request timed out after 30000ms`
>   - `Connection prematurely closed BEFORE response`

> [!warning] 문제
> - **핵심 로그 (12:23:08)**:
>   ```
>   WARN HikariPool-1 - Failed to validate connection (No operations allowed after connection closed.)
>   Possibly consider using a shorter maxLifetime value.
>   ```
> - **원인**: MySQL 서버가 유휴 커넥션을 `wait_timeout`에 의해 자동 종료했으나, HikariCP는 이를 인지하지 못하고 이미 끊어진 커넥션을 사용 시도
> - **근본 원인**: `application.yml`에 커넥션 검증 설정 누락
>   ```yaml
>   # 기존 설정 (문제)
>   hikari:
>     maximum-pool-size: 200
>     minimum-idle: 10
>     connection-timeout: 30000
>     idle-timeout: 600000
>     # max-lifetime: 없음 → 기본값 30분 사용
>     # connection-test-query: 없음 → 커넥션 검증 안함
>   ```

> [!success] 해결
> - `application.yml`에 HikariCP 설정 추가
>   ```yaml
>   hikari:
>     maximum-pool-size: 200
>     minimum-idle: 10
>     connection-timeout: 30000
>     idle-timeout: 600000
>     max-lifetime: 1740000           # 29분 (MySQL wait_timeout보다 짧게)
>     connection-test-query: SELECT 1 # 사용 전 커넥션 유효성 검증
>     validation-timeout: 5000        # 검증 타임아웃 5초
>   ```

> [!tip] 배운 점
> - **max-lifetime**: MySQL의 `wait_timeout`보다 짧게 설정해야 함 (서버가 끊기 전에 먼저 교체)
> - **connection-test-query**: 커넥션 사용 전 `SELECT 1`로 살아있는지 확인
> - **validation-timeout**: 검증 쿼리 타임아웃 설정 (죽은 커넥션 빠르게 감지)
> - 운영 환경과 로컬 환경의 설정 차이 주의 (local에는 있던 설정이 prod에 없었음)

### HikariCP 주요 설정 정리

| 설정 | 설명 | 권장값 |
|------|------|--------|
| `max-lifetime` | 커넥션 최대 수명 | MySQL wait_timeout - 1분 |

| `connection-timeout` | 커넥션 획득 대기 시간 | 30000ms (30초) |
