## 1. vroong-swagger-tool 세션 복구 기능 구현

> [!info] 배경
> - 배민에서 보안 강화로 기존 BrightData + Axios 방식의 로그인이 막힘
> - 현재 세션 복구 과정이 복잡함:
>   1. vroong-backend/relay-server를 로컬에서 실행
>   2. Swagger로 login → 2FA → verify API 호출
>   3. 로컬 Redis에서 세션 키 조회
>   4. AWS Redis에 수동으로 SET 명령어 실행
> - 이 과정을 vroong-swagger-tool 하나로 통합하고 싶음

> [!warning] 문제
> - relay-server 없이 독립적으로 Puppeteer + Redis 기능 필요
> - TypeScript 타입 오류 다수 발생:
>   - `puppeteer` 모듈 타입 선언 못 찾음
>   - `ioredis` 모듈 타입 선언 못 찾음
>   - `Cookie` 타입이 puppeteer에서 export 안 됨
>   - `Window.electronAPI` 타입 충돌
>   - `path` 모듈 esModuleInterop 이슈
> - 세션 데이터 형식이 relay-server와 달랐음 (ISO string vs Unix timestamp)

> [!success] 해결
>
> **1. 새로운 파일 생성:**
> - `src/main/services/baeminAuth.ts` - Puppeteer 기반 배민 로그인 자동화
> - `src/main/services/redisService.ts` - AWS Redis 직접 연결
> - `src/renderer/pages/SessionRecoveryPage.tsx` - 5단계 위저드 UI
>
> **2. 타입 오류 해결:**
> ```typescript
> // Cookie 타입 → Protocol.Network.Cookie 사용
> import puppeteer, { Browser, Page, Protocol } from 'puppeteer-core';
>
> // path import 수정
> import * as path from 'path';
>
> // Window.electronAPI 타입은 vite-env.d.ts에 통합
> ```
>
> **3. 세션 형식 통일 (초 단위 Unix timestamp):**
> ```typescript
> {
>   sessionId: "...",
>   userName: "cleanstar1212",
>   loginStatus: "AUTHENTICATED",
>   cookies: {...},
>   createdAt: 1773106533.600297200,      // Date.now() / 1000
>   lastAccessTime: 1773106566.758243700  // Date.now() / 1000
> }
> ```
>
> **4. puppeteer → puppeteer-core 변경:**
> - 시스템에 설치된 Chrome을 직접 사용 (Chromium 번들 제외)
> - `findChromePath()` 함수로 Windows/Mac/Linux Chrome 경로 탐색

> [!tip] 배운 점
> - puppeteer-core는 Chromium을 번들하지 않아 용량이 작음
> - Electron의 contextIsolation 환경에서는 preload.ts를 통해 IPC 통신 필요
> - ioredis v5+는 타입 선언이 내장되어 있음
> - Protocol.Network.Cookie는 puppeteer의 쿠키 타입
