## 1. relay-server 유령 세션 자동 정리

> [!info] 배경
> - ECS 재배포 후 Redis `active_sessions`에 유령 세션이 남아있음
> - `bdfa6cd0` 세션이 Redis에는 있지만 로컬 browserManager에는 없음
> - ECS desiredCount=1 환경에서 old task가 종료 전 세션을 다시 Redis에 추가하는 race condition 발생

> [!warning] 문제
> - `keepAllSessionsAlive()`에서 로컬에 없는 세션을 "다른 인스턴스의 세션"으로 간주하고 건너뜀
> - 단일 인스턴스 환경에서는 다른 인스턴스가 없으므로 유령 세션이 정리되지 않음
> - `active_sessions: 2개`로 표시되지만 실제 유효 세션은 1개

> [!success] 해결
> - `keepAllSessionsAlive()`에 유령 세션 판별 및 정리 로직 추가
> - 로컬에 없는 세션 발견 시:
>   - Redis에 세션 데이터도 없음 → 즉시 `active_sessions`에서 제거
>   - Redis에 세션 데이터 있음 + 30초 이상 경과 → 유령 세션으로 판단, 세션 데이터와 함께 삭제
>   - Redis에 세션 데이터 있음 + 30초 미만 → 건너뛰기 (배포 과도기 대기)

**수정 파일:**
```
relay-server/server.js (keepAllSessionsAlive 함수)
```

**수정 코드:**
```javascript
if (!localSession) {
    const sessionData = await getSessionFromRedis(sessionId);
    if (!sessionData) {
        // Redis에 세션 데이터도 없음 = 확실한 유령 세션
        console.log(`[Keep-Alive] 세션 ${sessionId.substring(0, 8)}... 유령 세션 (데이터 없음) - active_sessions에서 제거`);
        await redisClient.sRem(`${REDIS_KEY_PREFIX}active_sessions`, sessionId);
    } else {
        const sessionAge = Date.now() - new Date(sessionData.lastChecked || sessionData.createdAt).getTime();
        const GHOST_SESSION_THRESHOLD = 30 * 1000; // 30초

        if (sessionAge > GHOST_SESSION_THRESHOLD) {
            console.log(`[Keep-Alive] 세션 ${sessionId.substring(0, 8)}... 유령 세션 (${Math.round(sessionAge/1000)}초 경과) - 정리`);
            await redisClient.sRem(`${REDIS_KEY_PREFIX}active_sessions`, sessionId);
            await redisClient.del(`${REDIS_KEY_PREFIX}session:${sessionId}`);
        }
    }
    continue;
}
```

## 2. 가상계좌 중복 생성 방지 및 협력사 변경 동기화

> [!info] 배경
> - `vr_virtual_account_list` 테이블에 같은 라이더에 대해 2개의 가상계좌 발견
> - SEQ 412: SUPPLY_ID=DP2504254627, 2025-12-17 (이전 협력사)
> - SEQ 569: SUPPLY_ID=DP2602231072, 2026-03-04 (새 협력사)
> - 라이더가 협력사를 변경했을 때 새 가상계좌가 생성되고 이전 계좌는 그대로 남음

> [!warning] 문제
> - `assignVirtualAccountToRider()`가 기존 계좌 존재 여부를 체크하지 않음
> - 협력사 변경 시 가상계좌의 SUPP_ID가 업데이트되지 않음
> - 협력사 변경이 "계약 해제 → 재계약" 흐름이 아닌 직접 SUPP_ID 변경인 경우 문제 발생

> [!success] 해결
> 1. **`assignVirtualAccountToRider()` 수정**: 이미 라이더에게 할당된 계좌가 있으면 새 계좌 생성하지 않음
> 2. **`updateVirtualAccountSuppId()` 신규 메서드 추가**: 가상계좌의 SUPP_ID 업데이트
> 3. **`batchMergeRiderInfo()` 수정**: 협력사 변경 감지 시 가상계좌 SUPP_ID도 함께 업데이트

**수정 파일:**
```
api/src/main/java/com/vroong/api/infrastructure/persistence/repository/vroong/VrRiderInfoQueryDslRepository.java
```

### 수정 1: 기존 계좌 확인 로직 추가 (assignVirtualAccountToRider)

// 이미 해당 라이더에게 할당된 가상계좌가 있는지 확인

### 수정 2: 가상계좌 SUPP_ID 업데이트 메서드 추가

```java
public long updateVirtualAccountSuppId(
    String riderId, String oldSuppId, String newSuppId, LocalDateTime updDate, String updUser) {
    QVrVirtualAccountListEntity vval = QVrVirtualAccountListEntity.vrVirtualAccountListEntity;

    long updatedRows = queryFactory
        .update(vval)
        .set(vval.suppId, newSuppId)
        .set(vval.updDate, updDate)
        .set(vval.updUser, updUser)
        .where(vval.riderId.eq(riderId)
            .and(vval.suppId.eq(oldSuppId))
            .and(vval.acctType.eq("R")))
        .execute();

    log.info("가상계좌 협력사 ID 업데이트 완료: riderId={}, oldSuppId={}, newSuppId={}, updatedRows={}",
        riderId, oldSuppId, newSuppId, updatedRows);

    return updatedRows;
}
```

> [!tip] 배운 점
> - 가상계좌 할당 시 기존 할당 여부를 먼저 체크해야 중복 생성 방지 가능
> - 협력사 변경은 "계약상태 변경"과 별개의 이벤트이므로 별도 처리 필요
> - 관련 테이블 간 데이터 정합성을 유지하려면 변경 감지 시 연관 테이블도 함께 업데이트
