## 1. 협력사→라이더 프로모션 정산 흐름 신규 구현

> [!info] 배경
> 고객사 요구사항: 협력사가 등록한 프로모션 미션(배달 건수 달성 시 보너스)의 지급액을
> 주정산 업로드 시점에 자동 계산하고, 협력사 보유 잔액과 비교해 보류 또는 입금 요청 처리.

---

### 이전 흐름 (변경 전)

```
[라이더 주정산 업로드]
  └─ 배달료 + 추가지급 - 운영수수료 = 과세배달료
  └─ 과세배달료 기준으로 보험료/세금 계산
  └─ 프로모션 지급액은 totAcctVal에 단순 가산 (세금 계산 후 추가)
     → 라이더가 프로모션에 대한 세금을 내지 않는 구조

[협력사 주정산 업로드]
  └─ 협력사별 주정산비용(acctVal) 저장
  └─ 프로모션 비용 추적 없음

[정산 적용 (Apply)]
  └─ 라이더: vr_account_mst에 RW 레코드 생성
  └─ 협력사: vr_account_mst에 SW 레코드 생성
  └─ 프로모션 비용 협력사 차감 없음

[출금 요청]
  └─ withdrawalYn = Y 여부만 체크
  └─ 프로모션 보류 금액 고려 없음
```

---

### 변경된 흐름 (변경 후)

#### A. 과세배달료 계산식 변경

```
[변경 전]
  과세배달료 = 배달료 + 추가지급 - 총운영수수료
  totAcctVal = 과세배달료 기반 지급액 + promotion (세금 계산 이후 추가)

[변경 후]
  과세배달료 = 배달료 + 추가지급 + promotion - 총운영수수료
               ↑ promotion이 과세배달료에 포함됨
  → 라이더가 프로모션 금액에 대한 세금(소득세, 고용보험 등)을 부담
  → calculateSettlementAmounts() 호출 전에 promotion 계산
```

#### B. 라이더 주정산 업로드 시

```
[RiderSettlementUseCase.uploadExcel()]
  1. 엑셀 파싱 → 각 라이더별 settlement 계산 (promotion 포함)
  2. saveAll() 완료 후:
     - saved 리스트에서 suppId별 promotion 합계 집계
     - promotion > 0인 suppId에 대해:
       promoSuppSettlementUseCase.saveFromRiderUpload(
           suppId, acctStDate, acctEdDate, totalPromoCost, userId)
       → vr_promo_supp_settlement INSERT (holdStatus = PENDING)
```

#### C. 협력사 주정산 업로드 시

```
[PartnerSettlementUseCase.uploadExcel()]
  1. 엑셀 파싱 → 협력사별 acctVal(주정산비용) 저장
  2. saveAll() 완료 후, 각 협력사에 대해:
     promoSuppSettlementUseCase.evaluateAfterPartnerUpload(
         suppId, acctStDate, acctEdDate, partnerAcctVal, userId)

     evaluateAfterPartnerUpload 내부 로직:
       - vr_promo_supp_settlement에서 해당 기간 레코드 조회
       - 레코드 없음 → 종료 (프로모션 없는 협력사)
       - partnerAcctVal >= totalPromoCost
           → holdStatus = HELD (출금 제한)
       - partnerAcctVal < totalPromoCost
           → holdStatus = PENDING_DEPOSIT
           → vr_notice + vr_notice_target에 입금 요청 공지 생성
              (협력사 로그인 시 팝업 표시)
```

#### D. 정산 적용 (Apply) 시

```
[SettlementUploadUseCase.applyUpload()]
  공통 선행 작업:
    pendingDepositSuppIds = promoSuppSettlementUseCase.getPendingDepositSuppIds(period)

  RIDER apply:
    - applyToAccountMstExcluding(acctStDate, acctEdDate, userId, pendingDepositSuppIds)
      → PENDING_DEPOSIT 협력사 소속 라이더는 RW 레코드 생성 제외
      → 나머지 라이더는 기존과 동일하게 처리

  PARTNER apply:
    - applyToAccountMstExcluding(acctStDate, acctEdDate, userId, pendingDepositSuppIds)
      → PENDING_DEPOSIT 협력사는 SW 레코드 생성 제외
    - promoSuppSettlementUseCase.applyHeldSettlements(acctStDate, acctEdDate, userId)
      → HELD 협력사별 totalPromoCost를 잔액에서 차감
      → vr_account_mst에 acctType = "PD" (프로모션 공제) 레코드 생성
      → holdStatus = PROCESSED 마킹
```

#### E. 출금 요청 시 (MobWithdrlUseCase.requestWithdrl)

```
[변경 전]
  withdrawalYn = Y → 차단
  요청금액 + 수수료 > 잔액 → 차단

[변경 후]
  withdrawalYn = Y → 차단
  협력사인 경우:
    heldAmt = promoSuppSettlementUseCase.getTotalHeldAmount(suppId)
    if heldAmt > 0:
      availableAmt = balAmt - heldAmt   (totalPromoCost + 300원 버퍼)
      if reqVal > availableAmt → 차단
  요청금액 + 수수료 > 잔액 → 차단
```

#### F. 강제 지급 (관리자)

```
POST /api/v1/promotion/supp-settlement/{suppId}/force
  → promoSuppSettlementUseCase.forceProcess() → boolean 반환
      true  = 처리 전 상태가 PENDING_DEPOSIT (라이더 RW 미반영)
              → riderSettlementUseCase.applyRidersForSuppId() 호출
                (vr_promo_rider_result에서 promotion 복원 → 전체 재계산 → RW 원장 반영)
      false = 처리 전 상태가 SW_ONLY (라이더 RW 이미 반영됨)
              → PD 레코드만 생성 (라이더 재처리 없음)
  → holdStatus: PENDING_DEPOSIT → FORCED
```

#### G. 주정산만 진행 (관리자)

```
POST /api/v1/promotion/supp-settlement/{suppId}/sw-only
  → promoSuppSettlementUseCase.processSwOnly()
      → SW 레코드 생성
      → holdStatus: PENDING_DEPOSIT → PROCESSED (SW_ONLY 상태 없음, 프로모션 소멸)
  → riderSettlementUseCase.clearPromotionAndApplyRiders()
      → vr_rider_acct_weekly는 이미 promotion=null 상태 (PENDING_DEPOSIT 판정 시 보정됨)
      → 그대로 saveToAccountMst 호출 (재계산 불필요)
      → vr_promo_rider_result 정리
```

### holdStatus 생명주기 (최종)

```
라이더 업로드
    │ saveFromRiderUpload → PENDING
    ▼
PENDING ──────────────────────────────────────────┐
    │                                             │
협력사 업로드 (evaluateAfterPartnerUpload)          │
    │                                             │
    ├── partnerAcctVal >= totalPromoCost           │
    │       │                                     │
    │       ▼                                     │
    │     HELD ──── apply 시 PD 차감 ──► PROCESSED │
    │                                             │
    └── partnerAcctVal < totalPromoCost           │
            │                                     │
            │ ※ 이 시점에 savePromoAndClearForPendingDeposit 자동 호출
            │   - 라이더별 promotion → vr_promo_rider_result 보관
            │   - vr_rider_acct_weekly promotion=null 재계산
            ▼
        PENDING_DEPOSIT
            │
            ├── 관리자 강제 지급 → FORCED
            │       └─ vr_promo_rider_result 복원 → 라이더 재계산 → RW 반영
            │
            └── 관리자 주정산만 진행 → PROCESSED
                    └─ vr_rider_acct_weekly 그대로 원장 반영 (promotion=null)
                    └─ vr_promo_rider_result 삭제 (프로모션 소멸)
```

## 📝 메모

- 재업로드 기능은 영구적으로 막혀 있음. `saveFromRiderUpload`는 INSERT only.
- HOLD_BUFFER = 300원 (totalPromoCost + 300원이 출금 불가 금액)
- 출금 제한은 suppId 기준 모든 HELD 기간 합산 (`getTotalHeldAmount`)
- 공지 생성 시 writerType = "C" (본사), writerId = "SYSTEM", popupYn = "Y", urgentYn = "Y"
- SW_ONLY 상태는 최종적으로 제거됨. processSwOnly는 PROCESSED 직행.
- `vr_promo_rider_result`는 임시 보관 테이블 — 처리 완료 후 반드시 삭제됨.
- PENDING_DEPOSIT 협력사의 라이더는 정산 화면에서 promotion=null 금액으로 표시됨 (혼동 방지).
- 강제진행 시 `saveToAccountMst(forceRw=true)`로 히스토리 체크 없이 RW 강제 생성.
- `vr_promo_rider_result`가 없는 강제진행 (프로모션 없는 협력사)도 정상 처리됨 (promotionMap 비어있으면 재계산 스킵).
