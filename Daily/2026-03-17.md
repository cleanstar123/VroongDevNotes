## 1. 수수료 END_DAY 관리 로직 변경

> [!info] 배경
> - 수수료 변경 시 기존에는 USE_YN='N'으로 비활성화 처리
> - 수요일 기반 적용 로직에서 새 수수료는 다음 주 수요일부터 적용되므로, 기존 수수료와 새 수수료 사이에 공백 기간 발생

> [!warning] 문제
> - 예: 3/11(화) 수수료 변경 시 → 기존 수수료 USE_YN='N' 처리
> - 새 수수료 START_DAY=3/12(수) 부터 적용
> - 3/11 당일 정산 시 적용할 수수료 없음 (기존은 N, 신규는 미래)

> [!success] 해결
> - USE_YN='N' 대신 END_DAY를 새 수수료 START_DAY - 1일로 업데이트
> - 기존 수수료: END_DAY=3/11 (USE_YN='Y' 유지)
> - 새 수수료: START_DAY=3/12, END_DAY=9999-12-31
> - 정산 시 날짜 범위로 적용 수수료 결정 (START_DAY <= date <= END_DAY)

> [!tip] 배운 점
> - 과거 정산 데이터 조회를 위해 USE_YN은 'Y'로 유지하는 것이 중요
> - 수수료 이력 관리는 END_DAY로 기간을 구분하는 것이 더 명확

## 3. findActiveDefaultRiderFeeBySupplier 쿼리 수정

> [!warning] 문제
> - 기존 쿼리가 USE_YN='Y'만 확인하고 날짜 범위를 필터링하지 않음
> - 과거 기본 수수료(END_DAY가 지난)도 함께 조회되는 문제

> [!success] 해결
> - START_DAY <= CURRENT_DATE AND END_DAY >= CURRENT_DATE 조건 추가
> - ORDER BY를 INS_DATE DESC에서 START_DAY DESC로 변경

```java
@Query("SELECT f FROM FeeMstEntity f " +
        "WHERE f.suppId = :suppId " +
        "AND f.feeType = 'S' " +
        "AND f.feeTarget = 'SRA' " +
        "AND f.useYn = 'Y' " +
        "AND f.startDay <= CURRENT_DATE " +
        "AND f.endDay >= CURRENT_DATE " +
        "ORDER BY f.startDay DESC")
List<FeeMstEntity> findActiveDefaultRiderFeeBySupplier(@Param("suppId") String suppId);
```
