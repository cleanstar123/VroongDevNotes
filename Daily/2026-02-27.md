## 3. 엑셀 내보내기 기능 구현 및 개선

> [!info] 배경
> - API 호출 결과를 엑셀 파일로 다운로드하는 기능 필요
> - 복잡한 JSON 응답을 사람이 읽기 쉬운 형태로 변환 필요

### 3.1 초기 구현 문제

> [!warning] 문제
> - 배열 데이터(centers)가 JSON 문자열로 한 셀에 들어감
> - 예: `[{"id":"DP2504258339","name":"강원동해A부릉라이더스1",...},...]`

### 3.2 배열 데이터 행 펼침 개선

> [!success] 해결
> - 주요 배열 필드(centers, data, items 등) 자동 감지
> - 배열의 각 항목을 별도 행으로 표시

```typescript
// 목록 조회용 주요 배열 필드명
const listFieldNames = ['centers', 'data', 'items', 'list', 'riders',
    'deliveries', 'orders', 'results', 'records', 'users', 'partners', 'suppliers'];

// 응답 타입 판별
if (Array.isArray(responseData)) {
    exportData = responseData;
    isListResponse = true;
} else {
    for (const fieldName of listFieldNames) {
        if (Array.isArray(responseData[fieldName])) {
            exportData = responseData[fieldName];
            isListResponse = true;
            break;
        }
    }
    // 주요 배열 필드가 없으면 단일 상세 조회로 판단
    if (!isListResponse) {
        exportData = [responseData];
    }
}
```

### 3.3 단일 상세 조회 데이터 평탄화

> [!warning] 문제
> - 협력사 상세 정보 조회 시 `deliveryZone` 배열만 내보내짐
> - 전체 객체 정보(id, name, status, contractBusiness 등)가 누락됨

> [!success] 해결
> - 주요 배열 필드가 없으면 단일 상세 조회로 판단
> - 전체 객체를 평탄화하여 한 행으로 표시
> - 중첩 객체: `contractBusiness.businessName` → `contractBusiness_businessName`

### 3.4 중첩 배열 데이터 추출 (dongs)

> [!warning] 문제
> - `deliveryZone` 배열 내의 `dongs` 배열 데이터가 누락됨

> [!success] 해결
> - 중첩 배열의 각 필드별로 값 추출
> - 예: `deliveryZone_dongs_dongName` → "천곡동, 평릉동, 송정동, ..."

```typescript
// 중첩 배열 처리
if (Array.isArray(fieldValue) && typeof fieldValue[0] === 'object') {
    const nestedFields = Object.keys(fieldValue[0]);
    nestedFields.forEach(nestedKey => {
        const allNestedValues: string[] = [];
        value.forEach((parentItem: any) => {
            if (Array.isArray(parentItem[fieldKey])) {
                parentItem[fieldKey].forEach((nestedItem: any) => {
                    if (nestedItem[nestedKey] != null) {
                        allNestedValues.push(String(nestedItem[nestedKey]));
                    }
                });
            }
        });
        // 중복 제거
        const uniqueValues = [...new Set(allNestedValues)];
        result[`${fieldNewKey}_${nestedKey}`] = uniqueValues.join(', ');
    });
}
```

## 5. 날짜 형식(dateFormat) 파라미터 지원

> [!warning] 문제
> - 협력사 배달비 정보 조회 API 호출 시 400 Bad Request
> - 에러: "날짜 형식은 YYYY-MM-DD여야 합니다"
> - 기존 코드는 모든 날짜를 `YYYYMMDD` 형식으로 변환

> [!success] 해결
> - `ApiParameter` 인터페이스에 `dateFormat` 필드 추가
> - API별로 날짜 형식 지정 가능 (`YYYYMMDD` 또는 `YYYY-MM-DD`)

```typescript
// apiEndpoints.ts - 인터페이스 확장
export interface ApiParameter {
    name: string;
    type: 'string' | 'number' | 'date' | 'boolean' | 'select';
    required: boolean;
    description: string;
    in: 'path' | 'query' | 'body';
    options?: { label: string; value: string }[];
    defaultValue?: string;
    dateFormat?: 'YYYYMMDD' | 'YYYY-MM-DD';  // 추가
}

// 협력사 배달비 정보 조회 API
{
    name: 'fromDate',
    type: 'date',
    required: true,
    description: '시작일 (YYYY-MM-DD)',
    in: 'query',
    dateFormat: 'YYYY-MM-DD',  // 형식 지정
}
```

```typescript
// ApiExecutor.tsx - 날짜 변환 시 dateFormat 적용
Object.keys(values).forEach((key) => {
    const value = values[key];
    if (value && dayjs.isDayjs(value)) {
        const param = endpoint.parameters.find(p => p.name === key);
        const dateFormat = param?.dateFormat || 'YYYYMMDD';
        processedValues[key] = value.format(dateFormat);
    } else {
        processedValues[key] = value;
    }
});
```

## 6. relay-server 유령 세션 정리 버그 수정

> [!warning] 문제
> - 서버 배포 후 Keep-Alive 로그에 이전 세션이 계속 남아있음
> - `active_sessions: 2개`로 표시되나 실제 활성 세션은 1개
> - 로그: `세션 a5f3d960... 로컬에 없음 - 건너뛰기`

> [!info] 원인 분석
> - Puppeteer 모드에서 `browserManager.getSession()`이 null을 반환하면 건너뛰기만 하고 Redis에서 제거하지 않음
> - 배포 시 이전 인스턴스가 종료되기 전에 세션을 Redis에 다시 추가하는 레이스 컨디션 발생

> [!success] 해결
> - 로컬에 없는 세션은 유령 세션으로 판단하여 Redis에서 완전히 제거

```javascript
// server.js - keepAllSessionsAlive() 함수 수정
if (USE_PUPPETEER) {
    const localSession = browserManager.getSession(sessionId);
    if (!localSession) {
        // 수정 전: 건너뛰기만 함
        // console.log(`세션 ${sessionId}... 로컬에 없음 - 건너뛰기`);
        // continue;

        // 수정 후: Redis에서 정리
        console.log(`세션 ${sessionId}... 로컬에 없음 - Redis에서 정리`);
        await redisClient.sRem(`${REDIS_KEY_PREFIX}active_sessions`, sessionId);
        await redisClient.del(`${REDIS_KEY_PREFIX}session:${sessionId}`);
        await redisClient.del(`${REDIS_KEY_PREFIX}status:${sessionId}`);
        continue;
    }
}
```

## 7. [미해결] 자동 재인증 SMS 웹훅 간헐적 미수신 문제

> [!bug] 현상
> - 배포 후 2FA 인증이 **간헐적으로** 실패함
> - 어떤 배포에서는 성공, 어떤 배포에서는 실패
> - Macrodroid가 SMS 웹훅 전송 후 **200 응답**을 받았으나, 서버 로그에 웹훅 수신 기록 없음

### 로그 비교

| 항목 | 성공 (15:45) | 실패 (15:26) |
|------|-------------|-------------|
| 서버 시작 | 15:45:12 | 15:26:38 |
| 2FA 요청 | 15:45:59 | 15:27:23 |
| SMS 웹훅 도착 | 15:46:02 (1초 후) | **없음** |
| 결과 | 재인증 성공 | 2분 후 타임아웃 |

> [!info] 원인 추정
> - **DNS 캐싱**: Macrodroid가 이전 인스턴스 IP를 캐시하고 있음
> - **로드밸런서 라우팅 지연**: 새 인스턴스로 트래픽 전환에 시간 소요
> - 배포 직후: 웹훅이 이전(종료 중인) 인스턴스로 감 → 200 응답
> - 배포 수 분 후: 라우팅 안정화되어 새 인스턴스로 정상 도착

### 시도한 해결책 (원복됨)

> [!warning] 2FA 코드 15초마다 재요청 로직
> - 의도: SMS 재전송으로 Macrodroid가 새 웹훅 요청 보내도록 유도
> - 결과: `[Puppeteer] 인증번호 받기 버튼 대기 타임아웃` 반복 발생
> - 원인: 첫 요청 후 버튼이 "재전송" 등으로 변경되어 기존 셀렉터로 찾지 못함
> - **원복**: 15초 재요청 로직 제거

### 검토 중인 해결 방안

1. **Macrodroid 웹훅 URL에 IP 직접 사용** (도메인 대신)
2. **배포 후 재인증 지연** - 서버 시작 후 1-2분 대기 후 재인증 시작
3. **재인증 실패 시 재시도** - 실패하면 1분 후 다시 시도

**관련 파일:**
- `relay-server/services/auto-reauth.js`
- `relay-server/server.js`
